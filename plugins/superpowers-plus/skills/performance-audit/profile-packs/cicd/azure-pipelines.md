# CI/CD performance module: Azure DevOps Pipelines
> Load when `azure-pipelines.yml`/`azure-pipelines.yaml` (or an Azure DevOps `Pipelines` definition) is detected — see the module map in `../cicd.md`. Core lanes + the Pipeline cost model live in `../cicd.md`; this file is the Azure DevOps Pipelines lens only.

## Azure DevOps Pipelines

> Scope: the pipeline YAML — `trigger`/`pr` filters, the `stages`→`jobs`→`steps` hierarchy, `strategy`, the `Cache@2` task, `checkout`, and the pool/parallelism it runs against. The Azure-distinct physics that the GitHub Actions lens does *not* transfer are two inversions and one hard cap. **Inversion 1:** stages run *sequentially* by default (GitHub jobs are parallel by default), so the footgun here is the opposite — a serial stage chain that should fan out, not a missing dependency edge. **Inversion 2:** caching is a *task you add* (`Cache@2`), not an input on a setup action, and a fixed key is permanently stale because Azure caches are immutable. **The cap:** declared parallelism (`maxParallel`, matrix legs, fanned-out jobs) only buys wall-time up to the org's *purchased parallel-job count* — everything beyond it silently queues, and a new org's free grant is tiny. Defer the lane definitions (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) and the **Pipeline cost model** to `../cicd.md`; read this for the provider-specific surface.

- **Stages serial-by-default where the work is independent** (lane `concurrency`): unlike GitHub's parallel-by-default jobs, multiple `stages` run one after another in declaration order unless you say otherwise — so two stages that share no data (e.g. integration tests vs. acceptance tests, or two independent regional deploys) silently serialize and double the wall-time on the critical path. The fix is `dependsOn: []` (an explicit empty list) on a stage to release it immediately rather than after the prior stage drains; `dependsOn: [StageA, StageB]` expresses real fan-in. Read the data dependencies, not the authoring order — but note the inverse of the GitHub footgun: here the *default* is the conservative-but-slow one, so the win is adding parallelism, not removing a false edge. Cross-reference the **Pipeline cost model** in `../cicd.md` for the critical-path framing (verify against the currency brief for your version).

- **Jobs within a stage left serial when they could fan out** (lane `concurrency`): jobs in one stage *do* run in parallel by default (matching GitHub) — the trap is a `dependsOn` chain among them that encodes the author's mental ordering rather than a real artifact/gate dependency, forcing fresh agents end-to-end when they could start at once. A job that consumes another's published pipeline artifact is a genuine edge; a `lint → test` edge where they share nothing is not. Each agent runs exactly one job at a time, so a false edge both lengthens wall-time and idles purchased parallel capacity (verify against the currency brief for your version).

- **Fan-out that exceeds the purchased parallel-job count just queues** (lane `concurrency`): this is the Azure-specific ceiling — `maxParallel`, matrix legs, and `strategy.parallel` only convert to real concurrency up to the *organization's purchased parallel jobs*, which are org-wide and shared across all projects; beyond that, jobs queue and run one at a time regardless of how wide the YAML fans. A new org's free grant is small — for private projects it's one Microsoft-hosted job (60-min cap, 1,800 min/month) and the free grant is currently gated behind a request form — so a 10-leg matrix on a one-job org is 10 serial runs, not a 10x speedup. The real levers are buying parallel jobs or attaching a self-hosted pool; right-size declared fan-out to actual capacity. Cross-reference **Job parallelization** in `../cicd.md` (verify against the currency brief for your version).

- **Self-hosted pool agent count is the true concurrency cap, not the YAML** (lane `concurrency`): for self-hosted pools the limit is the number of registered agents that satisfy the job's `demands` — each agent runs one job at a time, so a 20-leg matrix against a 4-agent pool runs 4 at a time and queues the rest, identical in effect to under-purchased hosted parallelism. Tight or mismatched `demands` (an OS/capability filter only one agent meets) collapse effective parallelism further by pinning many jobs to a single agent. Audit declared fan-out against real agent throughput, not the matrix size (verify against the currency brief for your version).

- **No `Cache@2` task, so dependencies reinstall from the network every run** (lane `data-access`): Azure has *no* implicit dependency cache — caching exists only if you add the `Cache@2` step pointing `path` at the package manager's *global* store (e.g. `$(npm_config_cache)`, the Maven local repo, the NuGet packages dir), keyed on a lockfile hash. This is also distinct from Pipeline Artifacts: artifacts move job-produced files *between jobs in a run* (and the consumer fails without them), whereas the cache is a best-effort speedup the job runs fine without. A pipeline doing `npm ci`/`mvn install`/`dotnet restore` with no `Cache@2` re-pays full network install on every run — usually the single highest-ratio fix. Cross-reference **Caching & artifact I/O** in `../cicd.md` (verify against the currency brief for your version).

- **A `Cache@2` key that is fixed (permanently stale) or always-miss** (lane `data-access`): Azure caches are *immutable* — once a key is written for a scope it is never updated — so a fixed-string `key` serves the first run's dependencies forever even as the lockfile changes, and there is no "clear cache" (you bump a literal segment like `version2 | ...` to rotate). The correct shape hashes a lockfile (`key: 'npm | "$(Agent.OS)" | package-lock.json'`) so it changes exactly when deps do, *plus* `restoreKeys` as prefix-fallback to load the most-recent near-match. The opposite failure — a key embedding `$(Build.SourceVersion)` or a timestamp — misses every run and pays restore+save for zero reuse. Both look "cached" in YAML; verify the key tracks the dependency set. Note key segments resembling file paths (containing `.`) are treated as files-to-hash and fail if absent unless quoted (verify against the currency brief for your version).

- **`Cache@2` cost/scope misjudged — small deps, wrong eviction expectations, or branch scope** (lane `data-access` / `payload-startup`): the cache round-trip is download + extract on restore and compress + upload on save (a *post-job* step), so for a small or fast-to-rebuild dependency set the cache can cost more wall-time than reinstalling — measure restore time vs. rebuild before adding it. Caches expire after **7 days of inactivity** and there's *no enforced size limit* (unlike GitHub's 10 GB repo cap), so the eviction model differs — don't import GitHub assumptions. Scope is per-pipeline/branch: a PR run *reads* the target/`main` cache but *writes* only to its intermediate ref, and a feature branch cold-restores from `main` on first run by design — a cross-branch "miss" is often correct, not a bug (verify against the currency brief for your version).

- **`checkout` doing an unnecessary deep fetch or tag sync every run** (lane `data-access` / `payload-startup`): the implicit `checkout: self` is the per-run git cost. New pipelines (post-Sept-2022) default to **shallow fetch depth 1**, so the footgun is an explicit `fetchDepth: 0` (full history) left on globally for one job that needed `git describe`/changelog/blame — that re-pays a full-history clone on a large repo every run. The two-directional audit: a `0` with no history consumer is waste; a history-consuming tool on the shallow default is a latent correctness bug. Separately, **tag syncing** fetches all tags and their objects and *defeats shallow fetch* — its default is `fetchTags: true` for pipelines created before the Sprint 209 change and `false` for newer ones, so an older tag-heavy pipeline benefits from explicit `fetchTags: false` (and `clean: true`, the default, re-pays `git clean`/`reset` each run on self-hosted) (verify against the currency brief for your version).

- **No path/branch filter, so the whole pipeline runs on every push** (lane `algorithmic`): the cheapest minute is the one never billed. An explicit `trigger` *replaces* the implicit "every branch" CI trigger; `trigger.paths` include/exclude skips a docs-only or single-package change in a monorepo, and `pr.paths` does the same for PR validation. `trigger: none` / `pr: none` disable a trigger entirely for a pipeline that shouldn't fire on every push (e.g. a manually/scheduled-only deploy). Caveats worth verifying: trigger paths are *case-sensitive*; an `exclude` without `include` means "all except"; and you can't re-`include` an excluded path unless you qualify a deeper subfolder. Cross-reference **Redundant & repeated work** in `../cicd.md` (verify against the currency brief for your version).

- **`trigger.batch` absent on a chatty branch; PR `autoCancel` not relied on** (lane `concurrency` / `algorithmic`): these attack trigger *frequency*, the biggest lever. `trigger.batch: true` coalesces rapid pushes — while a run is in progress, later pushes are held and built together in one run when it finishes, instead of one run per push (note: `batch` is *not* supported in repository-resource triggers, and with multi-stage/approval pipelines the run must reach a terminal state first, so batching there can stall — split build from deploy). For PRs, `pr.autoCancel` already defaults to **`true`**, cancelling an in-progress PR run when a new commit lands — so the real defect is someone having set it to `false` and burning agents on superseded commits; `pr.drafts` (default `true`) is the other knob — set it `false` to skip building draft PRs. Note YAML `pr` triggers don't apply to Azure Repos Git (use branch-policy build validation there) (verify against the currency brief for your version).

- **No `strategy.matrix` for version/config fan-out, or tests not sliced across agents** (lane `concurrency`): `strategy.matrix` dispatches one job per variable set (OS/version/shard) from a single definition — testing N configs serially is N× the matrix wall-time — and `maxParallel` caps simultaneous legs to fit purchased capacity (`matrix` and `parallel` are mutually exclusive; `maxParallel` only applies to `matrix`). For a long single test suite, `strategy.parallel: N` duplicates the job N times and injects `System.JobPositionInPhase`/`System.TotalJobsInPhase` so a slicing-aware test runner splits the suite across agents — cutting the longest job, which is what the critical path sees. Weigh per-slice fixed cost (fresh agent + checkout + install) against time saved; past a point more slices add more setup than they remove (verify against the currency brief for your version).

- **A container job or heavy custom image paying an image-pull cold start every run** (lane `payload-startup`): a `container:` job or step pulls its image on each run (the agent is ephemeral), so a fat custom image vs. a `UseNode`/`UsePythonVersion`-style task on a bare agent is a real startup-cost trade-off. For Docker *builds*, Azure has no built-in layer cache: either prime BuildKit's local layer cache through `Cache@2` (`--cache-from`/`--cache-to type=local` over a cached `path`) or build in ACR — without one, every run rebuilds all layers from scratch. The Docker-image cache lives in `Cache@2`, separate from the dependency cache and from artifacts; see the Docker module for layer-ordering. Cross-reference **Image, artifact & cold-start cost** in `../cicd.md` (verify against the currency brief for your version).

- **Templates and `condition` not used to skip stages/jobs that don't apply** (lane `algorithmic`): `extends`/`template` factor shared pipeline shape, but the perf lever is pairing them with `condition` so a stage/job that can't apply to this run is *skipped*, not run-and-no-op'd — e.g. `condition: eq(variables['Build.SourceBranch'], 'refs/heads/main')` to keep a deploy stage off feature branches, or `Build.Reason`-based conditions to skip steps on PR vs. CI runs. Gotcha to verify: a custom `condition` *replaces* the implicit `succeeded()` guard, so the idiom is `and(succeeded(), <custom>)` — a bare custom condition runs the stage even after an upstream failure, doing avoidable work. A skipped stage consumes no parallel job; a no-op stage that still spins an agent does (verify against the currency brief for your version).

## Sources

Authoritative Azure DevOps documentation fetched for this module (learn.microsoft.com):

- **Pipeline caching / `Cache@2`** — "Pipeline caching" (`/pipelines/release/caching`): `key`/`restoreKeys`/`path`, `|`-separated key segments and lockfile hashing, file-path-segment hashing gotcha, immutability ("once created, content can't be modified"; fixed key can't update — use `restoreKeys`), prefix-fallback returning the most recent match, cache-vs-artifacts guidance, branch/PR cache scopes, 7-day inactivity expiry, no enforced size limit, no clear-cache (bump a literal segment), the Docker/buildx `--cache-from`/`--cache-to type=local` example.
- **Stages** — "Stages in Azure Pipelines" (`/pipelines/process/stages`): stages run sequentially by default in declaration order, `dependsOn` for fan-out/fan-in, `dependsOn: []` runs a stage in parallel immediately, custom `condition` removing the implicit success guard (`and(succeeded(), …)`), `trigger: manual`, `isSkippable`, 256-jobs-per-stage.
- **Jobs** — "Jobs in Azure Pipelines" (`/pipelines/process/phases`): jobs parallel-by-default unless `dependsOn` set, fresh agent per job on Microsoft-hosted, one job per agent at a time, `strategy.matrix`/`maxParallel`/`parallel` (mutual exclusivity), slicing via `System.JobPositionInPhase`/`System.TotalJobsInPhase`, default 60-min job timeout, `demands`.
- **Configure and pay for parallel jobs** — "Configure and pay for parallel jobs" (`/pipelines/licensing/concurrent-jobs`): each running job consumes one org-wide parallel job; over-fan-out queues; private free grant = 1 Microsoft-hosted job, 60-min cap, 1,800 min/month (free grant temporarily gated behind a request); parallel jobs shared across all projects; self-hosted charged per concurrent job not per agent.
- **Triggers** — "Triggers in Azure Pipelines" (`/pipelines/build/triggers`): implicit CI/PR triggers enabled by default, `trigger: none` / `pr: none` to disable, an explicit `trigger` replaces the implicit one.
- **`pr` schema** — "pr definition" (`/pipelines/yaml-schema/pr`): `autoCancel` default `true` (cancel running PR builds on new commit), `drafts` default `true`, `branches`/`paths` include/exclude, YAML `pr` unsupported on Azure Repos Git (use branch-policy build validation).
- **Azure Repos Git triggers** — "Build Azure Repos Git repositories" (`/pipelines/repos/azure-repos-git`): `trigger.batch: true` coalesces pushes during an in-progress run (not supported in repository-resource triggers; multi-stage caveat), `paths` include/exclude (case-sensitive, implicit root include, exclude-then-re-include rule), default checkout command `git fetch --force --tags --prune --depth=1`.
- **`checkout` schema** — "steps.checkout definition" (`/pipelines/yaml-schema/steps-checkout`): default `checkout: self`, shallow fetch depth 1 default for new pipelines (Sprint 209, Sept 2022), `fetchDepth: 0` = full history overriding shallow, `fetchTags` default `true` (pre-209) / `false` (post-209) and that tag sync defeats shallow fetch, `clean` default true, `submodules`/`lfs` default off, sparse-checkout options.
