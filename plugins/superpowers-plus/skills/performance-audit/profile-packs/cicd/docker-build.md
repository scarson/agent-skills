## Container image builds (Dockerfile / BuildKit)

> Scope: the performance of *building* a container image and the cost the resulting image imposes downstream â€” build wall-time, layer cache hit rate, and final image size (which drives registry push/pull time and container cold-start). The dominant theme is that **each Dockerfile instruction is a cache layer keyed on the instruction text plus its inputs, and changing one layer invalidates every layer after it** â€” so the *order* of instructions, not just their content, determines how much of a build is reused. Two failures account for most slow CI image builds and are the first things to land: (1) the dependency-install layer is busted on every commit because source is copied before deps are installed, and (2) the ephemeral CI runner has no persistent layer cache and nothing is imported, so caching does nothing. Everything else here is depth on top of those two. Cross-reference the core lanes and the **Pipeline cost model** in `../cicd.md` (the runner is ephemeral; build time is on the critical path of every push); the buildx GitHub Actions cache backend is noted here but the runner/workflow mechanics live in the sibling [`github-actions.md`](github-actions.md).

- **COPY-then-install ordering is the highest-leverage cache fix.** A layer's cache is keyed on the instruction and its inputs, and busting a layer busts every layer after it â€” so `COPY . .` *before* `RUN <install deps>` means **any** source change (a one-character code edit) invalidates the COPY layer and forces the slow dependency install to re-run every build. Copy only the lockfile/manifest first (`COPY package.json package-lock.json ./`, `COPY requirements.txt ./`, `COPY go.mod go.sum ./`), run the install, *then* `COPY . .` the source â€” so the expensive install layer stays cached across code-only commits and reruns only when the lockfile actually changes. This single reordering is usually the largest build-time win and is the first thing to check in any Dockerfile.

- **A frequently-changing instruction placed early invalidates everything after it.** Cache invalidation cascades forward, so the durable ordering rule is **least-frequently-changing â†’ most-frequently-changing**: base image and system packages first, dependencies next, application source last. An `ARG`/build-timestamp/commit-SHA, or a `COPY` of fast-moving files, placed near the top busts the entire tail on every build. Docker's own guidance places expensive/rarely-changing steps early and frequently-changing steps near the end, so a downstream change can't force the costly layers to rebuild. Read the Dockerfile top-to-bottom and flag any volatile input sitting above an expensive layer (verify exact mechanics against the currency brief for your version).

- **No `.dockerignore` bloats the build context and silently busts the cache.** The build context is uploaded to the daemon/builder on every build; without a `.dockerignore`, `COPY . .` sweeps in `.git`, `node_modules`, `target/`, `dist/`, local virtualenvs and build output â€” inflating the upload *and* changing the COPY layer's inputs (so it cache-misses) whenever any of that churns. Confirm a `.dockerignore` exists and excludes VCS metadata, dependency dirs, and build artifacts; its absence is both a context-size and a cache-hit problem at once (verify against the currency brief for your version).

- **A huge build context is paid on every build, before any layer runs.** Even with perfect layer caching, BuildKit must transfer the context to the builder first â€” building from a deep/monorepo root, or from a directory containing large data/asset trees, uploads gigabytes each run. The fix is narrowing the context (a tighter `.dockerignore`, or pointing the build at a subdirectory) rather than tuning layers. Flag when the build root is far larger than what the image actually needs; this matters most with remote/ephemeral builders where the transfer crosses the network.

- **Missing multi-stage build ships the entire toolchain in the final image.** Compilers, build-time dev headers, and package caches that exist only to *produce* the artifact have no business in the runtime image â€” they bloat it, slowing every registry pull and container cold-start. Use a builder stage (`FROM â€¦ AS builder`) and `COPY --from=builder` only the compiled artifact (or installed site-packages) into a slim runtime stage; per Docker, "None of the build tools required to build the application are included in the resulting image." Reference stages by name (`AS builder`) so reordering doesn't break `--from`. Flag any single-stage Dockerfile that installs a build toolchain and then runs the app from that same image.

- **BuildKit cache mounts persist a package manager's download cache without baking it into a layer.** `RUN --mount=type=cache,target=â€¦` gives the install step a persistent scratch cache that is *not* part of the image and survives across builds â€” so even when the install **layer** is cache-busted (lockfile changed), only new/changed packages are re-downloaded rather than the whole set re-fetched. Docker: "even if you need to rebuild a layer, you only download new or changed packages." Targets are tool-specific (`/root/.cache/pip`, `/root/.npm`, `/go/pkg/mod`, and `/var/cache/apt`+`/var/lib/apt` with `sharing=locked` because apt needs exclusive access). This is the modern complement to COPY-ordering: ordering keeps the layer cached when nothing changed; the cache mount blunts the cost when it *did*. Note that in ephemeral CI the cache-mount contents must themselves be persisted/imported or they start empty too (verify flags against the currency brief for your version).

- **On an ephemeral CI runner, layer caching does nothing unless cache is imported.** A fresh runner starts with no local layer cache, so every build is fully cold regardless of how well the Dockerfile is ordered â€” Docker calls external cache "almost essential in CI/CD build environments" precisely because they have "little-to-no persistence between runs." Use `docker buildx build --cache-from`/`--cache-to` with a **registry** backend (`type=registry,ref=â€¦`) or, on GitHub-hosted runners, the **gha** backend (`type=gha`) to export/import the layer cache across runs. The absence of any `--cache-from` on a CI build is a red flag that the layer-caching work is wasted; cross-reference the Pipeline cost model in `../cicd.md` (ephemeral runner â‡’ no carried state). The Actions-side wiring lives in [`github-actions.md`](github-actions.md) (verify backend syntax against the currency brief for your version).

- **Base image choice dominates final image size and pull time.** The base is typically the largest single contributor to image size, which sets registry pull and cold-start cost: a full `python:3.x`/language base (hundreds of MB) vs `-slim` (drops compilers, dev headers, and docs) vs `alpine` (Docker notes its base is "under 6 MB") vs `distroless`/`scratch` (runtime libs + app only â€” no shell or package manager, per Docker's distroless guidance). Picking the smallest base the app actually runs on is a direct, durable win on every pull. Flag a full-fat base where a slim/distroless runtime stage would carry the same app (verify current tags/sizes against the currency brief for your version).

- **Alpine's size win can be a net loss for Python (musl vs glibc).** Alpine uses musl libc, and PyPI binary wheels are built for glibc â€” so packages with C extensions often **compile from source** on Alpine instead of installing a prebuilt wheel, dragging build time the opposite direction from the size goal â€” the canonical pythonspeed measurement is a `pip install` of matplotlib+pandas at ~30s on Debian-slim vs ~1557s on Alpine. musl also differs in DNS resolution and default thread stack size, which can surface as runtime bugs. The durable judgment: Alpine is a natural fit for statically-linked Go/Rust binaries, but for a Python (or other glibc-wheel) stack, `-slim` usually wins overall despite being larger. Don't recommend Alpine purely on the base-image size number (verify wheel/`musllinux` availability against the currency brief for your version).

- **Combine `apt-get update && install && rm -rf /var/lib/apt/lists/*` in one RUN â€” but don't over-combine.** Splitting `apt-get update` into its own `RUN` lets Docker reuse a stale update layer while the install layer changes, installing outdated packages; Docker's rule is to "always combine `RUN apt-get update` with `apt-get install` in the same `RUN` statement," and to delete the apt lists in that same layer so the cache isn't frozen into the image. The counter-pressure: chaining *too much* into one `RUN` collapses cache granularity, so an unrelated change re-runs the whole block. Name the trade-off â€” combine what must stay consistent (update+install+cleanup), keep independently-changing concerns in separate layers (verify against the currency brief for your version).
