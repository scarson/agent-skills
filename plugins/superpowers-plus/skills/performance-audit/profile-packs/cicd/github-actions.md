# CI/CD performance module: GitHub Actions
> Load when `.github/workflows/*.yml` is detected — see the module map in `../cicd.md`. Core lanes + the Pipeline cost model live in `../cicd.md`; this file is the GitHub Actions lens only.

## GitHub Actions

> Scope: workflow YAML under `.github/workflows/` — the trigger/filter layer, the job DAG, the matrix, the cache, and the runner choice. The recurring theme is that GitHub Actions wall-time and cost are dominated by three levers that live *outside* the steps a script runs: **don't run it at all** (trigger and path scoping — the cheapest minute is the one never billed), **don't run it twice** (caching and the fresh-per-job runner, which silently re-pays checkout + install on every job), and **don't run it serially** (the `needs` DAG and the matrix, which decide how much fans out). Billing multiplies all of this — Linux/Windows/macOS minutes are not priced equally — so a wasteful macOS lane costs roughly an order of magnitude more than the same waste on Linux (verify against the currency brief for your version). The cross-cutting **Pipeline cost model** and the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) live in `../cicd.md`; this file is the GitHub Actions provider lens — defer the lane definitions there and read this for the provider-specific surface.

- **Concurrency group missing on PR-triggered workflows** (lane `concurrency`): without a `concurrency:` block with `cancel-in-progress: true`, every push to an open PR queues a *new* run while the superseded one keeps burning a runner to completion — pure waste, since only the latest commit's result is ever read. A group keyed on `${{ github.workflow }}-${{ github.ref }}` cancels the in-flight run of the same workflow on the same ref when a newer one queues. This is the single highest-ratio fix for chatty PR branches; the deliberate exception is `deploy`/release workflows, where cancelling a half-finished deployment is *worse* than letting it finish, so those groups should NOT cancel (verify against the currency brief for your version).

- **`fail-fast` set against the run's actual purpose** (lane `concurrency`): in a `strategy.matrix`, `fail-fast: true` (the default) cancels every in-progress and queued matrix job the moment one variant fails — saving minutes on a *gating* CI run where one red cell already blocks merge. But on a run whose job is to report *which* of N variants/OSes broke (nightly compatibility, flaky-test triage), `true` throws away the signal you're paying to collect, forcing a re-run. The judgment is run-purpose, not a global default — confirm what the maintainer reads from the run before recommending either value (verify against the currency brief for your version).

- **`needs` modeled as a chain where the work is a fan-out** (lane `concurrency`): jobs run in parallel by default; `jobs.<id>.needs` exists only to express a *real* prerequisite (job B consumes job A's artifact, or must not start until A's gate passes). A `lint → test → build` chain where lint and test share no data forces ~3 cold runners end-to-end when all three could start at once; the inverse error is one mega-job doing lint+test+build sequentially on a single runner, leaving every other runner idle. Read the data dependencies, not the author's mental ordering, and only keep a `needs` edge that a real consumer justifies. Cross-reference the **Pipeline cost model** in `../cicd.md` for the critical-path framing.

- **Path filters absent on a monorepo or docs-heavy repo** (lane `algorithmic` / `data-access`): `paths`/`paths-ignore` (and `pull_request`/`push` branch filters) decide whether the workflow starts *at all* — the "don't run it" win that beats any in-run optimization. A full CI matrix firing because a README changed, or every package's tests running when one package changed, is wasted minutes the filter removes. Note `paths` and `paths-ignore` are mutually exclusive on the same event; complex include/exclude needs `paths` with `!`-negation, where a later negative pattern excludes and a later positive re-includes (order matters). For per-job (not per-workflow) skipping inside a monorepo, a change-detection action is the idiom (`dorny/paths-filter` or equivalent). Cross-reference **Redundant & repeated work** in `../cicd.md` (verify against the currency brief for your version).

- **Built-in `setup-*` caching left unused in favor of a hand-rolled `actions/cache`** (lane `idiom-currency`): `actions/setup-node`/`setup-python`/`setup-java` expose a `cache:` input that caches the package manager's global store keyed on the detected lockfile automatically (it calls `actions/cache` under the hood) — the modern, less error-prone path. A workflow carrying a hand-written `actions/cache` block to do the same thing is both more code and more ways to get the key wrong; prefer the built-in unless the cache target is something `setup-*` doesn't cover. Note `cache:` caches the *download/global* store, not `node_modules`/the venv itself — the install step still runs but skips network fetches; flag a claim that it skips install entirely (verify against the currency brief for your version).

- **Cache key that is stale or always-miss** (lane `data-access`): a manual `actions/cache` only pays off when the key changes *exactly* when the cached contents should. A key that never changes (e.g. a static string) serves stale dependencies and silently diverges from the lockfile; a key embedding `${{ github.sha }}` or a timestamp changes every run and never hits, so you pay full restore-and-save overhead for zero reuse. Correct shape is `key` on `hashFiles(<lockfile>)` plus `restore-keys` as a prefix to fall back to the most recent near-match on a partial dependency change. Both failure modes look "cached" in the YAML — verify the key actually tracks the dependency set (verify against the currency brief for your version).

- **`restore-keys` and cache scope misunderstood** (lane `data-access`): `restore-keys` does *prefix* matching and returns the most recently created cache among partial matches — so it only helps if keys share a stable prefix before the hash. Scope is also branch-bounded: a run restores caches created on its own branch and the default/base branch, **not** from arbitrary sibling feature branches — so a feature branch's first run after a cache change always cold-restores from `main`, by design. Repo cache storage is capped (10 GB default) with least-recently-accessed eviction and a 7-day idle-eviction, so an oversized or churny cache evicts the entries other workflows depend on. A cross-branch cache "miss" is often correct behavior, not a bug — confirm before flagging (verify against the currency brief for your version).

- **Caching small/cheap dependencies, or treating artifacts as a cache** (lane `payload-startup`): restoring a cache is itself download + decompress + (on save) compress + upload; for a small or fast-to-rebuild dependency set the cache round-trip can cost *more* wall-time than just reinstalling — measure restore time against rebuild time before adding a cache. Separately, `upload-artifact`/`download-artifact` are for moving data *between jobs in one run* or *out of a run* (test reports, build outputs), **not** a cross-run cache — using artifacts to carry deps run-to-run is the wrong tool (use caching); and a large artifact passed between jobs may cost more to upload+download than recomputing it in the consuming job. Weigh transfer size against recompute cost (verify against the currency brief for your version).

- **`upload-artifact`/`download-artifact` on a deprecated major or with default retention** (lane `idiom-currency`): the v4 line changed behavior — artifacts are immutable, so you can no longer append to one artifact name across multiple jobs (matrix uploads need a per-cell name suffix), and v3 reached deprecation (artifact v3 closed down end-of-2024) so a workflow still on v3 will eventually fail outright, not just run slow. Cost levers: `retention-days` (default 90) holds storage you may not need, and `compression-level: 0` is the faster choice for already-compressed payloads. Treat the exact version status as currency-sensitive — GitHub has retired action majors and runner images on a schedule (verify against the currency brief for your version).

- **`fetch-depth: 0` set without a consumer that needs history** (lane `payload-startup`): `actions/checkout` defaults to `fetch-depth: 1` (a shallow single-commit clone) precisely because full history is expensive on a large repo; a workflow that sets `fetch-depth: 0` pays a full-history clone on *every* run. That cost is justified only when a step genuinely needs history or tags — `git describe`, changelog/version derivation, SonarQube blame, `nx affected`/`turbo` since-ref diffs — and conversely such a tool silently misbehaves on the shallow default. The audit is two-directional: a `0` with no history consumer is waste to remove; a history-consuming tool on the default depth is a latent correctness bug, not a perf one (verify against the currency brief for your version).

- **Per-job re-checkout and re-install treated as free** (lane `data-access` / `payload-startup`): each job runs on a *fresh, isolated runner VM*, so checkout, toolchain setup, and dependency install repeat in full for every job in the DAG — five jobs that each `setup-node` + `npm ci` pay that cost five times. Levers, in order of preference: collapse jobs that have no reason to be separate (one job, shared setup), share an expensive build output between jobs via an artifact instead of rebuilding it per job, or — if the duplication is unavoidable — ensure each job's dependency cache actually hits so only the cache restore (not the network install) repeats. The fresh-VM model is the structural reason "just add another job" is never free (verify against the currency brief for your version).

- **Runner choice and container-job cold-start as cost levers** (lane `payload-startup`): the runner OS multiplies the bill — standard Linux is the 1x baseline, Windows ~2x, macOS roughly 10x per consumed minute — so a job that doesn't *need* macOS/Windows (anything not building Apple/Windows artifacts) is paying a large multiplier for nothing; pin the OS to the cheapest one that satisfies the job. Larger runners cut wall-time on parallelizable CPU/RAM-bound work but bill at a higher per-minute rate, so they win only when the wall-time saving outweighs the rate (Amdahl-bounded — a serial bottleneck won't speed up). And a container `job`/`container:` step pays an image-pull cold start each run, so a heavy custom image vs a `setup-*` action on a bare runner is a real startup-cost trade-off. Cross-reference the **Pipeline cost model** in `../cicd.md` for wall-time-vs-billed-minutes (verify against the currency brief for your version).
