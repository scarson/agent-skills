# CI/CD performance module: GitLab CI/CD
> Load when `.gitlab-ci.yml` is detected — see the module map in `../cicd.md`. Core lanes + the Pipeline cost model live in `../cicd.md`; this file is the GitLab CI/CD lens only.

## GitLab CI/CD

> Scope: `.gitlab-ci.yml` (plus `include:`d fragments and `extends`/`!reference` chains) — the pipeline-creation rules, the stage/`needs` DAG, the cache-vs-artifact split, the clone strategy, and the runner pool the fan-out actually lands on. The recurring theme mirrors GitHub Actions but the primitives differ: GitLab's wall-time and runner-minute cost are dominated by **don't create the pipeline at all** (`workflow:rules` and `rules:changes` — the cheapest minute is the one never queued, and GitLab's duplicate-pipeline trap *doubles* the bill silently), **don't re-fetch what you already have** (`cache:` keyed on the lockfile, shallow `GIT_DEPTH`, the dependency proxy — every job is a fresh runner that re-clones and re-installs by default), and **don't run serially when the data lets you fan out** (`needs:` breaks the default stage barrier; `parallel`/`parallel:matrix` shards). The cross-cutting **Pipeline cost model** and the core lanes (`algorithmic`=redundant work, `data-access`=caching & artifact I/O, `concurrency`=job parallelization, `payload-startup`=image/artifact/cold-start, `idiom-currency`=runner/version currency) live in `../cicd.md`; defer the lane definitions there and read this for the GitLab-specific surface.

- **Branch-pipeline + merge-request-pipeline double-run with no `workflow:rules` guard** (lane `algorithmic`): this is the highest-value GitLab-specific finding. If a job's `rules`/`only` match *both* branch pipelines (`$CI_PIPELINE_SOURCE == "push"`) and merge-request pipelines (`merge_request_event`), then a push to a branch that has an open MR creates **two** pipelines for the same commit — the whole suite runs twice, doubling wall-time and minutes, and nobody reads the duplicate. The fix is a top-level `workflow:rules` block that drops the branch pipeline when an MR is open: `if: $CI_COMMIT_BRANCH && $CI_OPEN_MERGE_REQUESTS` → `when: never`, ordered before the bare `if: $CI_COMMIT_BRANCH` fallback (and `if: $CI_PIPELINE_SOURCE == "merge_request_event"` to keep MR pipelines). Absence of *any* `workflow:` block on a repo that uses MR pipelines is the tell. Cross-reference **Redundant & repeated work** in `../cicd.md` (verify against the currency brief for your version).

- **Default sequential `stages:` where the work is a DAG** (lane `concurrency`): without `needs:`, every job waits for the *entire* previous stage to finish before any job in the next stage starts — so one slow job in `test` blocks all of `deploy` even where no data flows between them. `needs:` lets a job start the moment *its named upstreams* finish, regardless of stage, turning the pipeline into a DAG whose wall-time is the **critical path**, not the sum of stage durations. The two distinct wins: `needs: [build]` on a job that only consumes the build (skip waiting for sibling test jobs), and `needs: []` on a job with no upstream at all (lint, secret-scan) so it starts immediately at pipeline launch instead of waiting out an irrelevant earlier stage. Verify independence first — a `needs:` edge also controls which artifacts download, so dropping a real one breaks the consumer. Cross-reference the **Pipeline cost model** in `../cicd.md` for the critical-path framing (verify against the currency brief for your version).

- **`cache:key` that never invalidates or never hits** (lane `data-access`): caching is **off by default** — a job with no `cache:` block re-downloads every dependency every run. But a present cache pays off only when `key` tracks the dependency set exactly. A static string key serves stale deps after the lockfile changes; a key embedding the commit SHA changes every run and never hits, so you pay restore+save overhead for zero reuse. The idiom is `cache:key:files` pointed at the lockfile (`package-lock.json`, `Gemfile.lock`, `poetry.lock`), which derives the key from those files' contents so it invalidates precisely when deps move. Note GitLab also splits cache by protected vs non-protected branch (separate `-protected`/`-non_protected` keys by default) and by the runner's local/distributed storage — a "miss" on a feature branch's first run or across un-shared runners can be correct, not a bug. Confirm the cache is *served*, not merely *written* (verify against the currency brief for your version).

- **`cache:` used where `artifacts:` is meant, or vice versa** (lane `data-access`): they are different tools and swapping them re-pays I/O every run. `cache:` is for regenerable dependencies, stored runner-side, and persists *across* pipelines; `artifacts:` move job outputs *within one pipeline* (downstream jobs pull them via `needs`/`dependencies`) and are stored in GitLab. Using `artifacts:` to carry `node_modules` from run to run is the classic mistake — artifacts aren't a cross-pipeline cache, so it re-uploads and re-downloads the whole tree every pipeline instead of restoring a runner-local cache. The inverse — caching a build output that a *single* downstream job needs *this* pipeline — should be an artifact. Also scope `dependencies:` (or `needs:`-artifacts) so a job downloads only the artifacts it consumes, not every upstream artifact by default (verify against the currency brief for your version).

- **`cache:policy` left at `pull-push` on jobs that only read** (lane `data-access`): the default `pull-push` policy makes *every* job both restore the cache and re-upload it at the end. A job that only consumes dependencies (a test shard, a lint job) gains nothing from the upload and pays compress+transfer on every run — set `policy: pull` there and reserve `policy: push` (or `pull-push`) for the one job that actually populates the cache (the install/build job). On a wide `parallel` fan-out this is multiplied across every shard, so the wasted upload is N× (verify against the currency brief for your version).

- **`rules:changes` absent on a monorepo or docs-heavy repo** (lane `algorithmic`): `rules:changes` (and legacy `only:changes`) decides whether a job is *included in the pipeline at all* based on which paths the diff touched — the "don't run it" win that beats any in-run optimization. A full build+test matrix firing because a README changed, or every package's jobs running for a one-package change, is minutes the path scope removes. Two judgment points: on a branch pipeline with no MR, `changes` compares against the previous commit, which can behave surprisingly on a new branch or force-push — pin the base with `rules:changes:compare_to` where determinism matters; and a *skipped* required job can block an MR unless paired with the right `when`/allow rules. Cross-reference **Redundant & repeated work** in `../cicd.md` (verify against the currency brief for your version).

- **No `interruptible: true` + auto-cancel, so superseded pipelines run to completion** (lane `concurrency`): when a newer commit lands on a branch, the in-flight pipeline for the old commit is now pointless, yet by default it keeps burning runners to the end. `interruptible: true` on jobs plus auto-cancel (project "Auto-cancel redundant pipelines", or `workflow:auto_cancel:on_new_commit`) cancels the superseded run. The values matter: `conservative` (the default) cancels only if no `interruptible: false` job has started yet — and once any non-interruptible job starts, the *whole* pipeline stops being cancellable; `interruptible` cancels just the `interruptible: true` jobs; `none` disables it. The deliberate exception is `deploy`/release jobs, which should be `interruptible: false` so a half-finished deployment is never cut off. Mark fast, idempotent test/build jobs interruptible; leave deploys alone (verify against the currency brief for your version).

- **A long suite run serially instead of `parallel` / `parallel:matrix` sharding** (lane `concurrency`): `parallel: N` (integer 1–200) spawns N instances of one job, each seeing `$CI_NODE_INDEX`/`$CI_NODE_TOTAL` so a test splitter divides the suite — the single longest job, which is what the critical path sees, drops to ~1/N. `parallel:matrix:` instead fans out across variable combinations (OS × version × suite) as the Cartesian product from one definition, replacing N copy-pasted jobs. Weigh the per-shard fixed cost: each shard is a fresh runner paying its own clone + install + cache restore + cold start, so past a point more shards add more setup overhead than they remove from the suite. Tune shard count against measured per-shard timing, not intuition (verify against the currency brief for your version).

- **Unnecessary full clone (`GIT_DEPTH: 0`) on every job** (lane `data-access`): GitLab CI clones **shallow by default** (`GIT_DEPTH` defaults to a small depth, e.g. 20) precisely because full history is expensive on a large repo. Setting `GIT_DEPTH: "0"` forces a full-history clone on *every* job of *every* run — justified only when a step genuinely needs history (a `git describe`-driven version, changelog generation, blame-based analysis), and conversely such a tool silently misbehaves on the shallow default. The audit is two-directional: a `0` with no history consumer is waste to remove; a history-consuming tool on the shallow default is a latent correctness bug, not a perf one. Pair with `GIT_STRATEGY`: `clone` re-clones from scratch each job, `fetch` reuses the runner's working copy (faster on a persistent runner), `none` skips git entirely for jobs that only need a prior job's artifacts — a job doing pure artifact processing on `GIT_STRATEGY: clone` is paying for a checkout it never reads (verify against the currency brief for your version).

- **Docker-in-Docker builds with no layer cache, re-pulling base images every run** (lane `payload-startup` / `data-access`): each dind job gets a *fresh, empty* Docker engine, so layer caching across runs is gone by default — every `docker build` rebuilds from scratch and every `FROM` re-pulls the base image. The registry-as-cache idiom restores it: `docker pull $CI_REGISTRY_IMAGE:latest || true` then `docker build --cache-from $CI_REGISTRY_IMAGE:latest`, and crucially the image must be built with `--build-arg BUILDKIT_INLINE_CACHE=1` for the pulled image to *carry* usable cache metadata (an image used with `--cache-from` must be pulled first). For heavier builds, `docker buildx` with `--cache-to type=registry`/`--cache-from type=registry` stores a dedicated cache image. Separately, route base-image pulls through the **dependency proxy** (`$CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX/<image>`) as a pull-through cache so jobs stop re-fetching upstream layers from Docker Hub (and dodge its rate limits). See the Docker module for Dockerfile-side layer ordering (verify against the currency brief for your version).

- **Copy-pasted heavy job setup instead of `extends` / `!reference` / `include`** (lane `idiom-currency` / `algorithmic`): GitLab's reuse primitives don't change wall-time per run directly, but duplicated job bodies drift — a cache key, a `rules:` guard, or an `interruptible` flag fixed in one copy and not the others quietly reintroduces the double-run, the always-miss cache, or the uncancellable pipeline this lens is built to catch. `extends:` merges a base job, `!reference` splices a fragment of one key into another, and `include:` pulls shared config across files. The performance angle is consistency of the *other* fixes: audit whether every test/build job inherits the same correct cache/`rules`/`needs` shape, or whether one hand-edited copy is silently re-paying the cost (verify against the currency brief for your version).

- **Declared `parallel`/fan-out exceeding real runner throughput** (lane `concurrency`): `parallel: 50` against a runner pool (or autoscaling cap) that executes far fewer concurrent jobs just *queues* — the extra shards add per-shard setup overhead and wait time without shrinking wall-time, since concurrency only helps up to available capacity. Runner **tags** also pin jobs to a specific pool, so an over-tagged job lands on a small dedicated fleet and serializes regardless of the `parallel` number. Right-size fan-out to the pool that will actually run it, and weigh shared/elastic runners (per-minute, scales) against a fixed self-hosted fleet (capacity-bounded, queues). The YAML's declared parallelism is an upper bound, not the achieved one. Cross-reference the **Pipeline cost model** in `../cicd.md` (verify against the currency brief for your version).
