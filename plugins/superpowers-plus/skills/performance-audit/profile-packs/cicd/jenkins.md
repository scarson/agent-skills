# CI/CD performance module: Jenkins
> Load when a `Jenkinsfile` (declarative or scripted Pipeline) is detected — see the module map in `../cicd.md`. Core lanes + the Pipeline cost model live in `../cicd.md`; this file is the Jenkins lens only.

## Jenkins

> Scope: a `Jenkinsfile` (declarative `pipeline {}` or scripted) — the stage graph, the `agent` allocation, the `options` block, the checkout, and the way data crosses stages. The recurring Jenkins-distinct theme is that Jenkins is **plugin-driven and executor-bound**, not minute-billed like a hosted SaaS: the scarce resource is an **executor slot on an agent**, and the dominant waste modes are *holding an executor while doing nothing* (a paused `input`, a `sleep`, a poll) and *doing real work in Groovy on the controller* (the CPS interpreter, the built-in node, `stash` routing through the controller) instead of in a `sh`/`bat` step on the agent. Jenkins also has **no first-class dependency cache** the way GitHub Actions does — reuse rides on the persistent per-node workspace or an external store, so the ephemeral-runner assumption from the core pack is *weaker* here and the audit changes accordingly. Because Jenkins is plugin-driven there are **few hard version pins** — most of this is durable idiom; the concrete step/option/plugin names are verify-tagged. The cross-cutting **Pipeline cost model** and the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) live in `../cicd.md`; this file is the Jenkins provider lens — defer the lane definitions there and read this for the provider-specific surface.

- **Independent stages left serial when they could be `parallel`** (lane `concurrency`): stages in a `stages` block execute *sequentially by default* — a multi-platform build, a version matrix, or independent test suites run one after another unless wrapped in a `parallel {}` block (declarative) or `parallel(...)` (scripted), so wall-time is the *sum* of stages where it could be the *critical path* through them. The judgment is real independence, not the author's stage ordering: a stage that consumes a prior stage's `stash`ed output or shared workspace state is not free to parallelize. Note `parallel` branches each grab their *own* executor, so a wide `parallel` can just queue against a small agent pool rather than speeding anything up (cross-reference **Job parallelization** in `../cicd.md` on runner-throughput as the real cap) (verify against the currency brief for your version).

- **An executor held during a manual `input` / approval** (lane `concurrency`, DEFEND): the `input` step *pauses Pipeline execution* waiting for a human, which can be minutes-to-days. If that wait happens while an `agent`/`node` is allocated, the executor stays occupied the whole time, starving every other queued build — the single most damaging Jenkins executor-occupancy footgun. The durable fix is to *not hold a node across the wait*: declarative deliberately evaluates `input` **before** allocating the stage's agent, and `agent none` at pipeline level holds no global executor — both exist so a paused approval consumes nothing. In scripted, the equivalent idiom is to place `input` *outside* the `node {}` block. Flag an approval stage that sits inside an allocated agent (verify against the currency brief for your version).

- **A long `sleep` / external poll holding the executor** (lane `concurrency`, DEFEND): the same occupancy defect as `input`, less obvious — a stage that `sleep`s waiting for a deployment to settle, or loops polling an external system (quality gate, smoke check) inside an allocated `agent`, burns an executor slot doing nothing. The cost is the *held executor against pool capacity*, not CPU. Where the wait is unavoidable, the question is whether it must hold a node at all; some waits belong outside any agent allocation, same as `input` (verify against the currency brief for your version).

- **Heavy Groovy on the controller via the CPS interpreter** (lane `algorithmic` / `payload-startup`): scripted Pipeline (and the Groovy glue in declarative) runs through the **CPS** transform on the *controller*, which is markedly slower than native Groovy because it provides durability (survivability across restarts) at the cost of an interpreting execution model — so a large loop, data-munging, or JSON/XML processing written in Pipeline Groovy is slow *and* loads the controller for every build. The durable rule is to keep Groovy thin and do real work in a `sh`/`bat` step **on the agent** (a helper script invoked once), reserving `@NonCPS` for the genuinely controller-side processing that can't move — and `@NonCPS` functions must not call Pipeline steps. Consolidating many small steps into one script step also cuts per-step CPS/controller overhead (verify against the currency brief for your version).

- **Build work running on the built-in (controller) node** (lane `algorithmic` / DEFEND): a stage with `agent any`/a missing agent that lands on the controller, or scripted code doing compile/test work on the built-in node, contends with the controller's own orchestration and serialization — Jenkins best practice is explicitly to *run builds on agents, not the controller*. This is a throughput defect (the controller becomes the bottleneck for every pipeline) as much as a safety one; the built-in node should carry zero or near-zero executors for real work (verify against the currency brief for your version).

- **No native dependency cache — reuse depends on workspace persistence or an external store** (lane `data-access`): unlike GitHub Actions' first-class keyed cache, Jenkins has *no built-in dependency cache*. Reuse comes from the **per-node workspace persisting between runs** (an agent that keeps its workspace reuses `~/.m2`, `node_modules`, etc.), a mounted cache volume, or an external repository manager (Artifactory/Nexus). The audit pivot: a *clean* workspace each run (a fresh ephemeral cloud agent, or an explicit `deleteDir()`/`cleanWs`) re-downloads everything — so verify *where* dependency state lives before assuming it's cached, and weigh workspace reuse (fast, but risks dirty-state correctness bugs) against a clean-but-cache-mounted agent (verify against the currency brief for your version).

- **Over-use of `stash`/`unstash` for large artifacts** (lane `data-access`): `stash` saves files for reuse on another node in the same run, but it archives them as a **compressed TAR routed through the controller** — the docs themselves say it is *designed for small files* and to consider alternatives in the **5–100 MB** range, since large files demand considerable controller CPU. Over-stashing (a built `node_modules`, a large image, a whole `target/`) puts that cost on the controller for *every* build. For large transfers prefer a shared workspace, the External Workspace Manager, or an artifact repository (Nexus/Artifactory) — and note that an S3/remote artifact-manager plugin lets stashes go agent→store directly, sidestepping the controller (verify against the currency brief for your version).

- **`checkout scm` paying a full-history clone every run** (lane `data-access`): the default checkout fetches full history on every build. The git plugin's `CloneOption` exposes the levers — `shallow: true` with a `depth` to fetch only the tip commits, `honorRefspec: true` to clone only the configured refspec ("save time, data transfer and disk space"), `noTags: true` to skip tag fetching, and a **reference repository** to seed from a local mirror. The two-directional check mirrors the GitHub `fetch-depth` one: a deep clone with no history consumer (blame, `git describe`, version-from-history) is removable waste, while a history-dependent tool on a shallow clone is a latent *correctness* bug. Separately, `skipDefaultCheckout` skips the automatic SCM checkout entirely for a stage that doesn't read source (verify against the currency brief for your version).

- **Superseded builds not cancelled — `disableConcurrentBuilds()` / `milestone()`** (lane `concurrency`): without these, an old commit's build keeps occupying an executor to completion after a newer commit lands on the same branch — wasted, since only the latest result is read (the Jenkins analog of GitHub's concurrency-cancel). `disableConcurrentBuilds()` in `options` serializes builds of one job; `milestone()` is finer — *"an older build will never be allowed to pass a milestone (it is aborted) if a newer build already passed it"*, so dropping milestones between stages aborts the superseded build at the next checkpoint. As with all cancel-on-supersede, exempt deploy/release jobs where killing a half-finished run is worse than letting it finish (verify against the currency brief for your version).

- **`agent` granularity and docker cold-start** (lane `payload-startup`): a pipeline-level `agent` holds *one* node for the whole run (including across `input`/waits — see above); pushing `agent` down to per-stage with `agent none` at top releases the executor between stages and lets stages pick the right label. A `docker`/`dockerfile` agent pays an **image pull + container cold-start** each time it's allocated — `reuseNode true` on a stage's docker agent runs the container on the *current* node's workspace instead of allocating a new node (note: declarative-only; unsupported in scripted), avoiding a re-allocation and workspace re-sync. Weigh per-stage isolation against the repeated allocation/pull cost it incurs (verify against the currency brief for your version).

- **Docker layer/build cache lost on ephemeral agents** (lane `data-access`): a docker `agent` on a torn-down cloud node starts with a *clean filesystem*, so on-disk caches (Docker layer cache, dependency caches) don't survive between runs — the same ephemeral-runner-has-no-cache problem as GitHub's, and the image-build half is the docker-build module's concern (cross-reference [`docker-build.md`](docker-build.md) for layer-cache strategy). The Jenkins-side lever is mounting a host cache into the agent via the docker `agent { args }` (e.g. `args '-v $HOME/.m2:/root/.m2'`) so caches live on the *agent host*, not the ephemeral container — useless, of course, if the agent host itself is also ephemeral (verify against the currency brief for your version).

- **`options {}` retention and fail-fast levers** (lane `payload-startup` / `concurrency`): `buildDiscarder(logRotator(...))` bounds artifact/log retention — its *absence* lets build logs and archived artifacts accumulate unbounded, bloating controller storage and slowing the UI/job over time (a slow-creep cost, not a per-run one). `parallelsAlwaysFailFast()` makes every `parallel` block abort siblings on the first failure — right for a cost-sensitive gate, wrong for a matrix whose purpose is reporting *which* variant broke (the same run-purpose judgment as GitHub's `fail-fast`, in `../cicd.md`); and `timeout(...)` caps a runaway stage from holding an executor indefinitely (verify against the currency brief for your version).
