# Cloudflare performance module: Cache API & edge caching
> Load when the Cache API (`caches.default`, `caches.open`, `cache.match`/`put`) or `fetch(..., { cf: { cacheEverything, cacheTtl } })` edge caching is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the caching lens only.

## Cache API & edge caching

> Scope: a Workers app caching at the edge to skip recompute or an origin/upstream round-trip — either by storing a `Response` you constructed via the Cache API (`caches.default.put`/`match`) or by making a `fetch` to origin cacheable via the request `cf` object (`cacheEverything`, `cacheTtl`). The two are different tools for two different things (compute-result caching vs subrequest caching), and the dominant findings are a key that **structurally never matches** (so the cache silently does nothing while you pay to write it) and reasoning about this cache as if it were global when it is **per-data-center**. The core's data-access lane already states the per-colo fact and the "no Cache API where recompute is avoidable" point — cross-reference, don't restate; this module adds the key-correctness, eligibility, TTL, and tiering depth. The store-choice question (Cache API vs the `cf` object vs KV vs a Durable Object) recurs and defers to `kv.md` rather than restating KV's model. The Workers execution model — `ctx.waitUntil`, request-scoped lifetime, per-isolate state — lives in `../cloudflare.md`.

- **A cache key built from a per-request value never hits** (lane `data-access` / `algorithmic`): the headline footgun. `cache.match`/`put` key off a `Request` (URL); if that URL carries a timestamp, a nonce, a request-id, or a volatile query param (`?t=...`, analytics/UTM params, a cache-buster), every request computes a *distinct* key, so `match` never finds the prior `put` — the expensive compute/fetch runs every single request and `put` writes entries nothing will ever read. This reads as "we added caching" while behaving as "no cache, plus the write cost." Verify the key is the resource's **cacheable identity**: normalize the URL (strip volatile params, canonicalize order) before `match`/`put`, or set `cf: { cacheKey }` to pin a custom identity. (Mirrors the generic `data-access` "cache that never hits.")

- **Reasoning about `caches.default` as a global store** (lane `data-access`): `caches.default` is scoped to the **originating data center** — contents do not replicate across colos, and `cache.delete` purges only the local colo (the core states this; cross-reference it). The consequence to audit: this is a *per-edge hit-rate* optimization, not a coherent global cache. A low-traffic colo cold-misses on its first request for a key regardless of how warm other colos are, and a `delete` leaves stale copies live everywhere else. If the code needs one coherent value worldwide, that's KV/D1, not the Cache API (cross-reference `kv.md`); if it needs purge-everywhere on update, that's the zone Cache purge API / cache tags, not `cache.delete`. Tiered Cache (below) is the lever that softens the cold-miss; it does not make the store global.

- **`cf: { cacheEverything, cacheTtl }` to cache a normally-uncacheable subrequest** (lane `data-access`): by default Cloudflare caches a `fetch` only for its default-cached file extensions/statuses and respects origin cache headers. `cacheEverything: true` treats the response as cacheable regardless of type, and `cacheTtl` (seconds) forces an edge TTL even when the origin sends none — the right lever for a **hot, slow, idempotent** origin/API call whose upstream omits cache headers. Reach for `cacheTtlByStatus` (e.g. `{ "200-299": 3600, "404": 30, "500-599": 0 }`) over a flat `cacheTtl` so a transient `5xx` or a `404` isn't pinned at the edge for an hour (verify against the currency brief for your version). `cacheEverything`/`cacheTtl` apply to `GET`/`HEAD` only.

- **Cache API vs the `cf` fetch object vs KV — the store-choice finding** (lane `idiom-currency`): three caching tools, easy to mis-pick. The **Cache API** (`caches.default.put`) caches a `Response` *you* construct — compute-result/assembled-response caching keyed by a request you control. The **`cf` object on `fetch`** caches the *subrequest to origin* transparently — origin-response caching. **KV** is a global, eventually-consistent store, not a request-scoped HTTP cache. Picking KV for per-request HTTP response caching (paying write cost and the 1-write/s/key cap for something the Cache API does for free), or reaching for the Cache API expecting global coherence, is the misfit to flag — cross-reference `kv.md`'s store-choice bullet.

- **A hot, cacheable response recomputed every request** (lane `data-access`): the inverse of the never-hit key — an endpoint that recomputes or refetches an *identical, cacheable* result on every request with no Cache API/`cf` caching at all. The core flags this in the data-access lane; the module-level test is whether the response is genuinely identical across requests (no per-user/per-request variance in the body) and cacheable (below) — if so, a `caches.default` lookup-then-`put`, or `cf: { cacheEverything, cacheTtl }` for an origin fetch, removes the recompute. Weigh staleness tolerance before pinning a TTL.

- **`cache.put` left on the response critical path** (lane `concurrency`): `cache.put` is a write that costs latency. Awaiting it before returning (`await cache.put(req, res); return res;`) adds that write to every cache-miss response the user waits on, for no user-visible benefit. Defer it: `ctx.waitUntil(cache.put(req, res.clone()))` and return immediately, so the write runs after the response is sent (cross-reference `waitUntil` in `../cloudflare.md`). Note the `Response` must be `clone()`d — a body can only be read once, and both the client and the cache need it.

- **"Caching" a response that is silently ineligible** (lane `data-access`): `cache.put` refuses several response shapes and the rejection is easy to miss — a response with a `Set-Cookie` header is **not cached** (strip it, or set `Cache-Control: private=Set-Cookie`, before `put`), `206 Partial Content` is rejected, `Vary: *` is rejected, and a `Cache-Control` that forbids caching (or an over-size body) returns a `413` from `put` (verify against the currency brief for your version). Only `GET` is cacheable. Code that builds a response with `Set-Cookie` and then `put`s it believes it's caching while every request is a miss — verify the response actually meets the eligibility rules, not just that `put` was called. Also note `stale-while-revalidate`/`stale-if-error` are unsupported by the Cache API.

- **Edge TTL vs browser TTL, and fighting `Cache-Control`** (lane `data-access`): the edge TTL (origin `s-maxage`, the `cf` `cacheTtl`, or the zone Edge Cache TTL) governs how long Cloudflare's network serves a cached copy; `max-age` governs the *visitor's browser*. Cloudflare respects origin `Cache-Control`/`Expires` by default and overrides only in narrow cases (verify against the currency brief for your version). The audit is matching TTL to **change rate**: an aggressive edge TTL on data that changes widens the staleness window (users see old data until expiry, since a low-traffic colo won't naturally evict it), while a too-short TTL throws away hits and re-pays origin. Free-plan Edge Cache TTL has a high minimum (verify) — don't assume a 1-second edge TTL is honored everywhere. For browser-facing assets this overlaps the HTML pack's caching/CWV concepts — cross-reference rather than restating them.

- **Tiered Cache for the per-colo cold-miss** (lane `idiom-currency`): when the problem is leaf colos cold-missing to origin (the per-colo limitation above), **Tiered Cache** is the named lever — a leaf-colo miss checks an upper-tier (and, with Regional Tiered Cache, a regional hub) colo before falling through to origin, so one warm upper tier serves many leaf colos and origin load drops (verify against the currency brief for your version). Flag it specifically when multi-colo traffic plus an expensive origin is the shape; note it is incompatible with the Cache-API `cache.put` path (it applies to the `fetch`/zone cache, not `caches.default`). Cache Reserve (a persistent backing tier) is an adjacent lever — confirm availability against the brief, LOW confidence offline.

- **Cache stampede on a cold, expensive key** (lane `concurrency`): when a hot key is cold (first request, or just-expired), many concurrent requests can all miss and all recompute/refetch the same expensive resource at once — a thundering herd against your compute or origin. Cloudflare's edge cache does some request coalescing for in-flight origin fetches, which blunts the origin side, but it does **not** coordinate *compute* you run in the Worker before `cache.put`, and coalescing behavior is not a guarantee to lean on (verify against the currency brief for your version). For a genuinely expensive single resource, name the concern and weigh a coordination point — a **Durable Object** as the single-writer that computes once and fans the result out (cross-reference `../cloudflare.md`'s Durable Objects notes) — rather than assuming the cache absorbs the herd.

## Sources

Authoritative Cloudflare documentation fetched for this module:

- **Workers — Cache API** (developers.cloudflare.com/workers/runtime-apis/cache/): `caches.default` vs `caches.open(name)`; `cache.put`/`match`/`delete` semantics (`put` is `GET`-only, rejects `206`, rejects `Vary: *`, `413` when `Cache-Control` forbids or response too large, silently rejects `301/302` when the cache key lacks a query string the `Location` carries); **`Set-Cookie` responses are not cached** unless the header is removed or `Cache-Control: private=Set-Cookie` is set; **"contents of the cache do not replicate outside of the originating data center"** and `cache.delete` purges only the invoking colo; `match` never makes an origin subrequest and respects `If-Modified-Since`/`If-None-Match`/`Range`; Cache API path is incompatible with tiered caching; `stale-while-revalidate`/`stale-if-error` unsupported.
- **Workers — Request `cf` object** (developers.cloudflare.com/workers/runtime-apis/request/): `cacheEverything` ("treats all content as static," defaults `false`, `GET`/`HEAD` only, caches beyond Cloudflare's default cached content); `cacheTtl` (seconds, forces caching regardless of origin headers, `0` = immediate expiry, `GET`/`HEAD` only); `cacheTtlByStatus` (per-status object, negative value = don't cache); `cacheKey` (requests sharing a key share a cached response).
- **Cache — Cache keys** (developers.cloudflare.com/cache/how-to/cache-keys/): default key = full URL (scheme/host/URI incl. query string) + origin header + select `x-`-prefixed headers; custom keys trade control for hit-rate / cache sharding; query-string include/exclude controls whether differing params share an entry; key fragmentation reduces hits, consolidation increases them; most customization is Enterprise-gated.
- **Cache — Edge & Browser Cache TTL** (developers.cloudflare.com/cache/how-to/edge-browser-cache-ttl/): Edge Cache TTL (network-level) vs Browser Cache TTL (visitor browser); Cloudflare respects origin `Cache-Control`/`Expires` by default, overrides when origin TTL < Browser Cache TTL setting or when neither header is sent; Free Edge Cache TTL minimum 2 h vs 1 s on Business/Enterprise; purging the edge does not affect browser-stored assets.
- **Cache — Tiered Cache** (developers.cloudflare.com/cache/how-to/tiered-cache/): lower-tier (visitor-nearest) colos query an upper-tier before origin; "limits the number of data centers that can ask the origin"; "dramatically increasing cache hit ratios"; Regional Tiered Cache adds a regional hub check before the upper tier.

Unverified / flagged: the Cache API docs page did **not** state a Cache-API-specific per-request call count or max object size (the core pack cites Free 50 / Paid 1,000 calls and a 512 MB max object from the Workers limits page — verify there). The Tiered Cache page did **not** mention request coalescing, cache stampede, or Cache Reserve by name — the coalescing behavior and Cache Reserve availability in the stampede bullet are flagged LOW-confidence / verify against the currency brief. `cacheTtl`/`cacheTtlByStatus`/`cacheEverything` numeric/plan specifics and the edge-TTL minimums carry verify-tags as they may move by plan/version.
