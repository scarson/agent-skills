# Cloudflare performance module: D1
> Load when a D1 binding (`[[d1_databases]]`, `env.<DB>.prepare`) is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the D1 lens only.

## D1

> Scope: D1 is Cloudflare's managed **SQLite** at the edge, reached from a Worker through a binding (`env.DB.prepare(...)`). The performance story is almost entirely about **round-trips and rows-read**, not CPU: each prepared-statement execution is a network call from the Worker to the D1 backend, the backend has a *primary location* a far-edge Worker pays latency to reach, and D1 **bills on rows read** — so a query that scans more rows than it returns costs both latency and money. The recurring traps are N+1 (a query per loop iteration), reads routed across a region to a distant primary, and full table scans from a missing index. Because D1 *is* SQLite, defer query-plan and index mechanics (`EXPLAIN QUERY PLAN`, covering indexes, predicate sargability) to `../sql.md` and read this for the D1-specific surface: the binding API's round-trip cost, `batch()`, the Sessions API / read replication, and the platform limits. Cross-reference the **Workers execution model** in `../cloudflare.md` (CPU budget, subrequest limit, stateless-between-requests, Smart Placement). The core lanes (`algorithmic`/`data-access`/`concurrency`/`idiom-currency`) are defined in `../cloudflare.md`; this is the D1 provider lens.

- **N+1: one `.run()`/`.first()`/`.all()` per loop iteration is one network round-trip per iteration** (lane `data-access`): each prepared-statement execution is a separate call from the Worker to the D1 backend, so a query inside a `for`/`map` over N items pays N sequential round-trips — the single highest-ratio D1 latency trap, and the one to weight hardest in a dashboard/API Worker where per-request DB latency dominates. The fix is to collapse the loop into one statement (an `IN (...)` / `JOIN` that fetches all N at once) or into one `db.batch([...])`. The tell is a `prepare()`/`bind()` *inside* an iteration construct.

- **`db.batch([stmt, stmt, ...])` sends multiple statements in ONE round-trip** (lane `data-access`): this is the canonical N+1 fix — Cloudflare's own framing is that batching "reduces latency from network round trips to D1." Reach for it whenever a fixed set of statements must run per request (e.g. a write plus its dependent reads). Note `batch()` takes *prepared statements*, so it suits a known statement list, not arbitrary streamed work; a genuinely data-dependent sequence (statement K depends on result of K−1) can't be batched and stays multi-round-trip.

- **`batch()` is also an implicit transaction — use it for multi-write consistency, not just round-trip reduction** (lane `data-access` / `concurrency`): batched statements run "sequentially, non-concurrently," and the batch is a SQL transaction — if one statement fails the whole sequence rolls back. So a multi-row write that must be all-or-nothing belongs in `batch()` for *both* atomicity and the single round-trip; conversely, do not assume statements in a batch run in parallel (they don't), and a long batch still runs under the per-query duration limit. D1 has no interactive multi-round-trip `BEGIN…COMMIT` across separate `.run()` calls — `batch()` is the transaction primitive.

- **D1 has a primary location; a Worker at a distant edge pays cross-region latency on every query to it** (lane `data-access`): without read replication, all queries route to the single primary instance, so a read-heavy Worker far from the primary is latency-bound by round-trip distance, not by query cost. This is where the **Sessions API + read replication** earns its place — and where it interacts with **Smart Placement** in `../cloudflare.md` (which instead moves the *Worker* near the data). Flag a read-heavy, globally-distributed Worker hitting a distant primary with neither lever engaged.

- **Sessions API routes reads to a nearby replica with read-your-writes via bookmarks** (lane `data-access` / `idiom-currency`): `db.withSession(...)` opens a session that can serve reads from a geographically closer read replica; `"first-unconstrained"` lets the first query hit any instance (lowest latency), `"first-primary"` forces the primary first. To preserve read-your-writes across requests, carry the bookmark from `session.getBookmark()` forward (the docs' example threads it through the `x-d1-bookmark` request/response header) and reopen the next session with `withSession(bookmark)` so the replica waits until it is at least as current as that bookmark — verify the exact header/method names against the currency brief for your version. Without this, a read after a write can land on a stale replica and miss the just-written row.

- **Replication consistency model is currency-sensitive — don't assert "eventually consistent" loosely** (lane `idiom-currency`): the Sessions API provides **sequential consistency** with bookmark-gated reads, not best-effort eventual reads; replicas lag the primary but a bookmarked session reads a state at least as fresh as the bookmark. The replication/Sessions surface is actively evolving (routing hints, defaults, header names), so treat consistency guarantees and the bookmark mechanics as brief-checked rather than recited from memory (verify against the currency brief for your version).

- **Missing index ⇒ full table scan ⇒ inflated rows-read (billed) and latency** (lane `data-access`): D1 *is* SQLite, so the query-plan reasoning — confirm index use with `EXPLAIN QUERY PLAN`, prefer covering indexes, watch predicate sargability — lives in `../sql.md`; defer to it. The **D1-specific** consequence is that a scan doesn't just slow the query, it reads (and bills) every row scanned, not just the rows returned. A `WHERE`/`ORDER BY` on an unindexed column is therefore a cost finding as much as a latency one.

- **Rows-read is the billing and latency driver — `SELECT *` / no `WHERE` / no `LIMIT` over-reads** (lane `data-access` / `algorithmic`): D1 charges on rows read, and a query reads what it must scan to satisfy the plan. Select only needed columns (a narrower covering index can satisfy the read without touching the row), filter in SQL rather than in the Worker, and paginate instead of pulling a whole table to slice it in JS. The anti-pattern is fetching broadly and reducing in the Worker — it bills the full scan *and* ships the rows over the wire.

- **String-interpolated values instead of `.bind()` parameters** (lane `data-access`): build the statement once with `?`/`?NNN` placeholders and supply values via `.bind()` so the prepared statement (and its plan) is reused across executions; interpolating values into the SQL string defeats statement reuse and is an injection vector. The tell is template-literal SQL with `${value}` in the predicate rather than a placeholder.

- **Large result sets cost Worker memory + transfer, and brush against hard response limits** (lane `payload-startup` / `data-access`): returning a big result materializes it in the Worker's memory and ships it across the binding. D1 caps individual query response size (and string/BLOB/row size) at 2 MB, bounds bound parameters per query to 100, statement length to ~100 KB, and query duration to 30 s, with database size capped (10 GB on Workers Paid) — so an unpaginated "return everything" query can fail outright, not merely run slow. Paginate with `LIMIT`/keyset cursors and keep payloads bounded (verify these limits against the currency brief for your version).

- **`run()`/`all()` metadata is the cheap measurement hook — read it before guessing** (lane `data-access`): a `D1Result` carries `meta` with `rows_read`, `rows_written`, and `duration`, so the rows-read cost and latency of any statement are observable directly rather than inferred. When auditing, prefer the reported `rows_read` vs rows-returned ratio as the scan-amplification signal over reasoning about the plan in the abstract (verify field names against the currency brief for your version).
