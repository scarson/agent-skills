# Cloudflare performance module: Durable Objects
> Load when a Durable Objects binding (`[[durable_objects]]`, `DurableObjectNamespace`, a class `extends DurableObject`) is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the Durable Objects lens only.

## Durable Objects

> Scope: a Workers app that reaches a Durable Object (DO) for stateful coordination — a chat/game room, a collaborative-document session, a rate limiter, a per-user actor, a sequential workflow. A DO is one thing: a **single-threaded, strongly-consistent, single-instance** coordination point with attached durable storage, addressed by name so every request for a given id lands on the *same* instance. Almost every finding here flows from that shape. Two questions catch most of them: **"how much traffic funnels through one DO id?"** (because one instance processes requests serially — a hot DO is a throughput ceiling, not a scaling unit) and **"is the work this DO does actually coordination, or did it pick the wrong store?"** (a DO used where KV's read-cache or D1's relational queries fit pays the single-threaded tax for nothing — cross-reference `kv.md` / `d1.md`). The recurring traps beyond store-choice are storage round-trips that ignore in-memory state, and idle WebSocket DOs that bill GB-s while doing nothing. The **Workers execution model** (request lifetime, `waitUntil`, subrequest limits, Smart Placement) and the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) are defined in `../cloudflare.md` — defer to it and read this for the DO-specific surface.

- **A single DO id processes requests serially — a hot DO is a throughput bottleneck, not a scaling unit** (lane `concurrency`): one instance is single-threaded (the runtime describes it as cooperatively multi-tasked, like browser JS), so it handles messages one at a time and has a soft ceiling around **1,000 requests/sec per object** (verify against the currency brief for your version). The DO is a *coordination point*; you scale by **sharding across many ids** — by user, room, shard key — so load spreads across instances (an unlimited number per namespace). The classic anti-pattern is routing *all* traffic through one id — a single global counter, a single lock, one "stats" object — which serializes the entire app behind one core. Flag any code that derives a constant or near-constant DO id on a hot path; the fix is a key that fans out, not a faster handler.

- **`blockConcurrencyWhile()` serializes ALL events to the object until its promise resolves** (lane `concurrency`): it "executes an async callback while blocking any other events from being delivered to the Durable Object until the callback completes" — correct and idiomatic for one-time init (load state from storage before serving any request), typically in the constructor. The footgun is putting slow or unbounded work inside it: an external `fetch`, a long retry loop, anything that can hang, stalls *every* request to that DO for its duration. It carries a timeout (a too-long block resets the object — treat the exact seconds as currency-sensitive, verify against the currency brief for your version), so a wedged block fails the object rather than blocking forever, but the latency hit until then is real. Use it for fast, bounded initialization, not as a general critical section.

- **Input/output gates already give consistency — don't hand-roll locking** (lane `concurrency`): the runtime gates I/O automatically. The **input gate** holds incoming events while a storage operation is outstanding ("while a storage operation is executing, no events shall be delivered to the object except for storage completion events"), so requests don't interleave around an `await`; the **output gate** holds outgoing network messages until in-progress writes are confirmed durable, so no external party observes an unpersisted result. Together they let you write straight-line, single-threaded-style code (`get` then `put` a counter) that is correct under concurrency with **no explicit mutex/transaction**. A manual locking layer (a `locked` flag, a promise queue) bolted on top fights the gates and adds latency for a guarantee you already have. Cross-reference the **Workers execution model** in `../cloudflare.md`. (Note: gating semantics are subtle — confirm behavior for your storage backend against the currency brief rather than reciting it.)

- **Re-reading storage every request when in-memory state would serve it** (lane `data-access`): a DO instance stays in memory between requests (alive while serving, and for a short idle window before hibernating), so its instance fields *are* a cache. `state.storage.get` / `put` are durable round-trips; a handler that reloads the same value from storage on every request — when it could read it once at init and keep the live copy in a field, persisting only on change — pays a round-trip it doesn't need. Conversely, code that mutates only an in-memory field and never persists loses state on eviction. The idiom is: hydrate from storage at startup, read/write the in-memory copy on the hot path, `put` to persist durably. Flag a `storage.get` on a path whose value is already (or could be) held in memory.

- **Storage ops that could be batched issued one at a time** (lane `data-access`): `get` / `put` / `delete` accept **multiple keys in one call** (up to **128**, verify against the currency brief for your version) — one round-trip instead of N. Separately, multiple `put`/`delete` calls issued **without an intervening `await`** are coalesced and committed atomically by the runtime, so a burst of writes in one synchronous stretch is already one durable operation. The anti-pattern is a loop awaiting one `storage.put` per item (the DO-storage form of N+1). Pass the batch, or let write-coalescing do its job by not awaiting between puts.

- **New DO classes should use the SQLite-backed storage backend, not legacy KV-style storage** (lane `data-access` / `idiom-currency`): Cloudflare recommends **all new namespaces use the SQLite storage backend**, which adds the SQL API (`state.storage.sql.exec(...)`), supports more data types and tables, and offers point-in-time recovery — alongside the same key-value API. SQLite-backed KV operations are also synchronous (they don't yield the event loop, so they execute atomically), a different perf profile from the async KV backend. Backend choice is set at class creation via migration and isn't trivially switched later, so flag a *new* class still on the legacy KV backend — but verify availability, the exact migration mechanics, and which features are GA against the currency brief for your version.

- **WebSocket DOs without the Hibernation API bill GB-s while idle** (lane `payload-startup` / `concurrency`): a DO holding open WebSockets normally stays resident in memory, accruing **duration (GB-s) billing the whole time even when no messages flow** — expensive for chat/realtime rooms that sit mostly idle. The **WebSocket Hibernation API** lets the runtime evict the DO from memory while keeping the sockets connected to Cloudflare's edge; "Billable Duration (GB-s) charges do not accrue during hibernation," and the object is re-initialized (constructor reruns, in-memory state resets) when a message arrives. The cost is that per-connection in-memory state is lost across hibernation, so it must be reattached via `serializeAttachment` / `deserializeAttachment` (verify names against the currency brief for your version). Flag a long-lived-WebSocket DO using the *standard* `addEventListener` API where hibernation fits; the deliberate exception is a connection that genuinely needs continuous in-memory work between messages.

- **DO location is fixed near first access; a distant, chatty Worker→DO path compounds latency** (lane `data-access`): a DO is created in one location (close to where it's first requested, subject to jurisdiction) and stays there; clients and Workers far from it pay round-trip latency on every call. A Worker at a distant edge that makes *several* sequential DO calls per request multiplies that distance. Levers: fewer, coarser calls (batch the interaction, prefer one RPC over a chatty sequence), and **Smart Placement** to move the *Worker* near the DO (cross-reference `../cloudflare.md`) — analogous to the D1-primary-distance problem in `d1.md`. Flag a hot path that round-trips to a far DO multiple times where one call would do.

- **Worker→DO calls: prefer RPC over the `fetch` shim** (lane `idiom-currency`): with a recent compatibility date, public methods on the DO class are callable as RPC on the stub (`stub.method(args)`) — typed, no fake-URL plumbing — versus the legacy `stub.fetch(request)` pattern that forces you to encode everything into an HTTP request and route it through a `fetch` handler. RPC is the current idiom; verify the enabling compatibility date and serialization constraints against the currency brief for your version. This is ergonomics-and-overhead, not a throughput cliff — weight it below the sharding and store-choice findings.

- **Use Alarms for deferred/periodic work instead of keeping the DO alive or polling** (lane `concurrency` / `idiom-currency`): `setAlarm(scheduledTimeMs)` / `getAlarm()` / `deleteAlarm()` schedule a future `alarm()` invocation that wakes the object even after it has gone idle or hibernated, with at-least-once retry. This is the right primitive for batched flushes, TTL cleanup, scheduled state transitions, or draining a queue — none of which should keep an instance resident (billing GB-s) or be driven by an external poller hammering the DO. A DO that stays alive purely to "check for work," or an outside cron that polls it, is the pattern alarms replace. Flag keep-alive loops and external pollers where a self-rescheduling alarm fits.

- **Store-choice: a DO used where KV or D1 fits adds the single-threaded tax for nothing** (lane `idiom-currency`): a DO is the right tool when you need **strongly-consistent, single-writer, coordinated** state — counters, rate limiters, locks, game/chat rooms, collaborative sessions, sequential workflows — exactly the workloads KV's eventual consistency and per-key write ceiling can't serve (cross-reference `kv.md`). But reach for it for *read-mostly* config or cacheable lookups and you've put a serial bottleneck in front of data KV would serve read-cached and cheaper; reach for it for relational queries over a dataset and you're hand-rolling what D1 does (cross-reference `d1.md`). The justification is the *need for coordination*, not "it can hold state." Flag a DO whose access pattern shows no real contention or consistency requirement — it likely wants a different store.
