# Cloudflare performance module: Hyperdrive
> Load when a Hyperdrive binding (`[[hyperdrive]]`, `env.<HD>.connectionString`) or a Worker connecting to Postgres/MySQL is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the Hyperdrive lens only.

## Hyperdrive

> Scope: Hyperdrive is Cloudflare's accelerator for a **traditional regional database** (Postgres/MySQL — Supabase/Neon/RDS/PlanetScale/CockroachDB) reached from a Worker. It exists because of two Workers realities from `../cloudflare.md`: a Worker is **stateless between requests** (it can't hold a connection pool) and runs at a **distant edge** from a regional DB. A raw driver therefore pays a multi-round-trip TLS+auth handshake from far away on *every* request and threatens to exhaust the DB's connection limit under fan-out. Hyperdrive solves exactly two things and no more: (1) it keeps a **global connection pool** near the origin so the Worker's per-request connection setup happens edge-local and is cheap, and (2) it **caches read-only query results** at the edge (default-on). It does **not** rewrite your SQL: N+1, missing indexes, and over-fetch survive Hyperdrive untouched — defer the query-level mechanics to `../sql.md` (and see the D1 N+1 framing in `d1.md`, which is the same shape over a different transport). The lanes (`data-access`/`concurrency`/`idiom-currency`/`algorithmic`) are defined in `../cloudflare.md`; this is the Hyperdrive provider lens.

- **The headline finding: a Worker connecting to a regional Postgres/MySQL *without* Hyperdrive pays connection setup per request** (lane `data-access` / `concurrency`): because a Worker can't pool connections across requests (the statelessness/no-pooling note in `../cloudflare.md`), a raw `pg`/`postgres`/`mysql2` client opens a fresh connection on every invocation — a multi-round-trip TLS+auth handshake from a distant edge to the DB's region — and a fan-out of requests piles new connections onto the origin's finite connection limit. Hyperdrive moves the pool *next to the origin* so the Worker's handshake is edge-local and a ready connection is reused. A direct-connect Worker with no Hyperdrive (and no other pooler) is the single highest-weight finding in this lens; weight it first.

- **Query caching is default-ON for read-only queries — a read-heavy workload gets edge-cached results for free** (lane `data-access`): Hyperdrive parses each query, classifies it as mutating vs non-mutating, and caches the **read-only** results at the edge (default `max_age` ~60 s, `stale_while_revalidate` ~15 s, `max_age` capped at ~1 hour — verify against the currency brief for your version). So a repeat-read API never reaches the origin. The corollary to set expectations on: **writes, transactions, and non-deterministic reads bypass the cache entirely** — a write-heavy or all-transactional workload gets the *pooling* win but not the *caching* win, so don't promise caching latency gains for a workload that's mostly mutations.

- **What is non-deterministic enough to skip the cache is narrower than people assume** (lane `data-access` / `idiom-currency`): only queries built from `IMMUTABLE` functions are cacheable; a query using volatile functions (`random()`, `lastval()`) **or even stable ones** (`now()`, `current_date`, `current_timestamp`) is treated as non-cacheable and always hits origin (verify against the currency brief for your version). A `SELECT ... WHERE created > now() - interval '1 day'` therefore silently never caches; Cloudflare's own guidance is to compute the timestamp in the Worker and bind it as a literal so the query becomes cacheable. The tell is a hot read that "should be cached" but embeds `now()`/`current_date` in the SQL.

- **Tuning `max_age` / `stale_while_revalidate` is the same staleness trade as any cache** (lane `data-access` / `idiom-currency`): a longer `max_age` cuts origin reads but serves staler rows; `stale_while_revalidate` serves a stale result immediately while refreshing in the background, so it trades freshness for tail latency on the revalidation window. Match the TTL to the data's actual change rate — there's no Hyperdrive-specific magic here, just the cache-coherence judgment, bounded by the ~1 hour `max_age` ceiling (verify against the currency brief for your version).

- **Disabling caching where correctness requires it — the inverse of the read-heavy win** (lane `idiom-currency`): for read-your-writes-sensitive or highly-dynamic data, a cached read can serve a row that's already been superseded. Disable per-instance (`wrangler hyperdrive update <id> --caching-disabled true`, or the config equivalent) when the workload can't tolerate `max_age`-window staleness (verify the exact flag/config key against the currency brief for your version). Flag the opposite mistake too: caching left on for a dashboard that must reflect a just-committed write.

- **Hyperdrive does NOT fix N+1, missing indexes, or over-fetch — each uncached query is still a round-trip** (lane `algorithmic` / `data-access`): Hyperdrive accelerates *connections* and caches *reads*; it does nothing for query *shape*. A query-per-row loop is still N sequential round-trips (now over a pooled connection instead of N handshakes — better, but still N trips), a missing index still drives a full scan at the origin, and `SELECT *` still ships unused columns. The SQL-level work — batch into one `IN (...)`/`JOIN`, index the predicate, narrow the projection — is unchanged; defer it to `../sql.md` and read the N+1 framing in `d1.md` as the same defect over a different transport.

- **Create the driver client per request — that's correct *with* Hyperdrive — but don't open more than one per request** (lane `concurrency`): because the pool is Hyperdrive's, the per-request `new Client(...)` is cheap, so the old serverless "cache a global connection in a module variable" hack is both unnecessary *and* unreliable on Workers (the core's no-warm-connection note). The remaining waste is *within* a request: opening multiple clients when one suffices, or not awaiting/closing the connection (use `ctx.waitUntil(client.end())` so cleanup doesn't block the response), which holds pool slots longer than needed.

- **Smart Placement complements Hyperdrive for chatty *uncacheable* workloads** (lane `data-access`): Hyperdrive removes the per-request connection-setup cost, but many *sequential, uncacheable* queries from a distant edge still pay the per-query round-trip latency on each hop. That's the regime where **Smart Placement** (move the Worker near the DB, per `../cloudflare.md`) earns its place — the two stack: Hyperdrive cheapens the connection, Smart Placement shortens the per-query distance. Reach for it when the workload is write/transaction-heavy (cache can't help) *and* makes many dependent queries per request.

- **Using Hyperdrive pulls a real DB driver (and often an ORM) into the bundle — a startup cost** (lane `payload-startup` / `idiom-currency`): Hyperdrive needs `nodejs_compat` plus a supported driver (`pg`, `postgres`, or `mysql2`), and the driver — more so a full ORM layered on top — adds parse/startup weight to every isolate (the core bundle-size lane). It's a justified cost when you're talking to a relational origin, but weigh the driver/ORM heft against a leaner client, and confirm `nodejs_compat` is pulling its weight (verify the supported drivers/flags against the currency brief for your version).

- **Pooling raises the ceiling but does not remove it — high fan-out can still queue on pool capacity** (lane `concurrency`): Hyperdrive deliberately holds the origin connection count down (it aims to use the fewest connections necessary), so a burst of concurrent requests larger than the pool/origin connection budget queues for a slot rather than overwhelming the DB. That's the intended protection, but it means a very wide concurrent fan-out is bounded by the origin's connection limit, not by Worker concurrency — size throughput expectations to the *origin's* capacity, not to "Hyperdrive made it infinite."
