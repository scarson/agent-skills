# Cloudflare performance module: Workers KV
> Load when a KV binding (`[[kv_namespaces]]`, `env.<KV>.get`/`put`) is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the KV lens only.

## Workers KV

> Scope: a Workers app reaching `env.<KV>` for config, feature flags, sessions, or a cache layer. KV is one thing — an **eventually-consistent, read-optimized, edge-cached** key-value store — and almost every performance *and* correctness finding here flows from that single shape. The recurring trap is using KV where the workload's consistency or write profile actually demands a different store: read-after-write logic on KV reads stale, and a high write rate per key collides with a hard ceiling. The two questions that catch most real findings are "does this read tolerate up-to-~60s-stale data?" and "how often does any *single* key get written?". The Workers execution model (request-scoped lifetime, `waitUntil`, no shared in-process state across isolates) lives in `../cloudflare.md` — cross-reference it; the **KV-vs-D1-vs-Durable-Objects** store choice recurs throughout and is handled by deferring to the `d1.md` and `durable-objects.md` siblings rather than restating their models here.

- **Read-after-write / strongly-consistent data on KV** (lane `data-access`): KV is eventually consistent — a write is *usually* immediately visible at the location that issued it but, by Cloudflare's own docs, **not guaranteed even there**, and takes **up to ~60 seconds or more** to propagate to other edge locations as cached versions expire (verify against the currency brief for your version). A request that writes a value and a *different* request (likely a different edge) that reads it back can see the old value — or, because negative lookups are cached too, a "does not exist" for a key you just created. Any logic that depends on reading your own recent write (a just-rotated session, a "did this just succeed" check, a uniqueness guard) is a latent correctness bug on KV, not a perf nit — that workload needs Durable Objects (single-writer, strongly consistent) or D1. Cross-reference `durable-objects.md` / `d1.md` rather than working around the staleness.

- **High write rate per key** (lane `data-access` / `algorithmic`): KV enforces a hard limit of **1 write per second to the same key** on both Free and Paid plans (verify against the currency brief for your version) — this is per *unique key*, not per namespace. Counters, rate-limiter state, per-request session bumps, a job-status row hammered by progress updates, or any "write on every request" pattern hits this ceiling and silently throttles/fails. The fix is rarely "shard the key"; it is the wrong store. Single-key contention with frequent writes is exactly the Durable Objects single-writer use case — cross-reference `durable-objects.md`. Flag the *write frequency of the hottest key*, not the aggregate write count.

- **Treating KV writes as cheap as reads** (lane `data-access`): the billing and performance shape rewards read-heavy/cache-like use and punishes writes — reads are cheap and edge-cached; writes, deletes, and lists are central, costlier operations. This is the design signal, not an incidental cost: a feature that writes KV on a hot path (logging, analytics increments, "last seen" timestamps) is fighting the store's grain even when it stays under the 1-write/s/key cap. Name the read:write ratio of the access pattern; KV is built for high reads relative to writes (verify against the currency brief for your version).

- **Cold-read latency on a value that could carry a `cacheTtl`** (lane `data-access`): the first read of a key from a given edge is a central round-trip (nearest regional tier → central tier → central store for a truly cold global read); subsequent reads from that same location are served from the local edge cache (fastest, cheapest). `cacheTtl` (seconds, **default 60**, **minimum 30**) tunes how long the edge caches the value (verify against the currency brief for your version). For write-once / write-rarely data — config, feature flags, static lookups — a longer `cacheTtl` cuts cold-read frequency at the cost of staleness; for data that changes, a long TTL widens the eventual-consistency window. The judgment is per-key tolerance for staleness, not a global knob — a hot, rarely-changing config key wants a long TTL; a value read once and never reused gains nothing from caching it.

- **`type` parameter left at the `text` default when the value is parsed every read** (lane `data-access` / `idiom-currency`): `get()` defaults to `type: "text"` (returns a string); options are `text`, `json`, `arrayBuffer`, `stream` (verify against the currency brief for your version). Code that fetches as text and then `JSON.parse`s on every read is doing parse work the runtime can do via `type: "json"`; conversely a large binary value pulled as `text`/`json` instead of `stream`/`arrayBuffer` forces a full in-Worker materialization. Match the read type to how the value is consumed — and note the multi-key form of `get` supports only `text`/`json`.

- **N separate `get`s for data always read together** (lane `data-access`): each `get` of a cold key is its own central round-trip; fetching N keys that are always consumed as a unit is N round-trips where one would do. Bundle related fields under a single key as one JSON blob (within the value-size limit, below), or use the multi-key `get`/`getWithMetadata` form. This is the KV form of N+1 — cross-reference the generic `data-access` lane. The trade-off: bundling couples the write granularity (the whole blob rewrites together, against the 1-write/s/key cap), so co-locate only fields that are read *and* written together.

- **A second `get` to fetch metadata that `getWithMetadata` would return in one** (lane `data-access`): KV values carry up to **1024 bytes** of metadata (verify against the currency brief for your version) returned alongside the value by `getWithMetadata()` in a single operation. Code that reads a value and then issues a *second* `get` to a sidecar "<key>:meta" entry for small auxiliary data is paying two round-trips and two operations where metadata yields one. Cloudflare states storing values in metadata is more efficient than a `list()`-then-`get()`-per-key — fold small per-key descriptors into metadata.

- **A hot path built around `list()`** (lane `data-access`): `list()` is a **central, non-edge-cached** operation — slower and costlier than `get`, returning at most **1000 keys per call** (the default and the maximum) with cursor pagination via `list_complete`/`cursor` (verify against the currency brief for your version). Scanning or filtering keys with `list()` on every request — "find all keys with prefix X then get each" — is an anti-pattern: it round-trips centrally and then fans out into per-key `get`s. Model the access so the path computes the *known* key(s) to `get` directly; reserve `list()` for admin/enumeration, not request serving. (Also: `list()` reflects the same eventual-consistency window and may return tombstoned-but-empty pages — check `list_complete`, not `keys.length`.)

- **Value size and large-value transfer cost** (lane `payload-startup` / `data-access`): limits are **value 25 MiB**, **key 512 bytes**, **metadata 1024 bytes** (verify against the currency brief for your version). Beyond the hard cap, a large value costs network transfer on every cold read plus memory inside the Worker isolate when materialized — a multi-megabyte blob read on a hot path is a payload problem even though KV "allows" it. Over-bundling (the inverse of the N+1 fix) inflates every read of a key when callers only need a slice; weigh blob size against how much of it each read actually consumes.

- **The store-choice decision: is KV even the right store?** (lane `idiom-currency`): KV (eventual, read-cached, global, read-optimized) is one of four Cloudflare storage primitives, and the most common KV finding is a workload mis-housed in it. Use **D1** for relational/queryable structured data with a high read:write ratio (cross-reference `d1.md`); **Durable Objects** for strongly-consistent, transactional, single-writer coordination — real-time state, counters, uniqueness, read-after-write (cross-reference `durable-objects.md`); the **Cache API** for request-scoped HTTP response caching tied to a URL, which KV is *not* a drop-in for. Flag any KV use whose consistency need (read-after-write), write rate (high per-key writes), or query shape (filter/scan via `list()`) is a tell that one of the siblings is the correct store — that re-housing is the highest-leverage finding, above any in-KV tuning.
