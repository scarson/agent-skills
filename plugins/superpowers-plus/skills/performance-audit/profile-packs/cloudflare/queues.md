# Cloudflare performance module: Queues
> Load when a Queues binding (`[[queues.producers]]`/`[[queues.consumers]]`, `env.<Q>.send`/`sendBatch`, a `queue(batch, env)` handler) is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the Queues lens only.

## Queues

> Scope: Cloudflare Queues is the durable, at-least-once message bus for async pipelines that run *off* the request path — email/webhook dispatch, ETL, fan-out, retriable background jobs. A producer Worker writes with `env.<Q>.send`/`sendBatch`; a push consumer Worker is invoked with a `queue(batch, env, ctx)` handler that receives a **batch** of messages and runs under a longer duration budget than a `fetch` handler. The entire performance story is **batching and throughput**: the platform hands the consumer a batch precisely so one downstream operation can cover many messages, and the dominant defect is code that throws that away by doing one I/O round-trip per message. The secondary axis is retry economics — at-least-once delivery plus whole-batch retry means a single bad message can re-run expensive work over an entire batch, and a poison message can burn invocations forever without a dead-letter queue. The Workers execution model (CPU budget, subrequest cap, `waitUntil` window, statelessness) lives in `../cloudflare.md` — cross-reference it. The per-store batching APIs the consumer writes *into* (D1 `batch()`, R2) live in `d1.md` / `r2.md`; defer to them rather than restating. Concrete defaults/limits are tagged "(verify against the currency brief for your version)" because they move.

- **Process the batch as a BATCH, not one I/O round-trip per message** (lane `concurrency`): the headline throughput finding. The consumer handler receives `batch.messages` *as a unit* — a `for (const msg of batch.messages)` loop that does one downstream operation per message (a D1 write, an HTTP POST, an R2 put) pays N sequential round-trips and throws away the batching the platform already did for you. Collapse the per-message work into **one** batched downstream operation over the whole batch: one `db.batch([...])` / one `IN (...)` write (cross-reference the D1 `batch()` point in `d1.md`), one multi-record API call. The tell is a binding/`fetch` call *inside* the `for...of batch.messages` loop. This also interacts with the subrequest cap (`../cloudflare.md`): per-message fan-out multiplies the subrequest count by batch size.

- **Tune `max_batch_size` / `max_batch_timeout` to the downstream's efficient batch size** (lane `concurrency` / `data-access`): a batch is delivered when *either* limit is hit first. Too-small a batch (or a tiny timeout) means many small invocations, each paying per-invocation overhead and a thin downstream write; too-large a batch delays processing of early messages by up to the timeout. Size `max_batch_size` to the downstream's natural batch (e.g. a DB bulk-insert width, an API's max items-per-call), and set `max_batch_timeout` by how long the *earliest* message can wait. Defaults: `max_batch_size` 10 (min 1, max 100), `max_batch_timeout` 5 s (min 0, max 60 s) (verify against the currency brief for your version). Name the latency-vs-throughput trade explicitly: larger batches/longer timeout raise throughput and cut per-message cost but add queue-dwell latency.

- **`max_concurrency` — match consumer autoscaling to what the downstream can absorb** (lane `concurrency`): a push consumer autoscales the number of *concurrent* invocations based on backlog growth and the failure ratio (verify against the currency brief for your version). Left unset (Cloudflare's recommended default — let it scale), it scales toward draining the backlog; but a rate-limited downstream (an external API with a QPS cap, a DB that throttles) can be *overwhelmed* by unbounded concurrency, turning into downstream errors that then drive *more* retries and *more* scaling. Cap `max_concurrency` (1–250, verify) to the downstream's safe concurrency when the bottleneck is downstream, not the queue. The judgment: too high overwhelms the dependency; too low lets the backlog grow — and a backlog that genuinely *can't* be drained faster is a signal the work belongs on a **pull consumer** (below), which lets the consumer set its own rate.

- **Producer `sendBatch([...])` instead of N `send()` calls in a loop** (lane `data-access`): a producer that loops and calls `env.<Q>.send(msg)` per item pays one round-trip *and one subrequest* per message; `env.<Q>.sendBatch([...])` enqueues many in a single call — one round-trip, counted against the Workers subrequest budget once rather than N times (cross-reference the subrequest cap in `../cloudflare.md`). The tell is `send()` inside a `for`/`map`. `sendBatch` accepts up to 100 messages, ≤128 KB per message, ≤256 KB total per call (verify against the currency brief for your version) — so a very large fan-out chunks into batched calls, still far fewer round-trips than per-message.

- **Explicit per-message `msg.ack()`/`msg.retry()` vs whole-batch outcome** (lane `algorithmic`): by default a batch is **all-or-nothing** for retries — if any single message fails (the handler throws, or the batch isn't fully acked), *the entire batch is retried*, re-processing every already-succeeded message in it. On a batch of 100 where message 100 fails, that re-runs the expensive work for the 99 that already succeeded — wasted compute *and* a non-idempotency hazard (duplicate writes). Call `msg.ack()` per success and `msg.retry()` per failure to scope retries to the single failing message (`ackAll()`/`retryAll()` are the batch-level forms; per-message calls take precedence) (verify against the currency brief for your version). Name the re-processing cost: the failure of one message should not re-charge the whole batch.

- **Dead-letter queue for poison messages** (lane `algorithmic`): a message that *always* fails — malformed payload, a permanently-missing dependency — is retried up to `max_retries` (default 3, max 100; verify) and then, **without a DLQ, deleted permanently**; with a `dead_letter_queue` configured it is routed there instead. The throughput rationale: until a poison message is acked, retried, or exhausts retries, it keeps re-entering delivery and burning consumer invocations (and dragging its batch into repeated retries). A DLQ drains poison messages out of the hot loop so they stop re-running, and gives you a place to inspect/replay them. Flag a consumer with retries but no DLQ where message failure is plausible — the failure mode is silent data loss *or* an invocation-burning retry loop, depending on `max_retries`.

- **Idempotency under at-least-once delivery + retry storms** (lane `algorithmic`): Queues is **at-least-once** — a message is not deleted until successfully consumed, and can be delivered *more than once* (redelivery on retry, plus the whole-batch-retry amplification above). Non-idempotent processing therefore does *duplicate* work on redelivery — re-sending an email, double-charging, re-inserting a row — which is both a correctness bug and, the perf face, **repeated expensive work** (re-running an LLM call, re-uploading to R2, re-hitting a paid API) every time a message is redelivered. Audit the handler for an idempotency key / dedup guard / upsert before concluding redelivery is cheap; a costly non-idempotent step inside a frequently-retried batch is the expensive case.

- **`waitUntil` vs Queues for deferred work** (lane `idiom-currency`): the core (`../cloudflare.md`) notes `ctx.waitUntil` keeps post-response work alive only within a ~30 s window (verify) and then cancels it, with no retry. Genuinely durable, retriable, or long-running background work (a multi-step ETL, a fan-out that may exceed 30 s) belongs on a Queue consumer, which runs under its own much longer wall-clock budget (verify: ~15 min per invocation) and gets the retry/DLQ machinery for free. Flag a long deferred chain parked on `waitUntil` that risks truncation — that's a Queue, not a fire-and-forget promise.

- **Message size / batch-size limits — pass a reference, not a blob** (lane `data-access`): a message body is capped (verify: ~128 KB per message, ~256 KB per `sendBatch` call), and the body must be structured-cloneable. Stuffing a large payload — an image, a full document, a big JSON export — into the message body brushes the cap, inflates every enqueue/dequeue, and can exceed it outright. Write the blob to **R2** and pass its key in the message instead (cross-reference `r2.md`); the consumer fetches by reference. The message should carry the *pointer and the work to do*, not the bytes. Choose the body `contentType` (`json` default post-`2024-03-18`, else `text`/`bytes`/`v8`; verify) to match how the consumer reads it rather than double-serializing.

- **Pull consumers for upstream-rate-limited or off-Workers processing** (lane `concurrency` / `idiom-currency`): a *push* consumer is the default and autoscales; a *pull* consumer (HTTP pull API, lease-based ack with a `visibility_timeout`, multiple concurrent clients each getting a distinct batch — verify) is the right shape when consumption rate is bounded by **upstream infrastructure** (a legacy system, a long-running task, an external API with strict limits) or the worker lives *outside* Cloudflare. The perf rationale: when the bottleneck is the consumer's own rate, push autoscaling can over-pull and overwhelm it (or fight a `max_concurrency` cap); pull lets the consumer set its own pace and acknowledge on its own clock. Flag a hard-rate-limited downstream forced through an autoscaling push consumer where pull would model the constraint natively.

- **Batching, Retries and Delays** — developers.cloudflare.com/queues/configuration/batching-retries/ (`max_batch_size` default 10 / min 1 / max 100; `max_batch_timeout` default 5 s / min 0 / max 60 s, whichever limit hit first delivers; `max_retries` default 3; explicit `msg.ack()`/`msg.retry()`, `ackAll()`/`retryAll()`, per-message precedence over batch-level; **without explicit acks "the entire batch is retried"** re-delivering succeeded messages; `delaySeconds` up to 24 h; the latency-vs-batch-size-vs-cost trade-off).
- **Consumer concurrency** — developers.cloudflare.com/queues/configuration/consumer-concurrency/ (push consumers autoscale on backlog size/growth and failed-vs-successful ratio; `max_concurrency` configurable 1–250; recommendation to leave unset for max scalability; explicit guidance to set lower concurrency / let backlog grow when limited by an upstream API).
- **JavaScript APIs** — developers.cloudflare.com/queues/configuration/javascript-apis/ (`send(body, options?)` body must be structured-cloneable and <128 KB; `sendBatch(messages, options?)` up to 100 messages, 128 KB per item, 256 KB total; `QueuesContentType` `json`(default after compat date `2024-03-18`)/`text`/`bytes`/`v8`; consumer handler `async queue(batch, env, ctx)`; `MessageBatch.ackAll()`/`retryAll()`; `Message.ack()`/`retry()`, `id`/`timestamp`/`body`/`attempts`).
- **Dead Letter Queues** — developers.cloudflare.com/queues/configuration/dead-letter-queues/ (DLQ receives messages after `max_retries` is reached; **without a DLQ, messages that hit the retry limit are deleted permanently**; configured via `dead_letter_queue` in wrangler or `wrangler queues consumer add ... --dead-letter-queue`; a DLQ is a normal queue needing its own consumer; undelivered DLQ messages persist 4 days).
- **How Queues works / delivery guarantees** — developers.cloudflare.com/queues/reference/how-queues-works/ (at-least-once: "messages are not deleted from a queue until the consumer has successfully consumed the message"; **no ordering guarantee**; one active consumer per queue minimizes but does not eliminate duplicates; default all-or-nothing batch retry — "If the last message in a batch fails to be processed, the entire batch will be retried"; push vs pull consumer types).
- **Pull-based consumers** — developers.cloudflare.com/queues/reference/pull-consumers/ (HTTP pull API, lease-ID ack/retry, `batch_size` default 5 / max 100, `visibility_timeout` default 30 s / max 12 h, short polling, multiple concurrent clients each get a unique batch; recommended when consumption is limited by upstream infrastructure or runs outside Workers).
- **Limits** — developers.cloudflare.com/queues/platform/limits/ (message size 128 KB incl. ~100 B metadata; batch 100 messages / 256 KB; `max_batch_timeout` max 60 s; message retries max 100; concurrent consumer invocations max 250 push-only; throughput 5,000 msg/s per queue; backlog 25 GB per queue; consumer wall-clock duration **15 min**; consumer CPU default 30 s, configurable to 5 min via `limits.cpu_ms`; `delaySeconds` max 24 h; `visibility_timeout` max 12 h).

**Unverified / flagged:** the at-least-once + whole-batch-retry *amplification framing* (one failed message re-charging an entire batch's expensive work) and the redelivery-as-repeated-expensive-work argument are reasoning *on top of* the documented at-least-once + all-or-nothing-retry behavior, not verbatim doc claims — the underlying mechanics are sourced above, the cost framing is this module's. The "~15 min consumer duration gives Queues a longer budget than `waitUntil`'s ~30 s" comparison is grounded (limits page = 15 min wall-clock; `waitUntil` ~30 s in `../cloudflare.md`) but both numbers move — verify against the currency brief for your version.
