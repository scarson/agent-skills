# Cloudflare performance module: R2
> Load when an R2 binding (`[[r2_buckets]]`, `env.<BUCKET>.get/put/list`) is detected — see the module map in `../cloudflare.md`. Core lanes + the Workers execution model live in `../cloudflare.md`; this file is the R2 lens only.

## R2

> Scope: R2 is Cloudflare's S3-compatible object store, reached from a Worker through a binding (`env.BUCKET.get/put/list/head/delete`). Its perf-and-cost shape differs from every other store in one decisive way: **zero egress fees**, so the cost driver is **operations, not bytes transferred out** — and operations split into pricier **Class A** (writes/lists/mutations) and cheap **Class B** (reads). The recurring app traps are treating an object as a buffer (pulling the whole thing to read a slice, or `await .arrayBuffer()`-ing a large body into the 128 MB isolate), `list`-ing the bucket on a hot path instead of `get`-ting a known key, and re-reading the same immutable object from R2 every request when the edge cache (or a presigned URL straight to the client) would keep the bytes off the Worker entirely. The Workers execution model — CPU budget, 128 MB isolate memory, subrequest cap, `waitUntil`, the Cache API's per-colo scope — lives in `../cloudflare.md`; cross-reference it. The store-choice question (R2 vs KV vs D1 for a given object-size/access pattern) defers to the `kv.md` / `d1.md` siblings.

- **Pulling a whole object to read a slice** (lane `data-access`): `get(key, { range })` (an `{ offset, length }` or `{ suffix }`, or an HTTP `Range` header) returns only the requested bytes (verify against the currency brief for your version). Reading a large object in full to extract a header, a row range, a video chunk, or a thumbnail wastes transfer time *and* materializes bytes the response never uses against the 128 MB isolate ceiling. The tell is a `get` of a known-large key followed by slicing/seeking in the Worker — push the slice into the `range` so R2 returns just the window. (This is the R2 face of the over-fetch note in `../cloudflare.md`.)

- **Buffering a body that should stream straight through** (lane `payload-startup` / `data-access`): `get` returns an `R2ObjectBody` whose `body` is a `ReadableStream`; you can hand it directly to `new Response(object.body, ...)` so bytes flow client-ward without ever fully residing in the isolate. Calling `await object.arrayBuffer()` / `.text()` / `.json()` / `.blob()` materializes the *entire* object in Worker memory — fine for a small config blob, a 128 MB-cap risk and a latency floor for a large asset. Flag a full-body read where the object is only being proxied to the response or to another `put`; stream it instead.

- **The cost model is operations-based with no egress — not bytes-out** (lane `data-access` / `idiom-currency`): R2 bills **Class A** operations (`PutObject`, `CreateMultipartUpload` / `UploadPart` / `CompleteMultipartUpload`, `CopyObject`, `ListObjects`) at a much higher per-million rate than **Class B** read operations (`GetObject`, `HeadObject`), and egress is **free** (verify the operation-class membership and rates against the currency brief for your version). The consequence that flips intuition from an egress-billed store like S3: serving bytes is cheap, but a hot path that **writes** or **`list`s** per request is the expensive pattern. Audit the request path for Class A ops on every request, not for bytes shipped.

- **`list()` on the hot path / over a large prefix** (lane `data-access`): `list` is a Class A operation, paginated — it returns at most **1000** entries per call with `truncated` / `cursor` to continue, plus `prefix` and `delimiter` filters (verify against the currency brief for your version). Scanning the bucket (or a wide prefix) on every request to find an object is the anti-pattern: it pays the pricier op class, fans out into pagination, and then often a per-entry `get`. Model the key naming so the path computes the *known* key and `get`s it directly; reserve `list` for enumeration/admin. This is the same shape as KV's `list()` trap — cross-reference `kv.md`.

- **Re-reading an immutable object from R2 every request instead of caching at the edge** (lane `data-access`): a frequently-served, rarely-changing object (an asset, an export, a static dataset) refetched from R2 on each request pays a `GetObject` and a round-trip every time. Front it with the **Cache API** (`caches.default`, keyed by request) or expose it via a **public bucket on a custom domain** so Cloudflare's CDN cache serves repeat reads — hits then skip R2 entirely. Note the Cache API is **per-colo, not global** (cross-reference the Cache-API note in `../cloudflare.md`), so it's a per-edge hit-rate win; a custom domain with cache rules gives broader coverage. Set cache lifetime to the object's real mutability — caching a mutable object widens a staleness window.

- **One giant `put` for a large upload instead of multipart** (lane `data-access`): large or unreliable uploads belong in multipart (`createMultipartUpload` → `uploadPart` → `complete`), which lets parts upload in parallel and a failed part retry without restarting the whole transfer (verify against the currency brief for your version). The judgment is part *size*: every part is its own Class A op, and parts must be uniform in size except the final one — too-small parts multiply Class A operations (and brush against a max part count), too-large parts lose parallelism and retry granularity. Pick a part size that balances op-count against resumability for the object size, rather than defaulting to a single `put` for everything.

- **A full re-GET where a conditional revalidation would do** (lane `data-access`): `get`/`put` accept `onlyIf` (`etagMatches` / `etagDoesNotMatch` / `uploadedBefore` / `uploadedAfter`, or HTTP `If-Match` / `If-None-Match` / `If-Modified-Since`); a conditional `get` that the object fails returns without the body (verify against the currency brief for your version). A sync/cache-refresh loop that re-GETs an object in full just to check whether it changed should carry the prior ETag in `onlyIf` so an unchanged object costs a cheap revalidation, not a full transfer. The tell is "fetch, compare, discard if same" implemented as an unconditional `get`.

- **Streaming large client transfers through the Worker instead of a presigned URL** (lane `concurrency` / `data-access`): for big user uploads or downloads, proxying the bytes through the Worker spends CPU/duration and the request's subrequest budget on pure data movement (cross-reference the subrequest cap in `../cloudflare.md`). A **presigned URL** (S3 API, expiry 1 s–7 days) lets the client talk to R2 directly, so the Worker only mints the short-lived URL and the bytes never touch it (verify against the currency brief for your version). Offload when the object is large and the Worker adds no per-byte logic; keep it in-Worker when you must transform, authorize per-byte, or rewrite the stream.

- **Wrong storage class for the access frequency** (lane `idiom-currency`): R2 offers **Standard** and **Infrequent Access**; Infrequent Access trades a lower per-GB storage rate for a per-GB **retrieval fee** and a **30-day minimum storage duration** (verify rates/duration against the currency brief for your version). Data read often on Infrequent Access pays retrieval fees that swamp the storage saving; data churned or deleted before the minimum duration is billed as if it stayed. Match the class to real access frequency and object lifetime — backups/archives suit Infrequent Access, a hot asset path does not.

- **Many tiny objects, one `get` each** (lane `algorithmic`): R2 prices per operation, so storing thousands of small records as individual objects and `get`-ing them one-by-one multiplies Class B ops (and round-trips) where one read should suffice. Small structured/keyed data is usually a better fit for **KV** (edge-cached small values, cross-reference `kv.md`) or **D1** (queryable rows, cross-reference `d1.md`); reserve R2 for genuine blobs, and where many small items are read together, consider batching them into one object. Match the store to the object size and access granularity, not just to "we needed somewhere to put it."
