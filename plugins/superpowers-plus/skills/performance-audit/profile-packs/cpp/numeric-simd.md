## Numeric & SIMD (Eigen/BLAS, intrinsics, auto-vectorization, SoA/AoS)

> Scope: dense/sparse linear algebra libraries (Eigen, xtensor, Armadillo) and their fast paths,
> tuned BLAS/LAPACK dispatch for large matrix kernels, compiler auto-vectorization and why it
> silently fails, `-ffast-math` and its correctness hazards, data layout (SoA vs AoS) and alignment
> for SIMD, and explicit/portable SIMD for the cases auto-vectorization provably cannot reach.
> The recurring theme is: prefer enabling and *confirming* the optimizer's and the library's fast
> paths over hand-writing vector code — most numeric loops want auto-vectorization, library
> dispatch, and the right memory layout, and hand intrinsics are a last resort for a kernel where
> auto-vec was checked and failed. Many element-wise kernels are memory-bandwidth-bound, so reason
> about the roofline before reaching for SIMD or threads at all.
> Cross-reference the **Memory & allocation** and **Algorithmic complexity** lanes and the
> Compiler, build & ABI notes in `../cpp.md` (which gate which SIMD ISA the optimizer may emit via
> `-march`/`-mtune`), and the **concurrency-parallelism** module for the thread/bandwidth boundary.

- **Auto-vectorization that silently didn't happen**: the compiler only vectorizes an inner loop when it can prove the access is safe to widen — no aliasing between the input and output pointers, a trip count it can reason about, and a target ISA that actually has the needed vector width enabled (the latter is gated by `-march`; see core Compiler notes, don't restate the flag). A loop that looks trivially vectorizable (`for (i) out[i] = a[i] + b[i]`) often stays scalar because the compiler must assume `out`, `a`, `b` may overlap. Confirm what actually happened with `-fopt-info-vec`/`-fopt-info-vec-missed` (GCC) or `-Rpass=loop-vectorize`/`-Rpass-missed=loop-vectorize`/`-Rpass-analysis` (Clang) rather than assuming — "it should vectorize" is not evidence (verify against the currency brief for your version).

- **Breaking the aliasing assumption that blocks vectorization**: when the compiler reports a missed vectorization due to possible aliasing, annotating the non-overlapping pointers with `__restrict` (also spelled `__restrict__`; MSVC, GCC and Clang all accept `__restrict` on pointers — C's `restrict` keyword is not standard C++ but every major compiler honors this spelling) tells the optimizer the regions are disjoint, often unlocking the SIMD loop. This is a *promise the compiler cannot check*: if the regions ever do overlap the result is undefined, so it applies only where disjointness is a genuine invariant of the API. Prefer expressing the loop over `std::span`/iterators or splitting input and output buffers so the property holds structurally before reaching for `__restrict` (verify against the currency brief for your version).

- **`-ffast-math` unlocks vectorization but changes results — scope it, never blanket it**: floating-point addition is not associative, so a reduction (`sum += a[i]`) cannot be vectorized into parallel partial sums without permission to reassociate; `-ffast-math` (and the narrower `-fassociative-math`/`-freciprocal-math`/`-ffp-contract=fast`) grants that and also enables FMA contraction, flush-to-zero of denormals, and the assumption that no NaN/Inf occur. Those change numerical results and break any code that inspects NaN/Inf, relies on signed zero, or needs bit-reproducibility. The recurring hazard is a project-wide `-ffast-math` that silently perturbs unrelated code, including code that depends on IEEE semantics, and infects headers. Recommend it only at per-TU or per-function scope around a kernel whose tolerance is understood — and as a correctness/repro hazard to flag, not a free speed knob (verify against the currency brief for your version).

- **Eigen fast paths — expression templates, `.noalias()`, and fixed-size**: Eigen builds an expression tree so `a = b + c.cwiseProduct(d)` evaluates in a single fused loop with no temporary per operation — code that forces intermediate `MatrixXd` temporaries (assigning sub-expressions to named matrices, or calling `.eval()` reflexively) discards that fusion. For matrix products Eigen conservatively allocates an aliasing temporary on `c = a * b`; `c.noalias() = a * b` skips it when the destination is known not to alias the operands. Small fixed-size matrices (`Matrix4f`, `Matrix<double,3,3>`) live on the stack and let Eigen fully unroll and vectorize, whereas dynamic `MatrixXd` heap-allocates and keeps loop overhead — use fixed size when dimensions are small and known. Eigen also needs its vectorization enabled by a real `-march` and respects alignment; a misaligned mapped buffer falls off the vectorized path (cross-reference the version index for Eigen specifics and the core `-march` note) (verify against the currency brief for your version).

- **Large matrix ops must hit a tuned BLAS, not a hand-rolled triple loop**: a naive `for i for j for k` GEMM leaves a large factor (commonly 10–100x) on the table versus a cache-blocked, vectorized, multithreaded kernel from OpenBLAS, Intel MKL/oneMKL, BLIS, or Apple Accelerate — those are the work of specialists and are very hard to beat by hand. Confirm a numeric-heavy project actually links one (Eigen, Armadillo, NumPy-adjacent C++ all can delegate) rather than falling back to a reference/pure implementation. A storage-order mismatch is a quieter tax: BLAS is column-major (Fortran), so passing row-major data without setting the transpose flag forces a transposing copy of every matrix on each call; match the order or use the layout-aware (CBLAS) interface (verify against the currency brief for your version).

- **SoA vs AoS for SIMD and bandwidth (deeper than the core layout note)**: the core pack flags hot-field scatter at a high level; for numeric kernels the specific lever is structure-of-arrays. When a loop touches one field across many elements, an array-of-structs (`struct P{float x,y,z,w;}; vector<P>`) interleaves the cold fields into every cache line and forces gather-style strided access that SIMD cannot load as a contiguous vector; the structure-of-arrays form (separate `vector<float> x,y,z`) makes the hot field contiguous, so a vector load grabs N consecutive elements and no bandwidth is wasted on unused fields. The trade-off is that code touching *all* fields of one element (random per-element access) is better as AoS — choose by the dominant traversal, not dogmatically. `std::mdspan` (C++23) gives a layout-flexible non-owning view (`layout_right`/`layout_left`/strided) so the same kernel can be written once over either layout (verify mdspan support against the currency brief for your version).

- **Alignment for SIMD loads**: vector loads are fastest from data aligned to the vector width (`alignas(16)` SSE, `alignas(32)` AVX, `alignas(64)` AVX-512, which also matches a cache line); unaligned loads are merely slower on modern x86 but a misaligned *aligned-load* instruction faults, and some ISAs are stricter. Heap buffers from plain `new`/`malloc` only guarantee `max_align_t` (typically 16), so a wide-SIMD kernel needs an over-aligned allocator (`std::aligned_alloc`, `operator new(size, align)`, an Eigen aligned allocator, or `std::pmr`). Over-aligning everything wastes memory and can hurt packing, so align the buffers that feed hot vector loops, not every object (verify against the currency brief for your version).

- **Explicit SIMD only where auto-vec provably failed — and prefer portable forms**: when the missed-optimization report confirms a hot kernel stays scalar and restructuring/`__restrict`/`-ffast-math` cannot fix it, reach for explicit SIMD — but raw `<immintrin.h>`/`<arm_neon.h>` intrinsics are non-portable (one code path per ISA), unreadable, and a lasting maintenance cost. Prefer a portable-SIMD layer first: `std::experimental::simd` (`std::simd` is being standardized), or libraries like Google Highway, xsimd, or Vc that emit per-target code from one source. For a binary shipped to varied CPUs, runtime ISA dispatch via function multi-versioning (`__attribute__((target_clones("default","avx2","avx512f")))` or `target` plus an IFUNC resolver) lets one binary pick the best implementation at load time instead of crashing on older hardware (the core pack's `-march=native` caveat is the reason this exists) (verify against the currency brief for your version).

- **Memory-bandwidth-bound kernels don't speed up from SIMD or threads**: a loop with low arithmetic intensity — a sum, a copy, `axpy` (`y[i] += a*x[i]`), any element-wise pass over a large array that does a couple of flops per element loaded — is limited by DRAM (or last-level cache) bandwidth, not by compute. Widening it to SIMD or adding threads saturates the memory bus sooner but the throughput plateaus at the bandwidth ceiling; wall time stops improving and can regress from contention. Estimate flops-per-byte against the machine's bandwidth (a roofline) or read hardware counters (`perf stat`, LIKWID, VTune) before optimizing — the fix for a bandwidth-bound kernel is usually fewer passes / better cache reuse (fusion, blocking) or a smaller data type, not more lanes or cores (cross-reference the **concurrency-parallelism** module and the **Algorithmic complexity** lane in `../cpp.md`) (verify against the currency brief for your version).

- **Redundant transcendental and `pow` cost in hot loops**: `std::pow(x, 2)` calls into libm and is far slower than `x*x` (and `std::pow(x, 0.5)` slower than `std::sqrt(x)`); a small integer or half-integer exponent in an inner loop is a recurring, easily-missed cost. Scalar `std::sin`/`exp`/`log` in a vectorized loop also serialize it, because the standard libm calls are scalar — a vectorized math library (Intel SVML, SLEEF, or the compiler's vector-libm hookup: Clang `-fveclib=<lib>` e.g. `=libmvec`/`=SVML`/`=SLEEF`, or GCC `-mveclibabi=<svml|aocl|acml>` which also needs `-ftree-vectorize -funsafe-math-optimizations`) provides SIMD-width transcendentals so the loop stays vectorized, and a lower-accuracy fast approximation is legitimate where the precision budget allows. `std::valarray` is rarely worth reaching for — it promises expression-style numeric arrays but no mainstream implementation optimizes it competitively with Eigen or a hand kernel; treat existing `valarray` use as a candidate to replace, not a target to adopt (verify against the currency brief for your version).
