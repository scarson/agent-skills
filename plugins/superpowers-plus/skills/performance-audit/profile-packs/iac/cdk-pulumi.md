## Programmatic IaC (AWS CDK / Pulumi)

> Scope: the `synth`/`preview`/`deploy` wall-time of IaC written in a *real programming language* — not whether the provisioned infra is fast (out of scope). The defining theme is that, unlike a declarative template, **the program runs in full on every operation**: `cdk synth` and `pulumi preview` execute your app top-to-bottom, building the entire construct/resource tree in memory before anything is diffed, so the language + tree-construction cost is paid every iteration. On top of that, **asset bundling usually dominates** for Lambda/container-heavy stacks — `NodejsFunction`/`PythonFunction`/`DockerImageAsset` (CDK) bundle *at synth time*, and a Docker-based bundle spins up a container per asset. Then the diff/deploy still goes through a cloud deployment engine — **CloudFormation** for CDK (so CFN stack limits and stabilization apply — cross-reference [`cloudformation-bicep.md`](cloudformation-bicep.md)), the **Pulumi engine + provider plugins** for Pulumi. So total cost = program-run + asset-build + engine-deploy, and the audit is *which of the three dominates* before optimizing any one. Cross-reference the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) and the **State & dependency-graph model** in `../iac.md`; this file is the CDK/Pulumi provider lens — defer the lane and state-model definitions there.

### AWS CDK

- **Docker-based asset bundling as the synth/deploy long pole** (lane `payload-startup`): `NodejsFunction`, `PythonFunction`, and `DockerImageAsset` bundle their code/image *during synth*, and the bundling mode decides the cost. `NodejsFunction` uses **local esbuild if `esbuild` is installed in the project, otherwise it falls back to a Docker container** — and Docker bundling pays a container spin-up per asset (the docs explicitly recommend local esbuild, noting Docker-volume performance is "really poor" on macOS). A repo that bundles many functions in Docker when a local toolchain would do is paying that container tax on every synth; installing `esbuild` (or whatever local bundler the construct supports) flips it to in-process. The inverse trap is `forceDockerBundling: true` left on — sometimes *required* (native modules that must build in a Lambda-compatible image), so confirm it's load-bearing before removing it (verify against the currency brief for your version).

- **Per-construct asset rebuilds instead of bundle-once** (lane `payload-startup`): every asset-producing construct bundles independently at synth, so N near-identical Lambda functions each defined with their own `fromAsset`/bundling step run N separate bundles. The CDK hashes asset *contents* to a source hash and reuses unchanged outputs, but that only helps across runs once the input is stable — within a synth, distinct construct instances each do the work. Where many functions share one build output, bundle once (a shared layer, a single pre-built artifact referenced by all) rather than wiring a bundling step into each construct. Judge by whether the per-construct builds are genuinely distinct or accidentally repeated.

- **Context lookups (`Vpc.fromLookup`, `fromLookup`, AMI/AZ/SSM lookups) re-querying AWS every synth** (lane `data-access`): these methods hit AWS account APIs *at synth time* to resolve real environment values, and CDK caches the result in **`cdk.context.json`**. If that file isn't committed (or is cleared), every synth re-queries AWS — adding network round-trips to the inner loop *and* making synth non-deterministic, since the looked-up values can change between runs. The fix is structural: commit `cdk.context.json` to source control (the docs require it, alongside `cdk.json`) so synth is offline-fast and reproducible; `cdk context --reset`/`--clear` is a *deliberate* controlled refresh, not something to run habitually. A cleared/missing context file is the classic "why is synth slow and flaky in CI" cause (verify against the currency brief for your version).

- **`cdk.json` `lookups: false` / `--no-lookups` as a synth-speed guard** (lane `data-access`): setting this makes synthesis *fail* rather than silently perform an uncached account lookup — useful in CI to guarantee no synth-time AWS round-trip slips in, but it's a correctness/repeatability lever, not a blanket speedup. Flag it as the enforcement mechanism for the committed-context discipline above, not as a fix on its own (verify against the currency brief for your version).

- **CDK deploys via CloudFormation, so CFN per-stack limits drive the split** (lane `payload-startup`): synth emits a CloudFormation template per stack, and an over-large single stack hits the **CFN per-stack resource quota** and is slow to deploy/validate as one unit — pushing large apps toward **nested stacks** (`NestedStack`). The trade-offs (the quota's exact value, the nested-stack hierarchy limit, the orchestration overhead of each child being its own tracked stack) are the CFN engine's, not CDK's — cross-reference [`cloudformation-bicep.md`](cloudformation-bicep.md) for the limit and the nested-stack cost model rather than restating it here. The CDK-specific point: construct nesting is *not* stack nesting; a deep construct tree can still synth to one giant stack unless you split into `NestedStack`/separate `Stack`s.

- **`--hotswap` / `--hotswap-fallback` for the dev inner loop** (lane `algorithmic`): for supported changes (Lambda code, Step Functions definitions, ECS container images, and others — list is version-specific), `--hotswap` updates the resource *directly via service APIs instead of generating and executing a CloudFormation change set*, collapsing a multi-minute deploy to seconds. The catch is named in the docs: it **deliberately introduces drift** in the CFN stack and implies `--no-rollback`, so it is **development-only** — never production. `--hotswap-fallback` is the safer variant (full CFN deploy when a change isn't hotswappable), and `cdk watch` uses hotswap by default. Recommend it for iteration speed *with* the drift caveat stated, and reconcile with a normal deploy (`--revert-drift`) afterward (verify the supported-resource list against the currency brief for your version).

- **Serial multi-stack deploy where `--concurrency` would parallelize** (lane `concurrency`): `cdk deploy` defaults to **`--concurrency 1`** — stacks deploy one at a time. `cdk deploy --all --concurrency N` deploys independent stacks in parallel *while honoring inter-stack dependencies*, so a multi-stack app with independent stacks is leaving wall-time on the table at the default. The bound is real: the docs warn you still hit CloudFormation/account rate limits, so N isn't free to crank. Apply where the stacks are genuinely independent (no cross-stack export chain forcing order) — same independence guard as the core `concurrency` lane (verify against the currency brief for your version).

- **`--method=direct` vs the default change-set deploy for fast iteration** (lane `algorithmic`): `cdk deploy` defaults to creating a CloudFormation change set then executing it — two round-trips. `--method=direct` skips the change-set and applies immediately (typically faster, at the cost of losing the change-set preview/progress detail). Like a CFN change set, the change-set step is a diagnostic you opt into; firing it on every trivial dev iteration where nobody reads the preview is wasted latency. Production gates may *want* the change set — judge by whether the preview is consumed (verify against the currency brief for your version).

- **Large construct tree re-run every synth; `cdk watch` for iteration** (lane `payload-startup`): because `synth` runs the whole app, a large construct tree — thousands of constructs, heavy Aspects that visit every node, big in-code maps/loops generating resources — pays its full construction + Aspect-traversal cost on *every* synth, independent of how little changed. This is the programmatic analogue of template compile time. `cdk watch` re-synths/deploys incrementally on file change (hotswapping where possible) to avoid re-running the full cold loop by hand. Before optimizing app code, confirm synth time (not bundling or deploy) is actually the bottleneck — profile the three phases first.

- **Asset publishing re-uploading unchanged assets** (lane `data-access`): after synth, `cdk-assets` publishes built assets to the bootstrap S3 bucket / ECR repo before the stack deploys; assets are keyed by content hash, so an *unchanged* asset shouldn't re-upload — but a build step that perturbs the bytes (non-reproducible bundle, embedded timestamps, changing file order) busts the hash and forces re-publish every deploy. `--asset-parallelism` (publish assets concurrently) and `--asset-prebuild` (build all assets before the first stack, default `true`) tune the publish phase. Look for hash churn on assets that didn't logically change — that's needless upload on every deploy (verify against the currency brief for your version).

### Pulumi

- **`pulumi preview`/`up` run the program through the language host + provider plugins** (lane `concurrency`): Pulumi's architecture is a **language host** (executes your program, e.g. `pulumi-language-python`), the **deployment engine** (in the CLI, diffs registered resources against state), and per-provider **resource plugins** (`~/.pulumi/plugins`) that make the actual cloud API calls. Each resource registration is a round-trip to the engine and ultimately the provider, and the program runs to completion on every preview/up. `pulumi up --parallel P` bounds how many resource operations run concurrently — **default 16**, `1` for fully serial — the direct analog of Terraform's `-parallelism`. A stack capped at low parallelism (or serialized by over-broad dependencies) under-uses the available concurrency; raising it trades against provider/cloud rate limits, so it's bounded like CDK's `--concurrency` (verify the default against the currency brief for your version).

- **State/checkpoint backend choice and `pulumi refresh` cost** (lane `data-access`): Pulumi tracks each stack's state in a checkpoint held by a **backend** — Pulumi Cloud (managed, the default) or self-managed (S3, Azure Blob, GCS, Postgres, local filesystem). The engine diffs *against stored state*; it does **not** re-query live cloud state on every op (the docs note that would be a perf problem on large stacks). `pulumi refresh` is the explicit opt-in that **queries every resource in the stack from its provider** to reconcile drift — the refresh-cost analog of a Terraform refresh, scaling with resource count, so wiring it into a hot loop is wasted round-trips. A self-managed blob backend also adds per-operation checkpoint read/write latency that a slow or distant bucket can dominate. Match the backend to the iteration pattern, and keep `refresh` as a deliberate step (verify against the currency brief for your version).

- **`--target`/`--replace` for scoped ops; per-resource provider round-trip** (lane `algorithmic`): `pulumi up --target <URN>` updates only the named resources and their dependents (wildcards supported), and `--replace` forces replacement of specific resources — both let you avoid driving the *whole* stack through the engine when you're iterating on one slice. Because every resource is a provider round-trip, a stack modeled as one mega-`ComponentResource` (or a flat sea of resources with no scoping) means every preview re-walks everything; component granularity plus targeted ops keep the iteration loop proportional to what changed. Apply targeting for dev speed, not as a standing deploy mode (it can skip dependency updates) (verify against the currency brief for your version).
