# IaC performance module: CloudFormation / Azure Bicep (ARM)
> Load when CloudFormation (`*.template.yaml`/`.json`, `AWSTemplateFormatVersion`) or Azure Bicep/ARM (`*.bicep`, `azuredeploy.json`) is detected — see the module map in `../iac.md`. Core lanes + the State & dependency-graph model live in `../iac.md`; this file is the CloudFormation/Bicep lens only.

## CloudFormation / Azure Bicep (ARM)

> Scope: the *deploy-loop* wall-time and stack-operation cost of declarative cloud-native templates — how long a `create`/`update`/`change-set`/`what-if` takes, how long the template takes to compile and validate, and how much of an estate one operation churns. NOT whether the provisioned infra runs fast (that is the workload's problem, out of scope). The recurring theme is that CFN and ARM are **provider-managed-state** systems: unlike Terraform's local/remote state file, the cloud tracks stack and deployment state server-side, so there is no client `refresh` to optimize — the "what changed?" cost shows up instead as the **change-set** (CFN) or **what-if** (ARM) round-trip, an explicit extra pass you opt into. Both engines **parallelize resource creation from the implicit dependency graph** (`Ref`/`Fn::GetAtt`/`!Sub` in CFN, symbolic references in Bicep), so the two biggest self-inflicted slowdowns are *over-declaring* an explicit ordering edge that serializes independent work, and *under-splitting* a monolith so every operation processes the whole estate. Cross-reference the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) and the **State & dependency-graph model** in `../iac.md`; this file is the CFN/Bicep provider lens — defer the lane and state-model definitions there.

- **Monolithic stack approaching the per-stack resource quota** (lane `payload-startup`): a single CloudFormation stack has a hard cap on the number of resources you can *declare in one template* (currently **500** — but this was raised from the older 200, so treat the number as version-sensitive and verify against the currency brief / the live quotas page for your version). A giant stack near that ceiling is also slow to compile, validate, and deploy as one unit. Nested stacks (the `AWS::CloudFormation::Stack` resource) split the template into reusable child templates and let common configs be shared — but the nested-stack *hierarchy* has its own separate limit on resources changed per operation (verify the current value), and each child is itself a tracked stack with its own create/update/rollback lifecycle, so the split buys modularity and reuse at the cost of orchestration overhead. Recommend it for size/reuse, not as a free win.

- **`DependsOn` used where an implicit reference already orders the resources** (lane `concurrency`): CloudFormation creates, updates, and deletes resources **in parallel to the extent possible**, automatically deriving order from implicit dependencies — a `!Ref`, `!GetAtt`, or `!Sub` from resource A to resource B already forces B before A. An explicit `DependsOn` *overrides* that default parallelism and adds a hard serialization edge; over-applied (e.g. chaining resources that share no data "to be safe"), it needlessly serializes independent creation and lengthens the deploy. Reserve `DependsOn` for genuine ordering the references don't express — the documented cases are narrow (VPC-gateway attachment before gateway-dependent resources, an IAM policy before the resource that needs the permission, an ECS service after its capacity provider). A `DependsOn` next to a resource that already `!Ref`s its dependency is pure serialization to cut.

- **Change sets / drift detection run in a tight loop** (lane `data-access`): a change set is an *extra round-trip* — CFN compares the live stack against your submitted template/params to generate a preview without touching resources, and you then execute it as a second operation. Valuable before a risky update, but it doubles the API interaction, so firing one on every trivial iteration is wasted latency. Drift detection is heavier still: it queries the **actual current state of every supported resource** and diffs it against the expected template values, so cost scales with stack size; note it does **not** recurse into nested stacks (each must be drift-checked on its own) and skips resources that don't support drift. Both are diagnostics, not deploy steps — don't wire them into a hot loop where the signal isn't read (verify against the currency brief for your version).

- **One huge stack instead of a lifecycle/blast-radius split** (lane `algorithmic` / `data-access`): when everything lives in one stack, every deploy operates on the whole thing and a failed update rolls back the *entire* stack — slow and high-blast-radius. Splitting by lifecycle (network vs data vs app, or per-team) means a change touches only the relevant stack, cutting both deploy time and the radius of a rollback. The trade-off is cross-stack wiring (Exports/`Fn::ImportValue` or nested-stack outputs) and more stacks to operate. Judge by *churn rate and coupling*, not by a fixed size — a stack that rarely changes shouldn't be split just to be small.

- **StackSets fan-out without tuned operation preferences** (lane `concurrency`): StackSets replicate a stack across many accounts/Regions; the operation's wall-time is governed by its *operation preferences* — *Maximum concurrent accounts* (how many target accounts deploy at once), *Failure tolerance* (failures per Region before the operation halts), and *Region concurrency*, which **defaults to Sequential** (one Region at a time; Parallel is opt-in). For a large multi-account utility estate, leaving Region concurrency Sequential serializes a fan-out that could run Parallel, multiplying wall-time by the Region count; a conservative max-concurrent-accounts count similarly stretches a large rollout into a long tail, while too aggressive a setting risks service throttling and widens blast radius before a failure is caught. Tune these to the estate size and the change's risk — they are a throughput-vs-safety dial, not defaults to leave alone (verify the current StackSets limits/preference names against the brief for your version).

- **Treating deploy time as a template problem when the long pole is one slow-stabilizing resource** (lane `concurrency`): CloudFormation holds a stack operation until each resource reaches a stable state, and some resources stabilize slowly by nature (an RDS instance, a CloudFront distribution, an ElastiCache cluster, an ACM-validated cert). A deep dependency chain that *waits on* such a resource — or simply having one in the graph — can dominate the whole deploy regardless of how lean the rest of the template is. Before optimizing template structure, identify the critical path: if the long pole is a single stabilizing resource, parallelizing the cheap resources around it won't move the wall-clock. Cross-reference the **State & dependency-graph model** in `../iac.md`.

- **ARM/Bicep `what-if` on every iteration** (lane `data-access`): `az deployment ... what-if` (and the `-Confirm`/`--confirm-with-what-if` switch) previews changes by predicting how the template would alter **current resource state** — an extra pass that, like a CFN change set, is the provider-managed-state stand-in for a refresh. It expands nested templates up to documented limits (e.g. a cap on nested-template count and on expansion time, after which remaining changes report as `Ignore`), so on a large deployment it is not instant. Valuable as a pre-deploy gate; costly when bolted onto a fast inner loop where nobody reads the diff. Note its known noise (it can't resolve the `reference()` function, so those properties always show as changing) — don't treat what-if latency-plus-noise as a deploy step (verify against the currency brief for your version).

- **Complete deployment mode where Incremental suffices** (lane `algorithmic`): ARM's default mode is **Incremental** — it only adds/updates the resources declared in the template and leaves everything else in the resource group untouched. **Complete** mode additionally **deletes** every resource in the scope that isn't in the template, so it must evaluate the whole resource group against the template (more work, far more risk). Using Complete where Incremental would do is wasted processing and a standing footgun (a copy-loop or a conditionally-skipped resource can silently delete live infra). Microsoft now steers deletions toward **deployment stacks** and is gradually deprecating Complete mode; note also that linked/nested templates support **incremental only**, and subscription-scope deployments don't support Complete at all (verify against the currency brief for your version).

- **A single massive ARM/Bicep template instead of linked templates / modules** (lane `payload-startup`): a large monolithic template is slow to validate and deploy as one unit, the same way a monolithic CFN stack is. Splitting into linked templates (ARM) or modules (Bicep) buys modularity, reuse, and independent deployment-history entries — and, like CFN, the engine deploys the pieces in parallel unless a dependency forces order. The cost side: each linked/nested deployment is its own tracked `Microsoft.Resources/deployments` operation, and the expansion is bounded by what-if/deployment limits, so an extreme module graph trades one slow validation for many orchestrated sub-deployments. Recommend for size/reuse, not as a guaranteed speedup.

- **`bicep build` compile + server-side validation on a large module graph** (lane `payload-startup`): Bicep is a transpiler — it compiles to ARM JSON, and a Bicep module graph is emitted as **a single ARM template with nested deployments**. A very large template or deep module tree adds local compile time *and* server-side validation time before any resource is touched, so build/validate latency (not just deploy) becomes a real part of the loop on big estates. This is the IaC analogue of build cost; weigh it when a repo's `bicep build`/`az bicep build` step is slow (verify against the currency brief for your version).

- **Over-declared `dependsOn` and scope mismatch in ARM/Bicep** (lane `concurrency`): like CFN, ARM/Bicep parallelizes within a deployment and derives ordering implicitly from symbolic references — an explicit `dependsOn` is rarely needed and, when added defensively, serializes independent resources or modules. The Bicep `@batchSize(n)` decorator deliberately deploys a `for`-loop's instances serially (or in small batches) — correct for rate-limit-sensitive resources, but a needless throttle elsewhere. Deployment **scope** (resource-group vs subscription vs management-group vs tenant) also shapes the fan-out and which operations are even available (e.g. Complete mode is root/RG-level). Audit `dependsOn` and `@batchSize` for serialization the references don't require (verify against the currency brief for your version).

- **Blocking on a deployment the caller doesn't need to wait for** (lane `algorithmic`): for genuinely fire-and-forget deployments (a non-gating side deployment, a long rollout whose result a later step doesn't consume), `az ... --no-wait` returns without polling to completion instead of holding the CLI/pipeline for the full stabilization. The judgment is whether a downstream step actually depends on the result: if it does, async just moves the wait; if nothing reads it inline, blocking is pure idle time. Same shape as not-blocking on independent work in the core `concurrency` lane — only apply where the deployment is truly independent (verify against the currency brief for your version).

- **ARM `what-if`** — learn.microsoft.com/azure/azure-resource-manager/templates/deploy-what-if (predicts changes against current state; supported scopes; nested-template expansion limits = 500 templates / 5 minutes; `reference()` noise; `--confirm-with-what-if`).
