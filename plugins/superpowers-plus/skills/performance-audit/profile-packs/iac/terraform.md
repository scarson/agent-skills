# IaC performance module: Terraform / OpenTofu
> Load when `*.tf`/`*.tofu` (+ `.terraform.lock.hcl`) is detected ‚Äî see the module map in `../iac.md`. Core lanes + the State & dependency-graph model live in `../iac.md`; this file is the Terraform/OpenTofu lens only.

## Terraform / OpenTofu

> Scope: the *iteration loop* ‚Äî `plan`/`apply` wall-time, refresh cost, state-operation latency, and how much of an estate one operation touches. NOT whether the provisioned infrastructure runs fast (that's the workload's problem, out of scope). The defining fact of this engine is **client-side state**: unlike the cloud-native templates in `../iac.md` whose "what changed?" cost is a server-side change-set, Terraform holds a state file and, before every plan, **reconciles it against the live world by reading each managed resource from its provider** (the *refresh*). Two structural levers dominate everything below: how big the state is (refresh and lock cost scale with managed-resource count), and how the **dependency graph** is shaped (Terraform walks independent nodes concurrently up to `-parallelism`, so a spurious edge serializes work that could fan out). The recurring failure mode is a **monolithic root module** ‚Äî one state, hundreds‚Äìthousands of resources ‚Äî where every trivial change pays a full-estate refresh and holds a team-wide lock. Cross-reference the core lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) and the **State & dependency-graph model** in `../iac.md`; this file is the Terraform/OpenTofu provider lens ‚Äî defer the lane and state-model definitions there. OpenTofu is a drop-in fork sharing this exact perf model; the few divergences are flagged inline.

- **Monolithic state ‚áí every `plan` refreshes ALL managed resources** (lane `data-access`): before planning, Terraform reads the live state of every resource recorded in state from its provider, so a single root module/state holding hundreds‚Äìthousands of resources pays a full-estate API sweep to plan a one-line change. Cost scales with *managed-resource count, not diff size*. The fix is structural: split by lifecycle/blast-radius into separate root modules/states (or workspaces, or a Terragrunt layout) so a change plans only its slice. Big state also slows every state read/write and widens the lock window. This is the single highest-leverage finding on large estates ‚Äî weight it heavily on a data-platform/telemetry/analytics account where one mega-state is the real pain. Cross-reference the **State & dependency-graph model** in `../iac.md`.

- **`-refresh=false` / `-target` as deliberate iteration escape hatches, not defaults** (lane `algorithmic`): refresh is normally the dominant plan cost, so `-refresh=false` (skip the state‚Üîreality reconciliation) and `-target=<addr>` (plan/apply only a subset and its dependencies) cut iteration latency dramatically on a slow estate. But both trade correctness for speed: `-refresh=false` plans against possibly-stale state and can miss drift; `-target` is documented for *exceptional* recovery use and silently masks drift in everything outside the target. Name them as a conscious fast-iteration or break-glass workflow, never as a standing default in CI ‚Äî a pipeline that always passes `-target` has hidden the rest of its estate from every plan (verify against the currency brief for your version).

- **Data sources re-execute on every plan** (lane `data-access` / `algorithmic`): a data source (`aws_ami` lookup, `http`, `external`, a remote API) is *read*, not cached ‚Äî when its arguments are known at plan time it runs during the refresh phase of every plan; when they depend on resources changing this run it defers to apply. Either way a slow, paginated, or rate-limited data source taxes every single run, and several of them compound. Pin or hard-code a value that is genuinely stable (a fixed AMI ID rather than a live "most recent" lookup), or move the lookup behind a variable, so a stable input stops costing a provider round-trip on every iteration (verify against the currency brief for your version).

- **`-parallelism` is a measured trade-off, not "higher = faster"** (lane `concurrency`): Terraform walks independent graph nodes concurrently up to `-parallelism` (default **10** on both `plan` and `apply`). Raising it can speed a *wide* apply (many independent resources), but it multiplies concurrent provider API calls and will trip the provider's rate limit / throttling, which then *slows* the run via retries; lowering it is the right move when you're already being throttled. The default exists because it's a safe middle. Recommend a change only with a stated reason (graph width vs. observed throttling), and cross-reference the **State & dependency-graph model** in `../iac.md` (verify against the currency brief for your version).

- **`depends_on` serializing independent resources** (lane `concurrency`): an *implicit* dependency ‚Äî referencing another resource's attribute (`aws_iam_role.x.arn`) ‚Äî already orders the graph correctly and lets unrelated nodes run in parallel. An explicit `depends_on` adds a hard edge that serializes work the references didn't require; **module-level** `depends_on` is worst, because it gates *every* resource and data source in the module behind the entire dependency, collapsing parallelism and widening the change's blast radius. It also makes plans more conservative ‚Äî Terraform marks more values "known after apply", forcing extra replacements. Reserve it for genuine hidden dependencies (a side effect with no attribute to reference); a `depends_on` sitting next to a resource that already references its dependency is pure serialization to cut.

- **`count` index-shift churn vs `for_each` stable keys** (lane `idiom-currency`): `count` addresses instances positionally (`res[0]`, `res[1]`), so inserting or removing a *middle* element renumbers everything after it ‚Äî Terraform sees `res[2]` become a different object and destroys/recreates every downstream instance. `for_each` keys by a stable string (`res["blue"]`), so changing the collection moves only the changed key. The wrong choice silently turns a one-resource edit into a mass destroy/recreate (and on stateful resources, real downtime). Flag `count` over a list that grows/shrinks in the middle; `count` is fine for a fixed N or a simple on/off toggle (verify against the currency brief for your version).

- **Provider plugin re-download every `init` without a cache** (lane `payload-startup`): provider plugins are large (hundreds of MB) and, by default, `terraform init` downloads them *per working directory*. Set `TF_PLUGIN_CACHE_DIR` (or `plugin_cache_dir` in the CLI config) so init reuses a previously-downloaded copy across configs, or in CI cache the `.terraform` directory keyed on `.terraform.lock.hcl`. Separately, *many distinct providers* slows init regardless of cache and spins up a separate plugin process/RPC client per provider at plan/apply ‚Äî consolidating providers or splitting the config reduces that startup tax (verify against the currency brief for your version).

- **`terraform_remote_state` chains couple stacks and add backend round-trips** (lane `data-access`): reading another configuration's outputs via the `terraform_remote_state` data source fetches that state's snapshot from its backend on every plan/apply; deep chains (stack A reads B reads C) compound the round-trips and tightly couple lifecycles, so a change in a leaf can stall everything downstream. HashiCorp recommends publishing values to a neutral store (SSM Parameter Store, a config object, DNS) and reading *that*, or using a data source on the real resource, instead of reaching into another team's state. On HCP/TFE the `tfe_outputs` data source is the lighter, less-privileged path (verify against the currency brief for your version).

- **Huge `for_each`/dynamic resource counts ‚áí a large graph + provider rate limits** (lane `concurrency` / `payload-startup`): a `for_each` (or fan-out module) producing hundreds‚Äìthousands of instances builds a correspondingly large dependency graph that's slower to construct and walk, and an apply over it hammers the provider's API rate limit (interacting with the `-parallelism` bound above). Consider splitting the collection across root modules/states by blast-radius, or configuring provider-level retry/rate settings where the provider exposes them, rather than just lowering parallelism globally. Graph size is a payload-startup cost paid on *every* operation, not only on the initial create.

- **State-lock contention on a shared monolith** (lane `data-access`): backends that support locking (e.g. S3+DynamoDB, an Azure blob lease) take a lock on the whole state file for any operation that could write it, so on one big shared state every teammate's `plan`/`apply` serializes behind whoever holds it. `-lock-timeout` only *queues* the wait ‚Äî it doesn't reduce contention. The real fix is the same state split as the refresh-cost bullet: smaller, lifecycle-scoped states mean independent teams lock independent files. Treat a long lock-wait on a monolith as a symptom of under-splitting, not a tuning problem.

- **OpenTofu parity and the few perf-relevant divergences** (lane `idiom-currency`): OpenTofu is a drop-in fork with the same plan/apply/refresh/state/graph perf model ‚Äî do **not** re-audit the shared model differently. Two OpenTofu-specific surfaces are worth a perf note: (1) **state/plan encryption** is a first-class OpenTofu feature (KMS/PBKDF2/etc.) and adds an encrypt-on-write / decrypt-on-read cost to every state operation that scales with state size ‚Äî a real, if usually small, tax that a monolithic state amplifies; (2) **`for_each` on provider configurations** exists in OpenTofu (a feature Terraform historically lacked), which can replace a hand-fanned set of aliased providers ‚Äî relevant where many near-identical provider instances were the slow/verbose path. Both are version-sensitive; verify the exact feature status and any default against the currency brief for your version.

## Sources

Authoritative Terraform/OpenTofu documentation fetched for this module:

- **`terraform plan` options** ‚Äî developer.hashicorp.com, "Command: plan" (`-refresh=false` disables state‚Üîremote sync before planning; `-target` focuses on matching instances + their dependencies, intended for exceptional use; `-parallelism` limits concurrent graph operations, default **10**).
- **`terraform apply` options** ‚Äî developer.hashicorp.com, "Command: apply" (apply supports plan's planning options; `-parallelism=n` default **10** confirmed).
- **`count`** ‚Äî developer.hashicorp.com, "The count Meta-Argument" (zero-based `count.index`, `res[0]` addressing; "use `for_each` when arguments must have distinct values not derivable from an integer index").
- **`for_each`** ‚Äî developer.hashicorp.com, "The for_each Meta-Argument" (instances addressed by map key / set member, stable across collection reordering vs `count`'s shifting indices).
- **`depends_on`** ‚Äî developer.hashicorp.com, "The depends_on Meta-Argument" (implicit attribute-reference dependencies vs explicit `depends_on`; "can cause more conservative plans that replace more resources than necessary"; module-level `depends_on` gates all resources/data sources in the module, reducing parallelism).
- **Data sources** ‚Äî developer.hashicorp.com, "Data Sources" (read during refresh phase when arguments are known; deferred to apply when arguments depend on changing resources; refresh runs by default before a plan).
- **Plugin cache** ‚Äî developer.hashicorp.com, "CLI Configuration File" (`plugin_cache_dir` / `TF_PLUGIN_CACHE_DIR`; init reuses cached plugin instead of re-downloading per working directory; plugins are hundreds of MB).
- **`terraform_remote_state`** ‚Äî developer.hashicorp.com, "The terraform_remote_state Data Source" (reads another config's root-module outputs from its backend state snapshot; recommends publishing values to a separate location, or `tfe_outputs` on HCP/TFE, instead).
- **State locking** ‚Äî developer.hashicorp.com, "State: Locking" (automatic on operations that could write state; not all backends support it; `-lock=false`/`force-unlock` exist; lock prevents concurrent writes).
- **OpenTofu state encryption** ‚Äî opentofu.org, "State and Plan Encryption" (encrypts state and plan data at rest; multiple key providers ‚Äî PBKDF2, AWS/GCP/Azure KMS, OpenBao).
- **OpenTofu provider configuration** ‚Äî opentofu.org, "Providers Within Modules" (`for_each`/multiple dynamic instances of a provider configuration; a single dynamic instance can be assigned per module instance).
