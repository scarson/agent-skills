# Observability performance module: Grafana Alloy (collector tier)
> Load when a Grafana Alloy config (`config.alloy`) or a (deprecated) Grafana Agent config is detected â€” see the module map in `../observability.md`. Core lanes + the Instrumentation cost model live in `../observability.md`; this file is the collector-tier lens only.

## Grafana Alloy (collector tier)

> Scope: the **performance and cost of the telemetry-collection pipeline** between an app and its backend â€” a Grafana Alloy (or deprecated Grafana Agent) instance, typically a k8s **DaemonSet on every node**, that scrapes/receives metrics, logs, and traces, processes them, and exports to a TSDB / Loki / trace backend. This is the **agent tier** that `metrics.md` and `opentelemetry.md` *defer to*: those modules say "keep labels bounded" and "drop/relabel at the agent" and "tail sampling lives in the collector" â€” **this file is where you enforce those at the collector**. The audit weights three levers, in order: (1) **drop high-cardinality series/labels at the agent before they reach the TSDB** â€” the single biggest cost lever at this tier, and the only place you can fix metrics you don't own (third-party exporters spraying series); (2) **batch before export**; (3) **bound agent memory** so a spike doesn't OOM the node. Do **not** restate cardinality theory here â€” the product-of-label-values rule, the histogram-bucket explosion, and the head-vs-tail trade are in `../observability/metrics.md`, `../observability.md`, and `../observability/opentelemetry.md`; this lens covers the **agent-side knobs** that act on it. The lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) are defined in `../observability.md`.

- **Drop high-cardinality labels and whole noisy series AT THE AGENT, before they reach the TSDB â€” this is the top cost lever at this tier** (lane `algorithmic`): `prometheus.relabel` (and the scrape-side `metric_relabel_configs` it mirrors) "rewrites the label set of each metric passed along to the exported receiver"; an `action` of `drop` removes whole matching series, `labeldrop`/`labelkeep` strip/retain label *names*, and "If no labels remain after the relabeling rules are applied, then the metric is dropped." Prometheus calls filtering series before ingestion "a commonly used use case." This is where you enforce `metrics.md`'s "keep labels bounded" rule for metrics you **don't own** â€” a third-party exporter spraying a `pod`/`id`/`path` label is uneditable at the source but droppable here. Weight this first: every series shed at the agent is RAM/CPU/disk *never paid* downstream (verify against the currency brief for your version).

- **`otelcol.processor.batch` â€” batch before export, or pay a round-trip per item** (lane `data-access`): unbatched export issues one outgoing request per span/log/metric; the batch processor coalesces them because "Batching improves the compression of data and reduces the number of outgoing network requests required to transmit data." It's the agent-tier analog of the SDK's `BatchSpanProcessor` (`opentelemetry.md`). The knobs are `send_batch_size` (default ~2000), `timeout` (default 200ms, the max wait before flushing a partial batch), and `send_batch_max_size` (a hard cap; 0 = unbounded). Flag a pipeline that exports without batching, or a `timeout`/`send_batch_size` so small the batch never fills under real traffic (verify against the currency brief for your version).

- **`otelcol.processor.memory_limiter` â€” bound agent memory, and put it FIRST** (lane `concurrency`/`payload-startup`): without it, a telemetry spike grows the agent's heap until the process (and on a DaemonSet, a chunk of the node) OOMs. The processor "prevents out-of-memory situations" via a soft limit (over it, "data is dropped and â€¦ errors [signal] to upstream components," i.e. backpressure) and a hard limit (forced GC). **Order is load-bearing**: define `memory_limiter` early in the pipeline and `batch` *after* it â€” Grafana's own guidance is to "Define the batch processor in the pipeline after the `otelcol.processor.memory_limiter` as well as any sampling processors â€¦ Batching should happen after any processing that drops data." Flag a pipeline with no memory_limiter, or with batch ahead of the limiter / sampling (verify against the currency brief for your version).

- **Scrape interval Ă— series count = recurring agent + TSDB load** (lane `data-access`): `prometheus.scrape` "borrows the scraping behavior of Prometheus" with a default `scrape_interval` of `60s` and `scrape_timeout` of `10s`; the cost is paid *every interval, forever*, so over-frequent scrapes of large targets multiply both the agent's render/parse work and the series written downstream. The fix is interval discipline plus dropping unused targets and metrics (relabel, above) â€” not a longer `scrape_timeout`. The injected `scrape_samples_scraped` vs `scrape_samples_post_metric_relabeling` metrics are the measurement hook for how much a target costs before vs after your drops (verify against the currency brief for your version).

- **Tail sampling at the collector moves the head-vs-tail trade here â€” and it COSTS agent memory** (lane `algorithmic`/`data-access`): `otelcol.processor.tail_sampling` decides per *complete trace* (keep all errors/slow traces, sample the rest), which is the outcome-aware option `opentelemetry.md` says lives in the collector, not the SDK. The cost it warns about is real and lands *here*: the processor must buffer "all spans for a given trace â€¦ by the same collector instance" until it decides, so it holds traces in memory â€” `num_traces` (default ~50,000) sets the buffer and "Increasing the number will increase the memory usage," `decision_wait` (default ~30s) is how long each trace is held. Flag tail sampling on a memory-constrained DaemonSet, or a `decision_wait`/`num_traces` sized without regard to trace rate Ă— span size (verify against the currency brief for your version).

- **An over-built pipeline taxes every node â€” each component and processor is CPU** (lane `payload-startup`): Alloy runs components as a graph and each processor runs *per item*; a pipeline stacked with redundant transforms, filters, and relabel stages pays that cost on every span/sample/line â€” and because the agent is usually a DaemonSet, that overhead is multiplied across **every node in the cluster**. Flag a heavy component graph where fewer stages (or pushing the work to one well-placed relabel) would do; the per-node multiplier is what turns "small per-item cost" into a cluster-wide tax (verify against the currency brief for your version).

- **Filter/transform cost is real â€” necessary for cardinality control, but not free** (lane `algorithmic`): `otelcol.processor.transform`/filter and heavy `prometheus.relabel` rule chains run **per item**, and a long regex-heavy rule list executes on every sample/span that flows through. This is the explicit trade against the top bullet: dropping cardinality at the agent is the highest-value lever, but the *mechanism* (relabel/transform) has CPU cost that scales with item rate Ă— rule count. Prefer a few targeted `drop`/`labeldrop` rules over a sprawling transform chain that recomputes the same thing per item; the justification is the rule's value vs the per-item cost it adds (verify against the currency brief for your version).

- **Grafana Agent is END-OF-LIFE â€” a config still on Agent should migrate to Alloy** (lane `idiom-currency`): Grafana Agent reached **End-of-Life on November 1, 2025**; the project's own notice reads "Grafana Agent has reached End-of-Life (EOL) on November 1, 2025. We recommend migrating to the Grafana Alloy collector, which is built on the foundation of Grafana Agent Flow." A config still on Agent (static *or* flow mode) is on a maintenance dead-end and misses Alloy's batching/queueing/performance work. **Alloy is "an OpenTelemetry Collector distribution with built-in Prometheus pipelines"** â€” so the OTel Collector's own `batch`/`memory_limiter`/`tail_sampling` apply directly if that's what's deployed. Flag a deprecated Agent as a *currency* finding, not a runtime one (verify against the currency brief for your version).

- **Metamonitoring is the measurement hook â€” the agent exposes whether IT is the bottleneck** (lane `idiom-currency`): the collector emits its own internal metrics â€” exporter queue length, send-failed / dropped sample and span counts, WAL size, and the scrape `scrape_samples_*` / `scrape_duration_seconds` series â€” and these are how you tell whether the *pipeline* is the constraint rather than the app or backend. A climbing queue depth or a nonzero drop count is the agent telling you it can't keep up; a growing WAL means it can't flush. Per the cost model in `../observability.md`, profile the agent as a first-class component â€” don't assume the collector is free, measure it (verify against the currency brief for your version).
