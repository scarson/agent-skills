# Observability performance module: Structured logging
> Load when a logging framework is detected (`structlog`/`logging`/`loguru`, `pino`/`winston`, `zap`/`zerolog`/`slog`/`logrus`, `tracing`/`log`, Logback/Log4j2/SLF4J, Serilog/`Microsoft.Extensions.Logging`, `spdlog`) — see the module map in `../observability.md`. Core lanes + the Instrumentation cost model live in `../observability.md`; this file is the structured-logging lens only.

## Structured logging

> Scope: a service emitting **structured logs** through its language's main logging library. The performance story is **the cost of producing and shipping a log line on the hot path**, not whether the right things are logged (that is *coverage* — `../observability.md`'s boundary applies). Three forces dominate, and they recur identically across every ecosystem: **eager work done before the level filter decides** (building a message / calling `.toString()` / serializing a struct that's then discarded at a disabled level), **synchronous I/O on the request thread** (a log write that flushes to disk or ships over the network inline), and **unbounded volume** (DEBUG/TRACE left on in prod, per-row logging) which costs CPU to serialize *and* egress/ingest/disk to store. This module organizes those forces as **durable cross-language principles**, each naming the concrete API in each ecosystem — it is the one module that must span Python, JS/TS, Go, Rust, JVM, .NET, and C++. The "logs are volume- and serialization-bound" note in `../observability.md`'s cost model is the cross-signal frame; this is the per-library lens. Several **language packs already touch the disabled-level footgun** (e.g. Python's lazy-`%`-args bullet in `../python.md`) — this module *deepens* that into the full cross-language principle rather than restating each pack. The lanes (`algorithmic`/`memory`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) are defined in `../observability.md`; this is the structured-logging provider lens.

- **Eager argument evaluation / message building runs even when the level is disabled — the headline cross-language footgun** (lane `algorithmic`): a log call at a filtered-out level still pays to *construct its message and arguments* unless the library defers that work. The fix is the same idea everywhere — hand the logger the *template + the values* (or a lazy field) so formatting happens only after the level check passes — but the API differs per ecosystem. **Python:** an f-string/`%`-preformatted message (`logger.debug(f"val={x}")`) evaluates regardless; `%`-style lazy args (`logger.debug("val=%s", x)`) defer formatting "until it cannot be avoided," and `logger.isEnabledFor(DEBUG)` guards expensive *arguments* (`../python.md` already flags this — deepen, don't restate). **JVM (Logback/Log4j2 via SLF4J):** `log.debug("entry " + i + " is " + entry)` concatenates + calls `toString` unconditionally; parameterized `{}` messages (`log.debug("entry {} is {}", i, entry)`) defer it — SLF4J states the `{}` form "will outperform [concatenation] by a factor of at least 30, in case of a *disabled* logging statement." **.NET:** `_logger.LogDebug($"item {item}")` interpolates eagerly; a message *template* (`LogDebug("item {Item}", item)`) defers, and `LoggerMessage`/source-gen + an `IsEnabled` guard skip boxing entirely (below). **Go:** building a `fmt.Sprintf` argument is always evaluated; `zap`/`slog` typed fields defer — but note slog's caveat that "the arguments to a log call are always evaluated, even if the log event is discarded," so an *expensive* argument still needs a `LogValuer`/`Enabled` guard. **Rust `tracing`:** field values are only recorded if a subscriber is interested in that callsite. The cost being avoided: message construction, `.ToString()`, and any serialization, on *every* call at a level that emits nothing.

- **Use the level/enabled guard for genuinely expensive arguments, and cache it in a tight loop** (lane `algorithmic`): deferral handles the *formatting*, but an argument that is itself costly to compute (a DB lookup, a deep serialization) is evaluated before the logger ever sees it — so guard the call site. **Python:** `if logger.isEnabledFor(logging.DEBUG):` — and the HOWTO warns `isEnabledFor` "can itself be more expensive than you'd like" for deeply nested loggers, so **cache the boolean** in a local before a tight loop. **Go slog:** `if !logger.Enabled(ctx, slog.LevelDebug) { … }` skips the whole block; **zerolog:** `if e := log.Debug(); e.Enabled()` computes output "only if enabled." **.NET:** `if (_logger.IsEnabled(LogLevel.Debug))` — when not configured for the level, `ILogger.Log` isn't called and "value-type boxing and an allocation of `object[]` … are avoided." **JVM:** `if (log.isDebugEnabled())` (the SLF4J FAQ notes this evaluates enablement twice vs the `{}` form, so prefer `{}` and reserve the guard for *expensive-to-build* arguments). The trade-off: the guard adds a branch and only pays off when the skipped work is real — don't wrap a guard around a cheap constant message.

- **Synchronous vs asynchronous appenders/transports — logging I/O on the request thread blocks it** (lane `concurrency` / `data-access`): a log write that flushes to disk or ships over the network *inline* stalls the request thread for that I/O. An async appender enqueues and a background worker drains. **JVM:** Logback's `AsyncAppender` wraps a target appender (verify the exact `queueSize`/`discardingThreshold` defaults against the currency brief for your version), and **Log4j2 async loggers** go further — built on the **LMAX Disruptor**, "a lock-free inter-thread communication library, instead of queues, resulting in higher throughput and lower latency," with "most of the work … done on the I/O thread." **JS:** pino's asynchronous mode buffers and "write[s] them in larger chunks" for "the minimum overhead of Pino," and transports run in a worker thread off the main loop. **.NET:** a Serilog async sink wrapper moves the write off the request thread (verify). **C++:** spdlog's `async_logger` is backed by a shared `thread_pool` with a bounded queue. The trade-off is universal: async = buffered, so on a crash/power-cut the most recently buffered lines can be lost (pino names this explicitly), and a *sustained* rate above the appender's throughput just makes the app "log at the speed of the slowest appender" (Log4j2) — async dampens bursts, it doesn't raise sustained capacity. Flag inline log I/O on any latency-sensitive path; see also the synchronous-export trap in `../observability.md`.

- **The async queue forces a drop-vs-block backpressure choice — know which is configured** (lane `concurrency`): a bounded log queue, when full, either *blocks* the producer (preserves lines, couples app latency to log-sink health) or *drops* (protects latency, loses lines). **C++ spdlog** makes this explicit: `async_overflow_policy::block` (default) "block[s] the caller until there is more room" vs `overrun_oldest` which drops/replaces the oldest, selectable via `create_async_nb` or the constructor. **JVM Logback** `AsyncAppender` drops events below a `discardingThreshold` when the queue nears full unless configured otherwise (verify). The audit point matches `../observability.md`'s telemetry-export rule: a *blocking* log queue silently turns a slow log backend into an application-latency incident — surface which behavior is in effect rather than assuming async is free.

- **Zero-allocation / reflection-free loggers vs reflection-based ones — per-log allocation + GC cost** (lane `memory`): on a hot path the logger's own allocation profile matters. **Go:** `zap` ships "a reflection-free, zero-allocation JSON encoder" with typed `Field` constructors (`zap.String`, `zap.Int`), because "reflection-based serialization and string formatting are prohibitively expensive — they're CPU-intensive and make many small allocations"; `zerolog`'s chained typed API (`.Str`/`.Int`) likewise writes JSON "by avoiding allocations and reflection." Both benchmark in the hundreds of ns with a handful of allocs for a 10-field message (zap ~656 ns/5 allocs, zerolog ~767 ns/6 allocs per their READMEs); `logrus`/`fmt.Sprintf`-style logging uses reflection and allocates far more (verify the relative figures against the currency brief). zap's `SugaredLogger` (loosely-typed, printf-style) trades some speed for ergonomics — it's "4-10x faster than other structured logging packages," while the base `Logger` "allocates far less … but only supports structured logging"; pick the typed `Logger` for the hottest paths. **JS:** **pino** is "super fast" and minimal by design; **winston** is heavier/more featureful per call — prefer pino on a high-throughput service (verify the relative cost). The defect is per-log allocation churn feeding the GC on a hot path, not the logger choice in the abstract.

- **JSON/structured serialization is CPU per event — reuse encoders/buffers, don't log whole objects** (lane `algorithmic` / `memory`): encoding fields to JSON on every log event is real CPU, and the cost scales with payload size. The zero-alloc loggers above win precisely by *not* reflecting and by reusing buffers; the inverse defect is logging a large or deeply-nested object in full (a whole request/response body, an entire entity graph) so the serializer walks and allocates it on every line. **.NET** message templates with the `@` destructuring operator (`{@Order}`) serialize the whole object — cheap to write, expensive at volume; **Go** logging a struct via `zap.Any`/reflection costs more than naming the few fields that matter. Log the handful of fields a query needs, not the object; this is the log-side instance of `../observability.md`'s "over-detailed telemetry that's read by no one."

- **Log volume & level discipline in prod is the cheapest control — and sampling bounds high-frequency lines** (lane `data-access` / `payload-startup`): DEBUG/TRACE left enabled in production multiplies *every* cost — CPU to serialize, bytes on the wire (metered egress/ingest), and disk — for output no one reads. The level is the single cheapest lever; set it correctly before optimizing anything else. For a line that *must* exist on a high-frequency path, **per-message sampling** caps it: **zerolog** offers `BasicSampler{N: 10}` / burst `LevelSampler`; Serilog and other stacks expose sampling/throttling sinks (verify the specific API against the currency brief). This is the logging face of `../observability.md`'s "ingest/egress volume is a metered cost" — and the asymmetry from the cost model holds: filtering by level can skip the *work*, whereas sampling after the message is built still paid to build it.

- **Logging on a hot loop / per-row multiplies a per-call cost by the iteration count** (lane `algorithmic`): a log call *inside* a `for`/`map`/per-row iteration pays its full construct-serialize-enqueue cost on every iteration — the same shape as the N+1 trap on the data-access side. Even a "cheap" structured line becomes a hot-spot when the loop runs hundreds of thousands of times. Hoist the log out of the loop (log a summary, or every Nth via a sampler), or guard it with a cached `Enabled` boolean (above). The tell is a logging call whose level/payload doesn't vary with the iteration sitting inside an iteration construct — it almost never needs to be there per-item.

- **Resolve the logger and bind context fields once — don't rebuild them per call** (lane `algorithmic` / `memory`): re-resolving a named logger, or re-attaching the same context fields (request id, tenant, trace id) on every call, repeats work that a bound child logger does once. **Go slog:** `logger.With("url", r.URL)` returns a logger where "built-in handlers will format that attribute only once, at the call to `Logger.With`"; **zap:** `logger.With(fields...)`; **pino/structlog:** child loggers (`logger.child({…})` / `structlog`'s `bind`); **Serilog:** `ForContext`. Bind the request-scoped context at the boundary and reuse the child, rather than passing the same fields into every call site. The cost avoided is repeated field assembly/serialization of context that doesn't change within the scope.

- **Compile-time vs runtime level filtering (Rust): strip disabled levels at build time for zero runtime cost** (lane `algorithmic` / `idiom-currency`): Rust's `tracing`/`log` expose Cargo features — `max_level_*` and `release_max_level_*` (e.g. `release_max_level_info`) — that set `STATIC_MAX_LEVEL` so "trace instrumentation at disabled levels will be skipped and will not even be present in the resulting binary." A release build compiled with `release_max_level_info` has its `trace!`/`debug!` callsites *removed*, not merely runtime-checked — strictly cheaper than the runtime filter, which still evaluates an enabled check per call. Flag a hot Rust service relying solely on a runtime level filter where a `release_max_level_*` feature would compile the verbose callsites out entirely (verify the exact feature names against the currency brief for your version); see `../rust.md` for the broader build-config lane.

- **Exception/stack-trace capture is expensive on a hot error path** (lane `algorithmic`): logging an exception *with its full stack trace* pays for the trace walk and symbolization, which is far costlier than a plain message — fine for a genuine error, a problem when an expected/handled condition is logged-with-exception inside a hot loop or on a high-frequency path (e.g. a validation failure logged at ERROR with a full trace per request). The capture cost is the same idea across runtimes (JVM `Throwable` fill-in-stack-trace, .NET exception `ToString`, Go error-with-stack libraries). Flag full-trace logging of routine, high-frequency conditions; reserve stack traces for genuinely exceptional paths and log the rest as a bounded message.

## Sources

Load-bearing per-language specifics — the disabled-level deferral idiom, the async-appender trade-offs, the zero-allocation logger claims, and the compile-time level-strip — are grounded in official docs / library READMEs fetched for this module. Concrete defaults (queue sizes, discard thresholds) and the relative benchmark figures carry verify-tags because they move. **Verified by fetch:** Python lazy `%`/`isEnabledFor`; SLF4J parameterized `{}`; Log4j2 async loggers + Disruptor; pino async/low-overhead; zap zero-alloc + Sugared/Logger trade-off; zerolog zero-alloc + `Enabled` + sampling; Go slog typed Attrs / `Enabled` / `With` / "args always evaluated"; .NET `LoggerMessage` source-gen + `IsEnabled`; Serilog message templates; spdlog `async_logger` + overflow policy; Rust `tracing` `max_level_*`/`release_max_level_*`. **Left verify-tagged (NOT confirmed by fetch):** Logback `AsyncAppender` `queueSize`/`discardingThreshold` defaults and drop behavior; winston's relative per-call cost vs pino; `logrus`'s reflection-allocation figures (the *principle* that reflection-based serialization is expensive is grounded via zap, but the specific logrus numbers are not); Serilog async-sink and sampling/throttling API names. Treat these as judgment-level claims to confirm against the library docs/currency brief, not recited facts.

- **Python — Logging HOWTO** (docs.python.org/3/howto/logging.html): "Formatting of message arguments is deferred until it cannot be avoided"; `isEnabledFor()` to skip expensive argument computation, with the worked `if logger.isEnabledFor(logging.DEBUG):` example; the caveat that `isEnabledFor()` "can itself be more expensive than you'd like" for deeply nested loggers and the advice to cache its result in a local for tight loops.
- **SLF4J FAQ — "What is the fastest way of (not) logging?"** (slf4j.org/faq.html): parameterized `{}` messages format and substitute "only if the decision [to log] is affirmative"; the `{}` form "will outperform [concatenation] by a factor of at least 30, in case of a *disabled* logging statement"; `isDebugEnabled()` guard evaluates enablement twice.
- **Log4j2 — Async Loggers** (logging.apache.org/log4j/2.x/manual/async.html): async loggers "based on LMAX Disruptor, a lock-free inter-thread communication library, instead of queues, resulting in higher throughput and lower latency"; "most of the work is done on the I/O thread"; sustained-rate caveat ("the application will end up logging at the speed of the slowest appender"); harder error signaling in async; audit logging recommended synchronous.
- **pino — Asynchronous logging** (github.com/pinojs/pino, docs/asynchronous.md): "Asynchronous logging enables the minimum overhead of Pino"; buffers and writes "in larger chunks"; "possibility of the most recently buffered log messages being lost in case of a system failure"; no one-to-one relationship between log calls and writes. Site tagline (getpino.io): "Super fast, all natural JSON logger for Node.js."
- **zap — README** (github.com/uber-go/zap): "Blazing fast, structured, leveled logging"; "reflection-free, zero-allocation JSON encoder"; "reflection-based serialization and string formatting are prohibitively expensive — they're CPU-intensive and make many small allocations"; `SugaredLogger` "4-10x faster than other structured logging packages", base `Logger` "even faster … allocates far less, but only supports structured logging"; ~656 ns/op, 5 allocs for a 10-field message.
- **zerolog — README** (github.com/rs/zerolog): "fast and simple logger dedicated to JSON output"; chaining API "write[s] JSON (or CBOR) log events by avoiding allocations and reflection"; typed `.Str()`/`.Int()`/`.Float64()`; `if e := log.Debug(); e.Enabled()` to "compute log output only if enabled"; ~767 ns/op vs zap; `BasicSampler{N: 10}` and `LevelSampler` burst sampling.
- **Go — `log/slog`** (pkg.go.dev/log/slog): `LogAttrs` with typed `slog.String`/`slog.Int` for allocation-free logging; `Logger.Enabled`/`Handler.Enabled` to skip work; "the arguments to a log call are always evaluated, even if the log event is discarded"; `LogValuer` to defer expensive values; `Logger.With` — "built-in handlers will format that attribute only once, at the call to `Logger.With`."
- **.NET — High-performance logging** (learn.microsoft.com/en-us/dotnet/core/extensions/logging/high-performance-logging): `LoggerMessageAttribute` + compile-time source generation "eliminat[es] boxing, temporary allocations, and message template parsing at runtime"; extension methods box value types and "parse the message template … every time"; `ILogger.IsEnabled(LogLevel)` guard — when not configured, `ILogger.Log` isn't called and "value-type boxing and an allocation of `object[]` … are avoided"; message templates are templates, not interpolated strings.
- **Serilog — Writing Log Events** (github.com/serilog/serilog/wiki/Writing-Log-Events): use message templates with named holes, not concatenation — "Treating the string parameter … as a message … will degrade performance and consume cache memory"; "Don't: `Log.Information("The time is " + DateTime.Now)`" vs "Do: `Log.Information("The time is {Now}", DateTime.Now)`"; events have a template "*not* a message"; `@`/`$` capturing operators.
- **spdlog — Asynchronous Logging** (github.com/gabime/spdlog/wiki/Asynchronous-Logging): `async_logger` backed by a shared `thread_pool` with a pre-allocated bounded queue; overflow policies `block` (default, "block the caller until there is more room") vs `overrun_oldest` (drop/replace oldest), via `create_async_nb` / `spdlog::async_overflow_policy`; single worker thread by default to preserve message order.
- **Rust `tracing` — level filters** (docs.rs/tracing/latest/tracing/level_filters/): compile-time Cargo features `max_level_*` and `release_max_level_*` set `STATIC_MAX_LEVEL`; "trace instrumentation at disabled levels will be skipped and will not even be present in the resulting binary unless the verbosity level is specified dynamically" — i.e. zero runtime cost vs the runtime dynamic filter.
