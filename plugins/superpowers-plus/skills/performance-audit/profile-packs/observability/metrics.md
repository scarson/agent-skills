# Observability performance module: Metrics (Prometheus / StatsD)
> Load when a metrics client (`prom-client`, `prometheus_client`, `prometheus/client_golang`, `micrometer`, `prometheus-net`, StatsD) is detected — see the module map in `../observability.md`. Core lanes + the Instrumentation cost model live in `../observability.md`; this file is the Metrics lens only.

## Metrics (Prometheus / StatsD)

> Scope: a service exporting **Prometheus / OTel / StatsD** metrics. The performance story here is **cardinality before anything else** — every unique combination of label values is a *separate active time series*, and the TSDB's RAM, CPU, disk, and query speed scale with **active series**, so a single unbounded label can OOM the backend and slow every scrape. This module is the metrics-specific lens on three load-bearing facts: cardinality is the **product** of each label's distinct values (not the sum); each classic **histogram bucket is its own series**, multiplying against every other label; and the recording call itself can take a **registry lock or label lookup** on a hot path. The fix vocabulary is bounded label *values*, route-template (not raw-path) labels, deliberate bucket choice, cached metric handles, and pushing high-cardinality identifiers into **exemplars / logs / traces** rather than labels. The cross-signal cardinality framing and the instrumentation cost model live in `../observability.md` (the `algorithmic` lane and the "Metrics are cardinality-bound" note); read this for the client-library surface. Defer scrape-side relabel/drop and agent batching to the **agent/collector** lens (`grafana-alloy.md`). The lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) are defined in `../observability.md`; this is the metrics provider lens.

- **Cardinality = the product of every label's distinct values, and active series is what bounds the TSDB** (lane `algorithmic`): Prometheus is explicit that "every unique combination of key-value label pairs represents a new time series," and each labelset is "an additional time series that has RAM, CPU, disk, and network costs." So a metric with labels of 5, 20, and 50 distinct values is up to 5,000 series *from one metric* — combinations multiply, they don't add. This is the #1 cost on the metrics signal and the thing to weight first: the active-series count, not the observation rate, is what OOMs the backend and slows every query. The docs put soft budgets in the single-to-low-double digits per metric and warn that millions of series "is too much for the current implementation of Prometheus" (verify against the currency brief for your version).

- **An unbounded label value is the cardinality killer — put the identifier in an exemplar/log/trace, not a label** (lane `algorithmic`): the docs name the offenders directly — "Do not use labels to store dimensions with high cardinality (many different label values), such as user IDs, email addresses, or other unbounded sets of values." A `user_id`, `request_id`, `email`, session token, raw error message, or full URL path turns one metric into *one series per distinct value, forever*. The defect is the dimension's **unboundedness**, not the label count: ten stable values is fine, ten million is fatal. High-cardinality identifiers belong on a **trace/exemplar** (an exemplar attaches a `trace_id` to an observation "without inflating your cardinality") or in structured logs — never as a label (verify against the currency brief for your version).

- **Label a route by its template, not the raw path** (lane `algorithmic`): a `path="/user/12345"` label produces one series *per id*; the normalized `route="/user/:id"` produces one series for the whole endpoint. This is the single most common way an HTTP-metrics middleware silently leaks cardinality — the raw URL, raw query string, or raw status *message* (vs the bounded status *code*) all do it. Confirm the instrumentation normalizes to the route/handler template before recording, and that the status label is the numeric code (or a small class like `2xx`/`5xx`), not free text.

- **Each classic histogram bucket is its own series, and buckets multiply against every other label** (lane `algorithmic` / `data-access`): the docs state "each bucket configured … will create a series suffixed with `_bucket`, no matter if the bucket is populated or not," *plus* `_sum` and `_count`. So a histogram's series cost is `(bucket_count + 2) × (product of label cardinalities)` — a 12-bucket latency histogram with a per-endpoint label is an order of magnitude more series than a counter on the same labels. Audit bucket count and label set *together*; an over-bucketed histogram across many label combos is a quiet cardinality multiplier. Choose buckets deliberately for the SLO you query, not "just in case" (verify against the currency brief for your version).

- **Labels whose value domain grows over time leak series even if each instant looks bounded** (lane `algorithmic`): `version`, `build_id`, `commit_sha`, `deploy_timestamp`, or a queue/job id as a label adds a *new* series on every deploy/build, and old series linger until staleness expiry rather than disappearing. The instantaneous cardinality looks small, but the series *accumulate* across the retention window. Flag any label whose set of possible values is monotonically growing with time — it's an unbounded label wearing a bounded-looking disguise.

- **Cache the metric handle; don't re-resolve labels on every observation** (lane `concurrency`): resolving a child metric by label values has real cost, and `client_golang` says it outright — "If you have to access the same set of labels very frequently, it might be good to retrieve the metric only once and keep a handle to it" (cache the `WithLabelValues(...)` result, "should be considered if performance is critical"). Creating a *new* child (a label combination seen for the first time) is costlier than updating an existing one because it inserts into the registry's map. On a hot path, pre-declare/resolve the label set once and reuse the handle; calling `.WithLabelValues(...).Inc()` inline re-does the lookup every call (verify against the currency brief for your version).

- **Registry lock contention serializes hot threads observing a shared metric** (lane `concurrency`): some clients guard the registry or a shared metric with a lock taken on *each* `.inc()`/`.observe()`, so a single hot metric observed from many threads becomes a synchronization point — contention scales with concurrency and observation frequency, *not* cardinality. The first observation of a new label combination is the worst case (map insert under the lock). Verify the specific client's model before asserting contention: atomic-counter clients don't take a lock for an existing child, lock-per-observe clients do (verify against the currency brief for your version).

- **`/metrics` scrape serializes every registered series each interval — registry size is a recurring CPU + payload cost** (lane `data-access`): a pull scrape transfers the full exposition of *all* active series every scrape, so the cost is `scrape_interval × series_count` paid forever, and the cardinality explosion that bloats the TSDB also bloats every scrape's render CPU, payload bytes, and parse time. The fix is the cardinality fix plus dropping unused metrics — not a bigger scrape timeout. Drop/relabel at the agent to shed series before storage (defer to `grafana-alloy.md`).

- **StatsD emits a UDP packet per event; Prometheus pull aggregates in-process — name the trade** (lane `data-access` / `concurrency`): naive StatsD sends one UDP datagram *per metric event*, which floods the network and the receiver on a hot path; client-side aggregation/buffering coalesces many events into one packet and is the fix. Prometheus's pull model instead aggregates in-process (a counter is a cheap atomic add; the cost moves to scrape-render, above), so the per-event cost is far lower but cardinality and scrape size become the dominant concern. Flag unbuffered per-event StatsD on a high-frequency path; for Prometheus, the per-event cost is rarely the problem — the series count is.

- **Match the instrument to the need — a summary computes quantiles client-side and can't aggregate** (lane `idiom-currency`): the docs are explicit that summaries "calculate streaming φ-quantiles within the instrumented program" (per-observation CPU) and "cannot be aggregated" across instances, while histograms expose bucket counts and let the *server* compute quantiles via `histogram_quantile()` — aggregatable, at the cost of bucket series. So a summary trades CPU + non-aggregatability for no bucket cardinality; a histogram trades buckets for cross-instance aggregation. Pick deliberately: a counter for monotonic counts, a gauge for an up/down value, a histogram for an aggregatable distribution, a summary only when client-side exact quantiles on a single instance are genuinely needed.

- **Native/exponential histograms, OTel↔Prometheus convergence, and exemplars are currency-sensitive** (lane `idiom-currency`): native (exponential) histograms collapse the per-bucket series explosion into "one (with a composite sample type)" series per histogram and the docs now say "if you can, use native histograms and prefer them over both classic histograms and summaries" — but availability and stability move, so consult the brief before recommending them. Likewise the OTel-metrics ↔ Prometheus-client convergence (OTLP export, delta vs cumulative temporality) and exemplar support (OpenMetrics exposition + `--enable-feature=exemplar-storage` on the server) are version-gated. Offline, flag at LOW confidence for manual currency check (verify against the currency brief for your version).

## Sources

Load-bearing specifics — the cardinality = product-of-label-values rule, the histogram-bucket = a series each fact, instrument-type trade-offs, and the client-side cache-the-handle guidance — are grounded in official Prometheus and client-library documentation fetched for this module. Concrete defaults and feature-availability claims carry verify-tags because they move.

- **Prometheus — metric and label naming** (prometheus.io/docs/practices/naming/): "Every unique combination of key-value label pairs represents a new time series, which can dramatically increase the amount of data stored"; "Do not use labels to store dimensions with high cardinality (many different label values), such as user IDs, email addresses, or other unbounded sets of values."
- **Prometheus — instrumentation** (prometheus.io/docs/practices/instrumentation/): each labelset is "an additional time series that has RAM, CPU, disk, and network costs"; soft cardinality budget ("keep the cardinality of your metrics below 10 … a handful across your whole system"; investigate alternatives above ~100); the node_exporter × per-user series-explosion example, "too much for the current implementation of Prometheus"; guidance to limit metric updates and avoid labels in very-high-frequency inner loops.
- **Prometheus — histograms and summaries** (prometheus.io/docs/practices/histograms/): "each bucket configured … will create a series suffixed with `_bucket`, no matter if the bucket is populated or not"; summaries "calculate streaming φ-quantiles within the instrumented program" and "cannot be aggregated", histograms compute quantiles server-side via `histogram_quantile()`; native/exponential histograms produce "one (with a composite sample type)" series and are now the recommended default ("if you can, use native histograms and prefer them over both classic histograms and summaries").
- **Prometheus Python client — exemplars** (prometheus.github.io/client_python/instrumenting/exemplars/): exemplars attach trace identifiers to counter/histogram observations "without inflating your cardinality"; rendered only in the OpenMetrics exposition format and require the server's `--enable-feature=exemplar-storage`.
- **prometheus/client_golang — `prometheus` package** (pkg.go.dev/github.com/prometheus/client_golang/prometheus): "If you have to access the same set of labels very frequently, it might be good to retrieve the metric only once and keep a handle to it"; keeping the returned Counter "should be considered if performance is critical" — i.e. cache the `WithLabelValues(...)` handle rather than re-resolving labels on every observation.
