# Observability performance module: OpenTelemetry
> Load when OpenTelemetry (`@opentelemetry/*`, `opentelemetry-sdk`, `go.opentelemetry.io/otel`, `OpenTelemetry.*`) is detected — see the module map in `../observability.md`. Core lanes + the Instrumentation cost model live in `../observability.md`; this file is the OpenTelemetry lens only.

## OpenTelemetry

> Scope: the **performance cost of OTel instrumentation** in a service that wires up the OTel SDK — tracer/meter/logger providers, span/log-record processors, OTLP exporters, samplers, and (optionally) zero-code auto-instrumentation, possibly fronted by an OTel Collector. This is an **overhead** lens: every span created, attribute built, metric observed, context propagated, and batch exported is CPU/allocation/latency/egress *added* to the workload — never a question of whether the service has *enough* tracing (that is coverage, and belongs to the skill's Measurability step). Two questions catch most findings: **"is telemetry export on the request thread, or off it?"** (synchronous per-event export is the classic hot-path killer) and **"how much does this cost *per span/observation* even when the data is sampled or dropped?"** (span/attribute construction is paid before the sampler or queue ever sees it). The standard lanes (`algorithmic`/`data-access`/`concurrency`/`payload-startup`/`idiom-currency`) and the **Instrumentation cost model** (head-vs-tail sampling, async+batched export, the observer effect) are defined in `../observability.md` — defer to it and read this for the OTel-specific surface. Metric label-cardinality depth lives in `../observability/metrics.md`; log-volume/serialization depth in `../observability/logging.md`.

- **`SimpleSpanProcessor` in a service is a hot-path killer — production wants `BatchSpanProcessor`** (lane `concurrency`/`data-access`): the simple processor passes finished spans to the exporter "as soon as they are finished" and serializes those export calls, so the code that ended the span **blocks on the export I/O** — one network round-trip per span on the calling thread. Fine for a debug/console exporter or a test; a throughput killer in a service. `BatchSpanProcessor` "create[s] batches of finished spans" and exports them asynchronously on a background worker, so the request thread enqueues and returns. Flag a `SimpleSpanProcessor` (or the synchronous **log-record** processor — the same Simple-vs-Batch split applies to the logs signal) on any non-trivial-volume path (verify against the currency brief for your version).

- **Head sampling at the SDK is cheap but blind to outcome; tail sampling sees the whole trace but costs the Collector** (lane `algorithmic`/`data-access`): exporting 100% of spans in a high-throughput service is CPU + egress + backend-storage cost. `ParentBased(TraceIdRatioBased)` head-samples a fraction by a deterministic hash of the trace id (consistent across the distributed trace), and is "easy to configure" and "efficient" — but decides *early*, so it "cannot make decisions based on complete trace data" and may sample away the very traces that contained an error. **Tail sampling** (keep all error/slow traces) lives in the **Collector**, not the SDK, and is "difficult to operate" because it is stateful and resource-intensive. Name the trade-off rather than prescribe: head = cheap, outcome-blind; tail = outcome-aware, costs Collector memory. `AlwaysOn` ("Returns `RECORD_AND_SAMPLE` always") on a hot service is the thing to flag (verify against the currency brief for your version).

- **Sampling-after-creation still pays the span work — only head sampling can skip it** (lane `algorithmic`): note the cost asymmetry. A head sampler can decline a span before its body runs, skipping the per-span allocation/attribute work; **tail sampling still *produces* every span** (created, attributed, exported to the Collector) before the decision, so work done for a span dropped at the tail is paid in full at the app and on the wire. "We tail-sample" is not a reason for unbounded span/attribute work in the process — that cost lands regardless (verify against the currency brief for your version).

- **High-cardinality span attributes blow up backend cost; attribute construction is CPU paid per span** (lane `algorithmic`): two distinct costs. (1) A high-cardinality attribute (`user.id`, a full URL with query string, a raw error string) multiplies backend indexing/storage/query cost the same way a high-cardinality metric label does — defer the cardinality *depth* to `../observability/metrics.md`, but name it here. (2) **Building** the attribute — string interpolation, struct serialization, an expensive accessor — is CPU + allocation spent *per span* before any sampler or queue sees it, so it is pure waste on a span that's sampled or dropped. Flag eager, expensive attribute construction on a hot span where a guard or lazy form would skip it.

- **Zero-code auto-instrumentation instruments *everything* — targeted manual spans cost far less on a hot path** (lane `payload-startup`/`algorithmic`): auto-instrumentation (the Java agent, `opentelemetry-instrument`, eBPF/bytecode hooks) "adds the OpenTelemetry API and SDK capabilities to your application typically as an agent" via "bytecode manipulation, monkey patching, or eBPF," wrapping every supported library — "requests and responses, database calls, message queue calls" — automatically. It is the fast way to *get* coverage, but it pays per-call interception overhead **across the board** plus a startup/agent-attach cost, and it produces spans for internal library paths that don't need them. Flag blanket auto-instrumentation on a very hot service (especially if it wraps a hot *internal* library); prefer a few targeted manual spans at the operation boundaries that matter (verify against the currency brief for your version).

- **A span per inner-loop iteration is per-span allocation + context work — instrument at the operation boundary** (lane `algorithmic`): starting a span per row, per item, or per iteration of a tight loop pays span allocation, attribute setup, and active-context bookkeeping on every pass, turning a negligible-per-call cost into a hot-spot because the call *is* the hot path. Instrument the *operation* (one span around the batch/loop), not the inner iteration. The justification is the call frequency, not tidiness.

- **Exporter protocol and `BatchSpanProcessor` queue config decide drop-vs-block and round-trip count** (lane `data-access`): OTLP over gRPC vs HTTP, plus the batch queue knobs, set the export profile. `BatchSpanProcessor` defaults `maxQueueSize` **2048** (spans are dropped once it fills), `maxExportBatchSize` **512**, `scheduledDelayMillis` **5000**, `exportTimeoutMillis` **30000** — an undersized queue silently drops under load, while too-small a batch (or an effectively unbatched exporter) round-trips per export. Audit the queue size and batch/delay together against the service's span rate (verify against the currency brief for your version).

- **The Collector `batch` + `memory_limiter` processors are the server-side analog — and ordering matters** (lane `data-access`): routing telemetry through an OTel Collector adds a hop with its own batching/buffering. The **batch** processor "places [spans/metrics/logs] into batches" because "batching helps better compress the data and reduce the number of outgoing connections" (defaults `send_batch_size` **8192**, `timeout` **200ms**, `send_batch_max_size` **0** = no upper bound). The **memory_limiter** processor "prevents the Collector from consuming excessive memory" (e.g. `limit_mib` / `spike_limit_mib` / `check_interval`). Order is load-bearing: batch "should be defined in the pipeline after the `memory_limiter` as well as any sampling processors" so batching happens after drops. Flag a Collector with no memory_limiter, no batching, or the wrong processor order (verify against the currency brief for your version).

- **Async (observable) instruments run their callback once per collection — a slow callback taxes every export** (lane `data-access`/`idiom-currency`): an Asynchronous Counter/UpDownCounter or Observable Gauge is "collected once for each export," i.e. its callback fires on every collection/scrape cycle. A callback that does real work — a syscall, a lock, an allocation, a directory walk — runs at the export interval whether or not anyone reads the value, so an expensive observable callback is a recurring tax on every scrape. Separately, **temporality** affects payload/statefulness: cumulative "requires the sender to remember all previous measurements, an 'up-front' memory cost proportional to cardinality" (stateful client), while delta "shift[s] the cost of cardinality outside of the process" (stateless client, stateful server). Flag heavy observable callbacks and a temporality mismatched to the backend (verify against the currency brief for your version).

- **Context propagation is a small but real per-request tax — a heavy custom propagator is a hot-path cost** (lane `concurrency`): propagation "serializes or deserializes the context object" to move trace context across process boundaries — W3C `traceparent` inject on every outbound request, extract on every inbound one — plus the cost of carrying active context across async boundaries (continuation/`Activity`/`contextvars` plumbing). The default W3C tracecontext path is cheap, but a **custom propagator** doing heavy work (extra hashing, lookups, large baggage) per request, or baggage that grows unbounded and is injected on every hop, is a tax that scales with request rate. Flag a costly custom propagator or large propagated baggage on a hot path.

- **Resource detectors that probe the environment add cold-start latency** (lane `payload-startup`): SDK init wires up providers, processors, exporters, and **resource detection**; detectors that probe cloud/k8s **metadata endpoints** (or an exporter resolving/handshaking an endpoint) make network round-trips during startup. Usually minor, but on a cold-start-sensitive surface (serverless, short-lived CLI) synchronous environment probing at init is on the critical path. Flag heavy resource detection / synchronous endpoint discovery where cold start matters (verify against the currency brief for your version).

- **OTel API/SDK/exporter versions drift independently — pin and verify against the brief** (lane `idiom-currency`): the API, SDK, exporters, and instrumentation packages version separately and the spec moves — the OpenCensus→OTel migration, exporter renames and the OTLP convergence (legacy Jaeger/Zipkin exporters superseded), the **logs** signal maturing later than traces/metrics. A far-behind or mismatched API/SDK pair can mean slower export paths, missing batching/compression options, or overhead regressions a current release fixed. Consult the currency brief for the version; offline, flag at LOW confidence for manual currency check (verify against the currency brief for your version).

## Sources

Authoritative OpenTelemetry documentation fetched for this module:

- **Trace SDK specification** — opentelemetry.io/docs/specs/otel/trace/sdk/ (`SimpleSpanProcessor` exports finished spans "as soon as they are finished" and serializes export calls; `BatchSpanProcessor` "create[s] batches of finished spans" and exports asynchronously; defaults `maxQueueSize` 2048 / `maxExportBatchSize` 512 / `scheduledDelayMillis` 5000 / `exportTimeoutMillis` 30000; `ParentBased`, `TraceIdRatioBased` deterministic-hash + "MUST ignore the parent `SampledFlag`", `AlwaysOn` "Returns `RECORD_AND_SAMPLE` always").
- **Sampling** — opentelemetry.io/docs/concepts/sampling/ (head sampling "easy to configure"/"efficient" but "cannot make decisions based on complete trace data"; tail sampling outcome-aware but "difficult to operate", stateful/resource-intensive; "Sampling is one of the most effective ways to reduce the costs of observability without losing visibility").
- **Metrics signal concepts** — opentelemetry.io/docs/concepts/signals/metrics/ (Asynchronous Counter/UpDownCounter and Observable Gauge each "collected once for each export"; Views customize which instruments/attributes/aggregation are reported).
- **Metrics data model** — opentelemetry.io/docs/specs/otel/metrics/data-model/ (cumulative "requires the sender to remember all previous measurements, an 'up-front' memory cost proportional to cardinality" = stateful client; delta "shift[s] the cost of cardinality outside of the process" = stateless client, stateful server).
- **Context propagation** — opentelemetry.io/docs/concepts/context-propagation/ (Propagators "serialize or deserialize the context object"; W3C `traceparent` inject on outbound / extract on inbound; propagation across process/network boundaries).
- **Zero-code instrumentation** — opentelemetry.io/docs/concepts/instrumentation/zero-code/ (adds OTel "typically as an agent" via "bytecode manipulation, monkey patching, or eBPF"; instruments third-party libraries — "requests and responses, database calls, message queue calls" — automatically; app code not typically instrumented without code-based instrumentation).
- **Collector configuration** — opentelemetry.io/docs/collector/configuration/ (memory_limiter "prevents the Collector from consuming excessive memory", example `limit_mib` 4000 / `spike_limit_mib` 500 / `check_interval` 5s; processor order in a pipeline "determines the order of the processing operations").
- **Collector batch processor README** — github.com/open-telemetry/opentelemetry-collector → `processor/batchprocessor/README.md` ("batching helps better compress the data and reduce the number of outgoing connections"; defaults `send_batch_size` 8192 / `timeout` 200ms / `send_batch_max_size` 0; "should be defined in the pipeline after the `memory_limiter` as well as any sampling processors").

Unverified / flagged: the OTel Collector **tail_sampling** processor is named here as the home of tail sampling (consistent with the sampling concepts page placing tail sampling in the Collector), but its specific config/defaults were **not** fetched — treat tail-sampling Collector specifics as unverified and check the currency brief. All concrete defaults above carry a verify-tag because SDK/Collector defaults move between versions.
