> Load when `torch`, `tensorflow`/`keras`, or `jax`/`flax` is detected — see the module map in `../python.md`. Core lanes + Runtime & interpreter notes live in `../python.md`; this file is the Deep learning & GPU lens only.

## Deep learning & GPU (PyTorch / TensorFlow / JAX)

> Scope: training and inference performance for the three dominant array/autodiff frameworks running on (usually) a GPU. The recurring themes are: **the GPU starves when the input pipeline can't feed it** (the single most common real bottleneck — a low GPU-utilization reading with the host pinned at 100% on one core is the signature); **the device queue is asynchronous, so any host-side read of a device value forces a synchronization** that serializes the otherwise-overlapped CPU/GPU pipeline (and silently breaks naive wall-clock timing); **the autograd graph is retained as long as a Python reference to a graph-carrying tensor lives**, so accumulating losses or logging tensors leaks memory toward OOM; and **graph-compiled frameworks recompile on input-shape/dtype change** — the dominant footgun in JAX `jit` and TF `@tf.function`, deadly for the variable-length sequence batches in load-forecasting and PMU-stream models. Cross-reference the **Concurrency** lane (GIL), the **Memory** lane, and the Runtime & interpreter notes (Python↔C boundary) in `../python.md`.

- **`DataLoader(num_workers=0)` is the default — data is decoded in the main process and the GPU idles between batches.** The default does single-process loading: every `__getitem__` and `collate_fn` runs synchronously on the training process's one core, so each step waits for I/O + decode + augment before the GPU sees a batch. Set `num_workers>0` to fetch in parallel subprocesses; this is the first thing to check when GPU utilization (via `nvidia-smi`/`nvidia-smi dmon`) sits low while one CPU core is saturated. Each worker forks and pays Python startup + dataset-copy cost, so over-provisioning workers also hurts — tune against the actual decode cost (verify against the currency brief for your version).

- **`pin_memory` and `persistent_workers` default to `False`; `prefetch_factor` defaults to 2 (and is `None` when `num_workers=0`).** `pin_memory=True` stages batches in page-locked host memory so the host→device copy can be asynchronous and overlap compute — but it only helps if the consuming `.to(device, non_blocking=True)` actually requests the async path (see below). `persistent_workers=True` stops the loader from tearing down and re-forking every worker at each epoch boundary — a large hidden cost for many short epochs (common in sequence-model training with small per-epoch windows). `prefetch_factor` controls how many batches each worker runs ahead. Cross-reference the **Data access & I/O** lane in `../python.md` (verify against the currency brief for your version).

- **A heavy Python `__getitem__`/`collate_fn` defeats the workers it runs in.** Workers parallelize across the GIL boundary (separate processes), but per-item Python-speed decoding, per-sample augmentation, or building tensors element-by-element is still slow per item — and for a fixed worker count it caps throughput. Decode/augment in batched/vectorized form, move augmentation to the GPU (e.g. a transforms-on-device or NVIDIA DALI pipeline), or pre-serialize to a format that decodes cheaply. For sequence models, windowing/normalizing each sample in pure Python inside `__getitem__` is a classic starvation source — precompute windows once or vectorize the slice. Cross-reference the Python↔C boundary note in `../python.md` (verify against the currency brief for your version).

- **CUDA operations are asynchronous, so any host-side read of a device tensor in the hot loop inserts a synchronization that serializes the queue.** `.item()`, `.cpu()`, `.numpy()`, `float(loss)`, `print(tensor)`, `.tolist()`, and boolean tests on device tensors all block the host until the queued kernels finish, collapsing CPU/GPU overlap. The frequent offender is `total_loss += loss.item()` (or `print(loss)`) called every iteration. Keep accumulation on-device — `running += loss.detach()` — and read the host value once per logging interval, not per step. (`.detach()` is what drops the graph; `.item()` additionally syncs — the two concerns are distinct, see the graph-retention bullet.) (verify against the currency brief for your version).

- **Naive `time.time()` around a forward/backward measures dispatch latency, not compute, because the kernels are still queued.** A surprisingly fast measured step is the tell. Call `torch.cuda.synchronize()` before reading the clock, or use `torch.cuda.Event(enable_timing=True)` with `start.record()`/`end.record()` then `synchronize()` + `start.elapsed_time(end)`. The identical trap exists in JAX (`block_until_ready()`) and underlies any "this op is basically free" conclusion drawn from an async framework (verify against the currency brief for your version).

- **Appending a graph-carrying tensor to a Python list retains the entire autograd graph and the activations behind it — steadily growing memory toward OOM.** `losses.append(loss)` or `outputs.append(pred)` (without `.detach()`) keeps every step's computation graph alive because the autograd engine cannot free buffers a live tensor still references. Store `loss.detach()` (stays on device, no sync) or `loss.item()` (host scalar, but syncs). Symptom: memory climbs across iterations of a loop that should be steady-state. This is the **Memory** lane's "retained references prevent reclamation" specialized to autograd (cross-reference `../python.md`) (verify against the currency brief for your version).

- **Inference without disabling grad still builds the backward graph — wasting memory and compute — and `model.eval()` does not disable it.** `model.eval()` only switches layer behavior (`Dropout` off, `BatchNorm` to running stats); it is *orthogonal* to gradient tracking. You need both: `eval()` for correct layer behavior and a no-grad context for performance. `torch.inference_mode()` is "the extreme version of no-grad" — it additionally skips version-counter bumps and view tracking for a larger speedup, with the restriction that tensors created inside it cannot later participate in an autograd-recorded computation; prefer it for pure inference (e.g. a deployed forecaster serving predictions) and fall back to `torch.no_grad()` when an output must feed back into a grad-tracked path (verify against the currency brief for your version).

- **`optimizer.zero_grad()` zeros each gradient buffer in place; `set_to_none=True` frees them instead — cheaper and the modern default behavior.** Setting grads to `None` skips a write over every parameter's gradient memory and lets the next backward allocate fresh, which is faster and slightly more memory-friendly than a memset. Forgetting to reset at all silently *accumulates* gradients across steps (a correctness bug that also inflates memory). For gradient accumulation across micro-batches this is exactly the lever: zero only once per effective batch, not per micro-batch (verify against the currency brief for your version).

- **fp32 leaves modern tensor-core throughput on the table; mixed precision and TF32 are the largest single throughput lever, but float16 needs loss scaling.** `torch.autocast` runs eligible ops in lower precision (defaults to float16 on CUDA); float16 has a narrow dynamic range, so small gradients underflow to zero — `torch.amp.GradScaler` multiplies the loss before backward to keep them representable. bfloat16 has fp32's exponent range and so needs **no** scaler, trading mantissa bits for stability (often the better default on Ampere+). Separately, TF32 (`torch.backends.cuda.matmul.allow_tf32` / the newer `set_float32_matmul_precision('high'|'medium')`) accelerates fp32 matmuls on Ampere+ at reduced mantissa precision — relevant because LSTM/GRU/transformer training is matmul-bound. Note autocast can change numerical results; validate accuracy, don't just chase the speedup (verify against the currency brief for your version).

- **Defaults that cost on convnets and fixed-shape pipelines: `cudnn.benchmark=False` and contiguous (NCHW) memory format.** `torch.backends.cudnn.benchmark=True` lets cuDNN autotune the fastest convolution algorithm — a win *only when input shapes are stable* (it re-benchmarks on each new shape, so variable shapes make it a net loss, and it introduces nondeterminism). `channels_last` memory format aligns convolutions with tensor-core-friendly layout for vision models. Both are conv-centric; for the sequence/transformer models common in load forecasting the bigger lever is `torch.compile` (graph capture + kernel fusion), which can also stall on shape churn — pin or bucket sequence lengths to avoid repeated recompiles (verify against the currency brief for your version).

- **`DataParallel` is single-process/multi-thread and bottlenecks on the GIL; `DistributedDataParallel` is one process per GPU and is the recommendation even on a single node.** The docs are explicit: DP uses multithreading (so the Python interpreter's GIL serializes the per-GPU coordination and replicates the model every step), DDP uses multiprocessing with a dedicated process per GPU, "avoid[ing] the performance overhead caused by [the] GIL." Reach for DDP for any real multi-GPU training. This is the GIL note from `../python.md`'s Runtime section made concrete at the framework layer (cross-reference Concurrency lane + Runtime notes) (verify against the currency brief for your version).

- **TensorFlow: a `tf.data` pipeline without `.prefetch`/parallel `.map` runs extract and train serially, so step time is their *sum* — the same GPU-starvation symptom.** Add `.prefetch(tf.data.AUTOTUNE)` so the input pipeline reads step s+1 while the model trains step s (reducing step time to the *max* of the two, not the sum), `.map(fn, num_parallel_calls=tf.data.AUTOTUNE)` to parallelize per-element transforms, `.cache()` to avoid re-doing deterministic preprocessing every epoch, and parallel `.interleave` for many-file reads. `AUTOTUNE` lets the runtime size buffers/parallelism dynamically rather than guessing (verify against the currency brief for your version).

- **TensorFlow: `@tf.function` retraces — recompiling the graph — on every new input *shape*, *dtype*, or *Python (non-Tensor) argument value*, and a Python scalar literal triggers a retrace per distinct value.** A `TraceType` is computed per call; a Python `int` like `num_steps=10` vs `20` is two distinct literal types → two graphs, whereas `tf.constant(10)` vs `tf.constant(20)` reuse one trace. Pin the shape with `input_signature=(tf.TensorSpec(shape=[None], ...),)` (use `None` for the dynamic/batch/time dimension so variable-length sequence batches share one graph), and pass tensors not Python literals for values that don't change graph structure. Repeated silent retracing erases the entire benefit of graph mode — the TF analog of the JAX recompile trap below, and acute for ragged sequence inputs. XLA (`jit_compile=True`) and a mixed-precision policy layer on top once retracing is controlled (verify against the currency brief for your version).

- **JAX: `jit` caches the compiled function keyed by input shape/dtype, so every new array *shape* (e.g. a different batch or sequence length) forces a full recompile — the dominant JAX footgun.** Tracing runs on abstract shaped values, so the compiled artifact is shape-specialized; feeding shape-churning batches recompiles each time (XLA compilation is slow — easily hundreds of ms). Worse, marking a Python value `static_argnums`/`static_argnames` recompiles for *every distinct value* of that argument. And calling `jax.jit` on a fresh lambda/`partial` *inside a loop* defeats the cache entirely (the docs show ~170× slowdown, 2.8ms → 477ms). Fixes: hoist the `jit` call out of the loop, pad/bucket variable-length inputs to a fixed set of shapes, and keep churning values as traced arrays rather than static args (verify against the currency brief for your version).

- **JAX: dispatch is asynchronous (results are `jax.Array` futures), so a loop that never blocks hides the real cost and benchmarks lie.** Reading shape/dtype doesn't block, but timing must call `.block_until_ready()` (the docs show 269µs measured dispatch vs ~4.92ms true compute). In the hot loop, host callbacks and `.item()`/`float(x)` force device→host transfers that serialize dispatch — keep the computation on device. Prefer `jax.vmap` over a Python loop for batching (it builds one vectorized kernel instead of unrolling), and `jax.lax.scan`/`fori_loop` over a Python `for` loop for long sequences: a value-independent Python loop is *unrolled into the trace* at compile time, inflating compile time and graph size — exactly the failure mode for long load-history or PMU time-series rollouts. `donate_argnums` lets XLA reuse an input buffer in place to cut allocation when the input is no longer needed (verify against the currency brief for your version).
