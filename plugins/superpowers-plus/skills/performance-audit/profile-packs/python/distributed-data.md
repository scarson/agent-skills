## Distributed & out-of-core data (Dask / PySpark / Ray)

> Scope: cluster and larger-than-RAM execution engines — Dask (DataFrame/Array/delayed + the distributed scheduler), PySpark (Catalyst/Spark SQL DataFrames), and Ray (tasks/actors/object store). The recurring themes: **the distributed framework is often the bottleneck, not the fix** — serialization, scheduling, and shuffle overhead are real costs a single-node pandas/Polars/DuckDB run on a 2GB extract avoids entirely, so the highest-value finding is frequently "this doesn't need a cluster"; **laziness is a graph, not a result** — these engines build a task DAG and re-execute it from scratch on every action unless you explicitly persist/cache intermediates, so repeated terminal calls silently recompute the whole lineage; **shuffles dominate** — any all-to-all repartition (groupby/join/sort/set_index) is the most expensive thing in the system and its partition count is a tuning knob, not an afterthought; **the driver is a single machine** — pulling a distributed dataset back to one process (`compute`/`collect`/`toPandas`/`ray.get`) reintroduces the memory ceiling you went distributed to escape; and **task granularity must amortize overhead** — both too-fine and too-coarse partitioning waste the cluster. This module extends the columnar/pushdown principles in [`data-stack`](data-stack.md) to the partitioned, multi-process setting; cross-reference the **Concurrency** lane and Runtime & interpreter notes in `../python.md`.

- **Reaching for a cluster when the data fits in one machine's RAM**: Dask, Spark, and Ray each add per-task scheduling (Dask quotes ~200µs–1ms per task), inter-process serialization, and (for Dask/Spark) shuffle overhead that a single in-process pandas/Polars/DuckDB run never pays — Dask's own "Start Small" guidance is to exhaust better algorithms, efficient file formats, and sampling before adding distributed machinery. For a 2GB AMI extract that fits in memory, single-node Polars-lazy or DuckDB-over-Parquet is usually faster *and* simpler than any of the three. The distributed framework earns its keep only when the working set genuinely exceeds one node's memory (billions of interval reads, multi-year SCADA history), not when a notebook author reached for `pyspark` by habit. This is the first thing to check, and the most common high-value finding (verify against the currency brief for your version).

- **Dask partition sizing far from the ~100MB sweet spot**: too many tiny partitions make scheduler overhead dominate (millions of tasks at ~200µs–1ms each push graph overhead into the 10-minutes-to-hours range per Dask's docs); too few huge ones kill parallelism and OOM individual workers. Dask's guidance is to size chunks so many fit in a worker's memory at once — roughly the low-hundreds-of-MB range (their array example moves 10MB chunks up to ~1GB on a 100GB/10-core box). `read_parquet` partitioning often inherits the source file/row-group layout, so a dataset written as thousands of tiny files arrives as thousands of tiny partitions; `repartition(partition_size="128MB")` after read fixes it (verify against the currency brief for your version).

- **Calling `.compute()` repeatedly re-runs the whole graph each time**: Dask collections are lazy task graphs, so every `.compute()` (and every `len(ddf)`, `.head()`, or implicit materialization) re-executes the DAG from the source — calling it in a loop "prohibits Dask from parallelizing different computations and from sharing intermediate results" (Dask docs). Compute multiple results in one call with `dask.compute(a, b)` so shared subgraphs run once, and use `.persist()` (distributed) to pin an expensive intermediate in cluster memory when many downstream computations branch off it. The pandas analog is the lazy-evaluation trap; the fix is the same shape as Spark `cache()` below (verify against the currency brief for your version).

- **`.compute()` to the driver then continuing in pandas — laziness thrown away**: `ddf.compute()` collects every partition into a single pandas DataFrame in the client process, reintroducing the single-machine memory ceiling and discarding parallelism for all subsequent work. `len(ddf)`, `ddf.head()`, and `.compute()` all trigger execution; a pipeline that computes early and then does the real work in pandas has used Dask only as an expensive Parquet reader. Keep operations on the Dask collection until the *reduced* result is small enough to land, or write partitioned output straight to storage with `to_parquet` rather than routing it through the driver (verify against the currency brief for your version).

- **Dask shuffle-triggering ops on non-index keys**: `set_index`, `sort_values`, `merge` on a non-index column, and `groupby(...).apply` force an all-to-all shuffle — every worker exchanges data with every other — which is the dominant cost in most Dask pipelines. Set the index *once* on the join/groupby key so subsequent `merge`/`groupby` on that key become shuffle-free, and reserve `map_partitions` for embarrassingly-parallel per-partition work (a transform that needs no cross-partition data) so it runs as one narrow task per partition with no exchange. For utility feature pipelines, partitioning Parquet by `meter_id` or `substation` up front turns later per-key aggregation into per-partition work (verify against the currency brief for your version).

- **`spark.sql.shuffle.partitions` (default 200) mismatched to data size**: this is the single highest-leverage Spark knob — it fixes the post-shuffle partition count for every join and aggregation. The 200 default is wrong for both ends: on a small result it spawns 200 near-empty tasks (pure scheduler waste); on a large one it makes each partition huge and spill/OOM. Adaptive Query Execution (`spark.sql.adaptive.enabled`, default `true` in current Spark) mitigates this by coalescing shuffle partitions at runtime (`spark.sql.adaptive.coalescePartitions.enabled`, default `true`) — so on an older cluster with AQE off, hand-tuning this knob matters far more than on a current one. Confirm AQE is actually on before assuming the default 200 is harmless (verify against the currency brief for your version).

- **`collect()` / `toPandas()` pulling the whole dataset to the driver**: both materialize *every* row in the single driver JVM/Python process and routinely OOM it — the Spark docs explicitly warn `toPandas()` (and `toArrow()`) "results in the collection of all records in the DataFrame to the driver program and should be done on a small subset of the data," *even with Arrow enabled*. Arrow (`spark.sql.execution.arrow.pyspark.enabled`, **default `false`**) only makes the transfer cheaper; it does not bound the size. Use `.limit(n)` before collecting for inspection, or `write.parquet(...)` to push results to storage and read them back single-node. A `toPandas()` on a billion-row meter table is an outage, not a slow query (verify against the currency brief for your version).

- **Plain Python UDFs serializing row-by-row through the Python worker**: a regular `udf()` ships each row from the JVM to a Python process and back one at a time and is opaque to Catalyst (no pushdown, no codegen) — the slowest option by a wide margin. Prefer, in order: built-in Spark SQL functions (`pyspark.sql.functions`, executed in the JVM); then vectorized Pandas UDFs / `mapInPandas` / `applyInPandas`, which use Arrow to move data in batches and operate on whole pandas Series at once (requires Arrow, default off); plain Python UDFs only when no vectorized form exists. A row-at-a-time UDF computing a per-reading feature over billions of rows is a classic order-of-magnitude regression versus the column expression (verify against the currency brief for your version).

- **Reused Spark DataFrame recomputed because it was never cached**: Spark DataFrames are lazy lineages — each *action* (`count`, `write`, `collect`) re-executes the entire chain from source. A DataFrame referenced by several downstream actions (e.g. a cleaned reading table feeding multiple aggregations) recomputes the full read+clean every time unless `cache()`/`persist()` is called before the first action; conversely, a `cache()` on a DataFrame used exactly once is wasted memory pressure and should be removed. Pair every `cache()`/`persist()` with `unpersist()` once the reuse window closes, or stale cached blocks evict live ones (verify against the currency brief for your version).

- **`repartition` (full shuffle) where `coalesce` (no shuffle) suffices, and missing partition pruning**: `repartition(n)` triggers a full shuffle to rebalance into `n` even partitions; `coalesce(n)` only *narrows* partition count by merging adjacent ones with no shuffle — use `coalesce` to reduce output-file count after a filter, and reserve `repartition` for genuinely rebalancing skewed data or *increasing* partitions. Separately, the columnar pushdown story from [`data-stack`](data-stack.md) applies at cluster scale: a `WHERE reading_date = ...` against date/substation-partitioned Parquet should prune to a few partitions, but only if the partition columns are real Hive partitions and the predicate references them directly — confirm via `explain()` that `PartitionFilters`/`PushedFilters` actually appear (verify against the currency brief for your version).

- **Spark join skew and missed broadcast joins**: joining a huge fact table to a small dimension via the default sort-merge join shuffles *both* sides; a broadcast join (`broadcast(df)` hint, or auto when the small side is under `spark.sql.autoBroadcastJoinThreshold`, default 10MB) ships the small table to every executor and avoids the large shuffle entirely — so a small substation/asset lookup joined without the hint is leaving an easy win on the table. Skew is the other failure mode: one hot key (a malfunctioning meter dumping disproportionate rows) creates a giant partition that straggles the whole stage; AQE skew-join handling (`spark.sql.adaptive.skewJoin.enabled`, default `true`) splits skewed partitions automatically, but only with AQE on (verify against the currency brief for your version).

- **Ray tasks too fine-grained — per-task overhead swamps the work**: Ray notes a remote task "usually comes with higher overhead than an ordinary function call"; their own example shows naive per-item parallelism running ~10x *slower* than serial, and batching (1000 items per task in the doc) running fastest. When each task does microseconds of work, scheduling and serialization dominate — batch many items per `@ray.remote` call so each does meaningful work. This is the Ray analog of rayon's too-small-`par_iter` and Dask's tiny-partition problem; the crossover is workload-dependent and must be measured (verify against the currency brief for your version).

- **`ray.get` inside a loop serializing what should run in parallel**: calling `ray.get(ref)` immediately after each `f.remote()` blocks until that one task finishes before submitting the next, serializing the whole batch — Ray flags this as the "Calling ray.get in a loop harms parallelism" anti-pattern. Submit all tasks first, collect the list of refs, then call `ray.get(refs)` once; if you want to process whichever finishes first rather than in submission order, use `ray.wait`. Relatedly, passing the *same* large object (a feature matrix, a config blob) by value into many tasks re-serializes and re-ships it every call — `ray.put(obj)` once and pass the resulting `ObjectRef` so it lands in the shared object store a single time (verify against the currency brief for your version).

- **Ray reloading expensive per-task setup instead of using an actor; object-store spill ignored**: a stateless `@ray.remote` task that loads a model (or opens a connection) on every invocation re-pays that cost per task — a stateful actor loads it once in `__init__` and reuses it across calls, which is the right tool when setup cost dominates and the object is reused (a forecasting model scored against many meter batches). Separately, when the object store fills, Ray spills objects to local disk (default under the temp dir, e.g. `/tmp/ray/session_*`) at disk speed — silent spilling shows up as a throughput cliff; treat heavy spill as a signal to shrink per-object size, raise object-store memory, or reduce in-flight objects rather than accepting the disk round-trips (verify against the currency brief for your version).

## Sources

- Dask — "Dask Best Practices" (docs.dask.org/en/stable/best-practices.html): per-task overhead 200µs–1ms, avoid `compute()` in loops / use `dask.compute(*results)`, "Start Small" before adding Dask, large-task-graph mitigation via `map_partitions`/`map_blocks`.
- Dask — "DataFrames Best Practices" (docs.dask.org/en/stable/dataframe-best-practices.html): "You should aim for partitions that have around 100MB of data each"; partitions should fit comfortably in memory (smaller than a gigabyte) but not be too many; `repartition` to fix tiny-partition layout.
- Apache Spark — "Performance Tuning" (spark.apache.org/docs/latest/sql-performance-tuning.html): `spark.sql.shuffle.partitions=200`, `spark.sql.adaptive.enabled=true`, `spark.sql.adaptive.coalescePartitions.enabled=true`, `spark.sql.autoBroadcastJoinThreshold=10485760` (10MB), `spark.sql.adaptive.skewJoin.enabled=true` (factor 5.0, threshold 256MB), broadcast hint syntax, cache/persist columnar storage.
- Apache Spark — "Apache Arrow in PySpark" (spark.apache.org/docs/latest/api/python/tutorial/sql/arrow_pandas.html): `spark.sql.execution.arrow.pyspark.enabled` "disabled by default"; `toPandas()`/`toArrow()` collect all records to the driver even with Arrow; Pandas UDFs / `mapInPandas` / `applyInPandas` use Arrow for batched vectorized execution vs row-at-a-time regular UDFs.
- Ray — "Design Patterns & Anti-patterns" index (docs.ray.io/en/latest/ray-core/patterns/index.html) and "Over-parallelizing with too fine-grained tasks" (docs.ray.io/en/latest/ray-core/patterns/too-fine-grained-tasks.html): per-task overhead > ordinary call, ~10x slowdown without batching, batch-1000 fastest; anti-patterns "Calling ray.get in a loop harms parallelism" and "Passing the same large argument by value repeatedly harms performance."
- Ray — "Object Spilling" (docs.ray.io/en/latest/ray-core/objects/object-spilling.html): spills to local-filesystem directory (default temp dir, e.g. `/tmp/ray/session_*`) once the object store is full; configurable via `object_spilling_directory`.
