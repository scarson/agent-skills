## Geospatial (GeoPandas / Shapely / rasterio)

> Scope: vector GIS on GeoPandas/Shapely (asset & network layers — poles, transformers, line segments; service-territory & outage polygons; meter→feeder→substation spatial joins) and raster GIS on rasterio/rioxarray (LiDAR, imagery, weather grids). GeoPandas is **pandas-backed**, so every footgun in [`data-stack.md`](data-stack.md) applies verbatim to the attribute columns and to the geometry column's dtype dispatch — this lens does not restate them; it covers the *spatial* axis those don't. The recurring themes are narrow and high-yield: **vectorize geometry ops** (Shapely 2.0 made them array ufuncs — a per-row `.apply(shapely_fn)` throws that away), **use the spatial index** (a brute-force pairwise predicate is O(n·m); `sjoin`/`sjoin_nearest`/`sindex` use an R-tree), **don't reproject or re-read more than needed**, and **push to the database / stay columnar** (GeoParquet, PostGIS — see [`../sql.md`](../sql.md)). The spatial-join-without-an-index and the read-the-whole-file traps are the two common big wins.

- **`.apply()` / Python loop over a geometry column defeats Shapely 2.0 vectorization**: Shapely 2.0 exposes geometry operations as **NumPy ufuncs over arrays** — `shapely.intersects(a, b)`, `shapely.buffer`, `shapely.distance`, etc. take `array_like` geometry and return arrays at C speed, and GeoSeries methods (`.buffer()`, `.distance()`, `.intersects()`, `.area`, `.centroid`) dispatch into them in one bulk call. A `gdf.geometry.apply(lambda g: g.buffer(5))` or a `for` loop calling a scalar Shapely method reverts to one Python-level call per geometry — the exact per-call Python↔C marshaling cost the **Runtime & interpreter notes** in `../python.md` warn about, multiplied by row count. Prefer the GeoSeries method or the array ufunc (verify against the currency brief for your version).

- **Nested `for` loop of `.within`/`.intersects`/`.contains` is the headline O(n·m) trap**: testing every meter against every feeder polygon (or every span against every outage polygon) by looping and calling a pairwise predicate is quadratic and the single most common avoidable cost in utility GIS. Replace with `geopandas.sjoin(left, right, predicate="intersects"|"within"|"contains")`, which builds an R-tree and evaluates only candidate pairs — `gdf.sjoin(...)` and `.clip()` use the spatial index internally. Cross-reference the **Algorithmic complexity** lane in `../python.md` (the accidental-O(n²) signal).

- **The `sindex` R-tree and querying it directly**: `GeoSeries.sindex` / `GeoDataFrame.sindex` lazily builds and caches a spatial index backed by `shapely.STRtree` (verify against the currency brief for your version). Use `sindex.query(geom_or_array, predicate=...)` to get candidate (and, with a predicate, confirmed) matches without the full pairwise scan, and `sindex.nearest(...)` for nearest-neighbor candidates — these are the right tool when a problem doesn't map cleanly onto `sjoin` (custom filtering, a one-off proximity lookup). Note `query` returns bounding-box-intersection candidates; a precise predicate still refines the set, but over an R-tree-pruned subset, not all n·m pairs.

- **Nearest-asset joins belong in `sjoin_nearest`, not a manual distance scan**: "which feeder is each pole closest to" or "nearest substation to each fault" computed by materializing an all-pairs distance matrix and taking the argmin is O(n·m) in both time and memory. `geopandas.sjoin_nearest(left, right, max_distance=..., distance_col=...)` uses the index for an R-tree nearest query; `max_distance` bounds the search radius so a sparse match doesn't scan the whole right frame (verify against the currency brief for your version).

- **Reprojecting inside a loop, or per-operation, instead of once**: `to_crs()` transforms every coordinate of every vertex — on high-vertex line/polygon assets this is not cheap. Calling it repeatedly (e.g. re-projecting one frame to match another on each iteration) repeats that whole pass. Project each layer **once** to a common, appropriate projected CRS up front. Distinguish `set_crs()` (label coordinates that already carry no CRS — no transform, cheap) from `to_crs()` (actually reproject — the expensive one); confusing them either corrupts coordinates or wastes a full pass (verify against the currency brief for your version).

- **Distance/area in a geographic CRS is both wrong and a hidden reproject trigger**: `.area`/`.length`/`.distance` on EPSG:4326 (degrees) return degree-based numbers — the GeoPandas docs warn area "may be invalid for a geographic CRS using degrees as units; use `GeoSeries.to_crs()` to project … to a planar CRS" first. Beyond correctness, code that "fixes" this by reprojecting at each measurement call pays the `to_crs` pass repeatedly; project once to a metric CRS (a UTM zone or a regional planar CRS for the service territory) and measure there. Operations across two layers in **different** CRSs also force an implicit alignment — keep all layers on one CRS.

- **The read engine and read-pushdown dominate `read_file` cost**: `geopandas.read_file` defaults to the **pyogrio** engine (vectorized GDAL/OGR), with **fiona** (row-by-row Python iteration) as the fallback — on a large asset layer the engine choice alone is a large constant factor, and `use_arrow=True` on pyogrio is documented as a further 2–4× over the default path. Just as important: push the filter into the read with `bbox=` / `mask=` (spatial pushdown), `columns=` (drop unused attribute fields), and `rows=` (row slice) so the whole statewide layer is never materialized to answer a one-substation question. This is the read-the-whole-file trap and usually the cheapest big win (verify against the currency brief for your version).

- **Shapefile/GeoJSON round-trips vs GeoParquet/Feather for repeated reads**: the core `../python.md` data-access lane already prefers columnar binary over CSV/pickle — the spatial analog is `to_parquet`/`read_parquet` (GeoParquet) and `to_feather`/`read_feather` over re-parsing Shapefiles or (worst) text GeoJSON on every run. GeoParquet preserves CRS and geometry encoding, supports column pruning, and is far faster to round-trip for an intermediate that's re-read across a pipeline; reserve Shapefile/GeoJSON for interchange at the boundary, not as a working format (verify against the currency brief for your version).

- **`overlay`, `dissolve`, and `union_all` are the expensive set ops — filter and simplify first**: a full-territory `overlay()`, a `dissolve()` aggregating many polygons, or `union_all()` (the current name; `unary_union` is the legacy alias — verify against the currency brief for your version) scale with total vertex count, so high-vertex geometries (detailed LiDAR-derived footprints, dense outage boundaries) dominate. Two levers before reaching for more hardware: pre-filter spatially (`bbox`/`sjoin`/`.cx`) so the op runs on the relevant subset, and `.simplify(tolerance, preserve_topology=...)` (Douglas–Peucker) to shed vertices that carry no analytical value at the working scale — a coarse buffer/overlay on simplified geometry is often the intended result anyway.

- **Per-pixel Python loops and whole-raster loads on rasterio**: reading a band into a full NumPy array and looping over pixels in Python is the raster form of the row-loop trap (Runtime notes: Python↔C per-call cost) — operate on the array with vectorized NumPy. For large rasters, read only what's needed: `rasterio` `Window` (`src.read(window=Window(...))`, or `Window.from_slices`) reads a subset without materializing the full array, and decimation via `out_shape`/overviews reads a downsampled view when full resolution isn't required. The read-the-whole-file trap applies to a weather grid or imagery tile exactly as to a vector layer.

- **Larger-than-RAM rasters: rioxarray + dask, and warp cost**: `rioxarray.open_rasterio(path, chunks=...)` (chunks as an int/tuple/dict, or `True`/`"auto"`) returns a **dask-backed** lazy array so a multi-GB LiDAR/weather raster is processed in chunks instead of loaded whole; the `rio` accessor's `.reproject()` is `rasterio.warp.reproject` under the hood — a full resample that is genuinely expensive, so reproject once and avoid re-warping per step, and prefer reprojecting after windowing/downsampling rather than the whole array (verify against the currency brief for your version).

- **Push the spatial filter/join to PostGIS instead of pulling everything to Python**: when the source is PostGIS, doing the spatial join or bbox filter **in the database** — against a GiST spatial index on the geometry column — and returning only the result set beats `read_postgis("SELECT * …")` followed by an in-Python `sjoin`, which transfers every row and geometry over the wire first. This is the geospatial face of the core data-access "push aggregation/filtering to the database" signal; the index, predicate (`ST_Intersects`, `ST_DWithin`), and SARGability concerns live in [`../sql.md`](../sql.md).
