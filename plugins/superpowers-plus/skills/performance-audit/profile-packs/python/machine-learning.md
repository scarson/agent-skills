## Machine learning (scikit-learn / gradient boosting / forecasting / survival)

> Scope: scikit-learn estimators and model-selection, the gradient-boosting libraries (XGBoost / LightGBM / CatBoost), classical time-series forecasting (statsmodels / Prophet / sktime), and survival/reliability modeling (lifelines / scikit-survival / `scipy.stats`) — the stack behind load forecasting and equipment-failure / predictive-maintenance work at a utility. This is a **performance** lens (training time, inference latency, memory, parallelism), not a model-accuracy lens. The recurring themes are: **build the expensive structure once and reuse it** (a binned `DMatrix`/`Dataset`, a fitted preprocessing transformer, a state-space filter) rather than rebuilding it per fold / per round / per backtest step; **count your threads** (`n_jobs` multiplied by BLAS/OpenMP threads is the classic over-subscription trap); **stay in one bulk call across the Python↔C boundary** instead of per-row Python loops; and **mind the dtype/sparsity** the matrix carries through the pipeline. Core vectorization, BLAS-linkage, and GIL realities live in `../python.md` and the Data stack module — this file deepens the model-training surfaces.

- **`n_jobs` × BLAS/OpenMP thread over-subscription — the "more workers, slower" trap**: scikit-learn's joblib parallelism (`n_jobs`) spawns workers (default backend `loky`, processes) *on top of* the BLAS/OpenMP thread pools each worker's numeric kernels spawn. A `GridSearchCV(n_jobs=8)` over a `HistGradientBoosting*` (itself OpenMP-parallel) on an 8-core box can request `8 × 8 = 64` threads, thrashing context switches and cache. Loky auto-caps inner threads to `n_cpus // n_jobs` and scikit-learn's OpenMP code uses `threadpoolctl` to avoid this — but **manually setting `OMP_NUM_THREADS`/`OPENBLAS_NUM_THREADS`/`MKL_NUM_THREADS` overrides and disables that mitigation**, so an env-var "tuning" can re-introduce the storm. This is the sklearn instance of the BLAS-oversubscription bullet in the Data stack module and the **Concurrency** lane in `../python.md`; verify the active backend and thread counts with `threadpoolctl` before adding workers (verify against the currency brief for your version).

- **Re-fitting preprocessing on every `GridSearchCV` fit instead of caching it**: a `Pipeline` re-runs every step's `fit`/`fit_transform` for each hyperparameter combination and CV fold, so an expensive transformer (PCA, `TfidfVectorizer`, feature extraction) is recomputed even when only the *final estimator's* params change. Set `Pipeline(..., memory=<dir or joblib.Memory>)` to cache fitted transformers keyed on params + input data; the search then fits each transformer once and reuses it across configurations. Caveats that bite: the **last step is never cached**, and with `memory=` set transformers are **cloned before fitting** so the original instance can't be inspected (use `named_steps`). Note the same `Pipeline` structure is also the correctness fix for fitting a scaler/imputer *inside* the CV split rather than on the full dataset (data leakage) — here the framing is cost, but the two motivations coincide (verify against the currency brief for your version).

- **A pipeline step that densifies a sparse matrix mid-flow**: `OneHotEncoder` returns a **sparse (CSR) matrix by default** (`sparse_output=True`), and many estimators (linear models, SGD, tree ensembles, `HistGradientBoosting*`) accept CSR natively — keeping high-cardinality categoricals sparse can be the difference between a few MB and tens of GB. The trap is one intermediate step that forces `.toarray()`: `StandardScaler(with_mean=True)` on sparse input raises/densifies (use `with_mean=False`), a `ColumnTransformer` falls back to dense when its `sparse_threshold` is crossed, or a PCA/estimator that only takes dense input. One such step materializes the full dense block and dominates memory. Trace the matrix's sparsity end-to-end and locate where it densifies (cross-reference the **Memory** lane in `../python.md`).

- **Paying for `float64` when `float32` would do**: scikit-learn's `check_array` with the default `dtype='numeric'` *preserves* an incoming `float32` array (it does not force `float64`), and many estimators declare `dtype=[np.float64, np.float32]` so they keep 32-bit input — passing `float32` then halves the feature-matrix footprint and often speeds the BLAS kernels. But some estimators force `float64` (their `check_X_y` lists only `np.float64`), silently doubling memory via a conversion copy; and passing a pandas `DataFrame` triggers a `DataFrame`→ndarray conversion (and, with `object`/extension dtypes, possibly an `object`-array detour) on every `fit`/`predict`. Verify the estimator preserves 32-bit before assuming the saving, and feed a contiguous ndarray of the right dtype rather than re-converting a DataFrame each call (verify against the currency brief for your version).

- **Old `GradientBoosting*` where `HistGradientBoosting*` belongs**: the histogram estimators (`HistGradientBoostingClassifier`/`Regressor`) bin features into integer bins (typically 256), cutting split-finding from sorting continuous values (~`O(n_features · n log n)`) to `O(n_features · n)`, and are **orders of magnitude faster than the exact `GradientBoosting*` once n_samples exceeds tens of thousands**. They are OpenMP-parallel and **natively handle NaN and categorical features** (`categorical_features=`), removing imputation and one-hot preprocessing entirely — directly relevant to gappy SCADA/smart-meter sensor features. Below a few thousand rows binning is approximate and the exact version may be preferable, so this is a large-N recommendation, not universal (verify against the currency brief for your version).

- **Per-row `.predict()` in a Python loop instead of one batched call**: calling `model.predict(x_row)` once per sample inside a `for` loop pays the Python↔C marshaling cost per row and re-validates input each time; `model.predict(X_all)` over the full matrix crosses the boundary once and runs the vectorized C/OpenMP kernel. For utility-scale scoring (every meter, every interval) this is often a 10–100× latency gap. This is the model-scoring instance of the Python↔C-boundary point in `../python.md` Runtime notes — the fix is one bulk call, not faster looping.

- **Exhaustive search where halving or a randomized search fits the budget**: `GridSearchCV` cost is the full Cartesian product of the grid × `n_folds` × per-fit cost; `RandomizedSearchCV` caps it at `n_iter` fits, and `HalvingGridSearchCV`/`HalvingRandomSearchCV` (successive halving) run all candidates on a small resource first and promote only survivors to larger resources — "much faster at finding a good combination" per the docs. Two further levers: `pre_dispatch` bounds how many jobs (and thus how many copies of the data) are queued at once, guarding memory when `n_jobs` is large; and `refit=True` adds one final full-data fit after the search — fine normally, but redundant cost if you only wanted the CV scores. Match the search strategy to the fit budget rather than defaulting to exhaustive grid (verify against the currency brief for your version).

- **Out-of-core / incremental training ignored when data exceeds RAM**: when the training matrix doesn't fit in memory, estimators exposing `partial_fit` (`SGDClassifier`/`SGDRegressor`, `MiniBatchKMeans`, `MultinomialNB`) stream mini-batches without ever materializing the full array, and `HistGradientBoosting*` is itself memory-frugal via binning. Reaching for a full in-memory fit (or up-sizing the box) when a streaming `partial_fit` loop over chunks would bound resident memory is the avoidable cost here (cross-reference the **Memory** lane in `../python.md` on bounding peak resident size).

- **Rebuilding the gradient-boosting data structure every round/fold instead of once**: XGBoost binning into a `DMatrix`/`QuantileDMatrix`, LightGBM into a `Dataset`, CatBoost into a `Pool` is a real, often-dominant cost; reconstructing it inside a CV loop or per boosting call repeats the quantization needlessly. Build it **once** and reuse it. With `tree_method='hist'`, `QuantileDMatrix` quantizes directly from the source and **avoids the intermediate full-precision DMatrix copy**, cutting peak memory (pass a `ref` for validation sets to share the bin edges). LightGBM's `Dataset` defaults to `free_raw_data=True`, discarding the raw data after binning — so a later operation needing the raw data forces a rebuild. Validation `Dataset`s must be built with `reference=train_set` to align bins (verify against the currency brief for your version).

- **`tree_method='hist'` (and GPU) vs the exact method on large data**: the histogram method is the fast path for large N and is the modern default surface for all three libraries; the exact greedy method scales far worse. For large matrices, GPU training (`device='cuda'` in recent XGBoost; LightGBM `device='gpu'`; CatBoost `task_type='GPU'`) is a step-change, not a constant factor. Also: native categorical support (`enable_categorical=True` in XGBoost — still flagged experimental; LightGBM `categorical_feature=`; CatBoost `cat_features=`) avoids the dimensionality blow-up of one-hot encoding on high-cardinality features (substation IDs, asset tags). Confirm the exact `tree_method`/`device` default for your version rather than assuming `hist` is implicit (verify against the currency brief for your version).

- **Boosting without early stopping, and pickling boosters instead of native save**: omitting `early_stopping_rounds` (with an eval set) runs the full `n_estimators` even after validation loss plateaus — pure wasted iterations; with it, stop and predict using `best_iteration` (LightGBM: `save_model(..., num_iteration=bst.best_iteration)`). For inference, XGBoost's `inplace_predict` skips the DMatrix construction, predicts in-place, and is **lock-free / thread-safe** for concurrent scoring. Persist with the **native `save_model`** (XGBoost JSON/UBJSON, LightGBM text) not `pickle`: pickle captures the full runtime object, "is not a stable serialization format and doesn't work [across] different Python version nor XGBoost version" — it's a checkpoint format, not an archival one. The native format is portable and faster to load (cross-reference the **payload-startup** lane in `../python.md` for model-load-at-startup cost) (verify against the currency brief for your version).

- **Re-estimating an ARIMA/SARIMAX from scratch at every rolling-backtest step**: a walk-forward / expanding-window load-forecast evaluation that calls `.fit()` afresh on each step re-runs the full MLE optimization every iteration — `O(n_steps)` expensive fits. statsmodels state-space results expose **`.append(new_obs, refit=False)`** (filter the longer series forward with the *existing* parameters), **`.extend(new_obs)`** (filter only the new data — much faster when the original series is large), and **`.apply(new_data)`** (reuse params on a different series). Use these to fix parameters once and merely filter forward; refit periodically, not every step. Separately, SARIMAX/state-space fit and filter cost scales with the **state dimension**, which a large seasonal order inflates — a heavy seasonal spec is a quiet cost multiplier; `enforce_stationarity=False`/`concentrate_scale=True` can ease the optimizer (verify against the currency brief for your version).

- **Prophet full-MCMC sampling, or fitting thousands of series in a serial loop**: Prophet defaults to **MAP estimation (`mcmc_samples=0`), seconds per series**; setting `mcmc_samples>0` switches to full Bayesian MCMC and the docs warn to "expect several minutes instead of several seconds" — only pay it when you genuinely need seasonality uncertainty intervals. The bigger utility-scale cost is forecasting many series (per-feeder, per-meter) in a plain Python `for` loop: these fits are independent, so parallelize across series (process pool / joblib) — but watch the same over-subscription trap if each backend (Prophet's Stan, or statsmodels' BLAS) is itself threaded (cross-reference the **Concurrency** lane in `../python.md`) (verify against the currency brief for your version).

- **Cox PH fit and concordance cost on large maintenance/failure datasets**: lifelines `CoxPHFitter` handles ties with **Efron's method** (more accurate, costlier than Breslow per tied block) and exposes a **`batch_mode`** that "can be faster for datasets with a large number of ties" (it picks automatically if left `None`) — relevant when many assets fail/are censored at the same coarse timestamp. Baseline-hazard estimation and `predict_survival_function`/`predict_cumulative_hazard` over many rows is its own cost; predict on the full matrix in one call, not per asset. Crucially, **concordance-index evaluation is naively `O(n²)` over comparable pairs** — on a large held-out asset population (smart-meter / fleet scale) the metric can cost more than the fit; subsample the evaluation set or use an implementation with the `O(n log n)` rank-tree algorithm. scikit-survival's tree/forest survival estimators likewise scale poorly in N — confirm the metric's complexity rather than assuming it's cheap (verify against the currency brief for your version).

- **Fitting a reliability distribution per asset in a Python loop**: `scipy.stats.<dist>.fit(data)` (Weibull `weibull_min`, `lognorm`, etc.) runs a **numerical MLE optimization per call** — one optimizer run per invocation. Looping `.fit()` over thousands of assets/components serially is `n_assets` independent optimizations dispatched one Python call at a time; parallelize across assets (process pool) and **fix known parameters** (`floc=0` to pin the location, which both stabilizes the optimizer and cuts its dimensionality) to speed and de-flake each fit. This is the survival/reliability instance of "independent work in a serial loop" — cross-reference the scientific-computing module if present, and the **Concurrency** lane in `../python.md` (verify against the currency brief for your version).
