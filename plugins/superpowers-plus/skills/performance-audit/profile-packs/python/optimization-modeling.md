## Optimization modeling (Pyomo / OR-Tools / CVXPY / Gurobi / CPLEX)

> Scope: the *modeling-layer* cost of mathematical-programming code (LP/MILP/QP/conic/CP) built through Pyomo, Google OR-Tools (CP-SAT and the linear solver), CVXPY, and the Gurobi/CPLEX Python bindings — not solution quality, and not the solver's internal numerics. The recurring themes: **building the model in Python routinely costs as much as or more than the actual solve** (every `Constraint`/`Var` is a Python object created by a Python loop, and the model must then be translated into the solver's representation); **rebuilding the whole model for every solve when you could modify-and-resolve is the single largest avoidable cost** — power-system studies (OPF, unit commitment, capacity-expansion/IRP, DER dispatch) re-solve the *same model structure with changed data* across a rolling horizon, a scenario fan, or a Monte-Carlo sweep, so model reuse, warm starts, and the persistent/parametrized interfaces are the dominant wins; and **the per-element Python round-trip taxes both directions** — scalar constraint construction in a `for` loop on the way in, and element-by-element solution read-back on the way out. The actual LP/MILP/conic solver kernel that runs underneath is C/C++ and not your bottleneck once the model stops being rebuilt; for the SciPy-level solvers (`scipy.optimize`, `linprog`/`milp`, sparse factorizations, warm-starting `x0`) one layer down, see the sibling [`scientific-computing.md`](scientific-computing.md). Cross-reference the **Algorithmic complexity** lane in `../python.md` for the loop footgun and the **Runtime & interpreter notes** on the Python↔C boundary — a model build is millions of tiny crossings, which is exactly why it is slow.

- **Rebuilding the model object from scratch for every solve in a scenario/rolling-horizon/Monte-Carlo sweep**: the foundational anti-pattern across all four ecosystems. When only the data changes (loads, renewable availability, prices, a horizon shift) but the constraint *structure* is fixed, re-instantiating the entire model re-pays the full Python-object-construction and solver-translation cost on every iteration — often dominating a study that runs hundreds of solves. The fix is ecosystem-specific (persistent solver, DPP `Parameter`, coefficient modification — below), but the *condition to recognize* is universal: a model-construction call inside the sweep loop. Hoist construction out; mutate-and-resolve inside.

- **Pyomo `ConcreteModel` assembled by Python `for` loops / rule functions creating thousands of per-element `Constraint`/`Var` objects**: each indexed component instantiates one Python object per index, and a `Constraint`-rule function is *called once per index* — for a unit-commitment model over thousands of (generator × time) pairs this is tens of thousands of Python calls before the solver sees anything. The build can rival the solve. Use Pyomo's indexed components with set-based rules rather than hand-rolled nested loops, keep rule bodies cheap (no per-call DataFrame lookups — pre-stage data into dicts/params), and profile the construct phase separately from the solve with `cProfile` (cross-reference **Algorithmic complexity** in `../python.md`) (verify against the currency brief for your version).

- **Pyomo file-based (non-persistent) solver interface re-writing an LP/NL/MPS file on every solve**: the default `SolverFactory('gurobi')` / `SolverFactory('cplex')` path writes the model to a temporary LP or NL file and launches the solver against it each call — so a resolve pays full model-write + file-I/O + solver-process-startup again, even if nothing structural changed. The persistent interfaces (`SolverFactory('gurobi_persistent')`, `SolverFactory('cplex_persistent')`, or the APPSI solvers `appsi_gurobi`/`appsi_highs`) instead "keep a reference to a gurobipy model between calls to solve()" and exist "to efficiently notify the solver of incremental changes to a Pyomo model" — no file round-trip, no re-translation of unchanged components. For any repeated-solve power-system study this is usually the largest single win (verify against the currency brief for your version).

- **Pyomo persistent solver used but the model mutated without notifying it**: persistent solvers are *manual* by default — "Users are responsible for notifying persistent solver interfaces when changes to a model are made," via `set_instance`, `add_constraint`/`remove_constraint`, `add_var`/`remove_var`, `set_objective`, and `update_var` (the documented exception: a `Var` may be modified directly if followed by `update_var`). Forgetting leaves stale components in the solver and a wrong solve; the trade-off is that this manual surface is what lets you change a few RHS values or coefficients and resolve without re-sending the whole model. The APPSI auto-persistent layer detects changes for you at the cost of a diff pass — choose manual for surgical edits on a hot loop, auto when the change set is broad or hard to track (verify against the currency brief for your version).

- **Pyomo `AbstractModel` rebuilt per scenario instead of `create_instance(data)` reused, and duals/warm-starts left on the table**: an `AbstractModel` is declared once as empty containers; `create_instance(data)` "constructs each of the model components in the order in which they were declared" out-of-place into a `ConcreteModel`. That construction is still Python-object work, so for a scenario sweep prefer the persistent-solver modify-and-resolve path over a fresh `create_instance` per scenario; `AbstractModel`+data is mainly an authoring/data-separation convenience, not a per-solve speedup. Separately, to retrieve duals/reduced-costs (LMPs, congestion prices) declare a `Suffix(direction=Suffix.IMPORT)` named `dual` (also `rc`, `slack`) — without it the information is discarded and a second solve to recover it is pure waste (verify against the currency brief for your version).

- **CVXPY `Problem.solve()` re-canonicalizing on every call because the problem is rebuilt or not DPP-parametrized**: canonicalization (lowering the high-level problem to solver-standard conic form) is the expensive CVXPY-side step, and it runs on *every* `solve()` unless the problem is DPP-compliant and the only thing changing is `Parameter` values. Per the DPP docs, "the first time a DPP-compliant problem is solved, CVXPY compiles it and caches the mapping from parameters to problem data," so "subsequent rewritings of DPP problems can be substantially faster." This is the single biggest CVXPY lever for repeated solves (rolling DER dispatch, scenario OPF): build the `Problem` once with data wrapped in `Parameter` objects, then assign `param.value = ...` and re-`solve()` in the loop — never reconstruct `Problem` inside the loop, and keep expressions DPP-compliant (a `Parameter`×`Parameter` or other DPP-violating product silently forces re-canonicalization) (verify against the currency brief for your version).

- **CVXPY `warm_start` benefit misjudged for the problem structure**: CVXPY warm-starts by default and "many solvers can exploit work from previous solves," reusing "the previous solution as an initial point or reuse[d] cached matrix factorizations" — the docs show a least-squares resolve dropping from ~11s to ~3s via "caching the KKT matrix factorization." But the critical caveat: "If `A` were a parameter, factorization caching would not be possible and the benefit of warm start would only be a good initial point." So whether a coefficient lives in a `Parameter` (changes across solves → no factorization reuse) versus stays constant (reuse possible) materially changes the payoff — don't assume warm start buys factorization reuse when the matrix itself is parametrized. Solver choice (CLARABEL/OSQP/SCS open-source defaults vs commercial GUROBI/MOSEK) is a separate lever to tune by measurement, not assumption (verify against the currency brief for your version).

- **OR-Tools CP-SAT solving with default workers, no hint, on a model rebuilt each time**: build the `CpModel` once and re-solve with a fresh `CpSolver` rather than rebuilding the model. `solver.parameters.num_search_workers` controls parallel portfolio search — CP-SAT runs a portfolio of differing strategies across workers, so this is a genuine lever (set it explicitly and measure rather than relying on the default core-count heuristic). `model.add_hint(var, value)` (alias `AddHint`) seeds a starting assignment that "does NOT need to satisfy all constraints" and "can decrease solving times substantially, especially when solving a similar model repeatedly" — exactly the repeated-similar-model case of a rolling commitment/dispatch study. Build linear constraints from arrays via `LinearExpr` rather than summing in a Python loop, and set `max_time_in_seconds` as a guard (verify against the currency brief for your version).

- **Gurobi/CPLEX model rebuilt instead of modifying coefficients on the existing model**: with the native bindings, mutate the live model — `Model.chgCoeff(constr, var, newval)` changes one matrix coefficient, and `constr.setAttr('rhs', ...)` / `var.setAttr('ub', ...)` change RHS/bounds — rather than tearing down and re-adding `Var`/`Constr` objects. For a rolling-horizon OPF where only loads and limits move between solves, this is the Gurobi/CPLEX analog of CVXPY `Parameter`s and the dominant reuse win. The trade-off is bookkeeping: you must hold references to the `Var`/`Constr` objects you intend to edit (CPLEX's `docplex` exposes the same modify-in-place surface) (verify against the currency brief for your version).

- **Gurobi lazy-update semantics misunderstood — reading attributes before the update is applied**: Gurobi modifications are queued, not immediate; queued changes apply only on an explicit `Model.update()`, on `Model.optimize()`, or on `Model.write()`. Adding a `Var`/`Constr` and then querying its attributes before any of those returns stale/empty state, and conversely calling `Model.update()` after every single edit in a build loop forces repeated flushes that defeat the lazy batching — make all edits, then update once. This is a frequent source of "my change didn't take" bugs *and* of accidental per-edit overhead; understand which of the three triggers your code is relying on (verify against the currency brief for your version).

- **Gurobi/CPLEX warm starts not supplied across a resolve sequence**: for MILP, set `Var.Start` (the MIP start vector the solver "will attempt to build an initial solution from"); for LP, supply a simplex basis via `VBasis`/`CBasis` (faster in Gurobi's own benchmarks) or a primal/dual point via `PStart`/`DStart`. Carrying the previous solution forward across successive OPF/commitment solves as load shifts can cut iteration count sharply — but note `PStart` requires `DStart` (and `VBasis` requires `CBasis`) to actually warm-start an LP; a half-specified start is ignored. This is the modeling-layer twin of the `x0` carry-forward in [`scientific-computing.md`](scientific-computing.md) (verify against the currency brief for your version).

- **Re-creating the Gurobi `Env` per model instead of reusing one across a sweep**: a Gurobi `Env` carries license acquisition and setup cost; the docs advise creating a single environment and reusing it across models because "reusing one environment is much more efficient than creating and destroying multiple environments." Building a fresh `Env` per solve in a scenario loop re-pays that setup every iteration. The one documented exception is multithreading — environments are not thread-safe, so a per-thread `Env` is required there (cross-reference the **Concurrency** lane and GIL notes in `../python.md`: solver C kernels run outside the GIL, but the Python binding objects are not thread-safe) (verify against the currency brief for your version).

- **Per-node Python callbacks (lazy constraints, cuts, branch hints) adding interpreter overhead inside the solve**: Gurobi/CPLEX/CP-SAT callbacks invoke *your Python code* at many nodes/incumbents during the solve — a heavy callback (DataFrame lookups, fresh allocations, recomputing invariants) multiplies Python cost across the whole search tree and can erase the gain the callback was meant to provide. Keep callback bodies minimal, pre-stage any data they read, and prefer formulating constraints into the model up front when the full set is tractable; reserve callbacks for genuinely exponential families (e.g. subtour/security constraints) where adding them lazily is the only feasible route (cross-reference **Algorithmic complexity** in `../python.md`) (verify against the currency brief for your version).

- **Reading the solution back element-by-element, and scalar constraint construction over vectorized/matrix form**: after a solve, looping in Python over thousands of variables to pull `var.value` / `var.X` / `solver.value(var)` one at a time re-crosses the Python↔C boundary per element (cross-reference the **Runtime & interpreter notes** in `../python.md`); pull values in bulk where the binding offers it (e.g. Gurobi `Model.getAttr('X', vars)` over a list). The same boundary tax applies on the build side — constructing constraints in matrix/vectorized form (CVXPY matrix expressions, OR-Tools `LinearExpr` over coefficient arrays, Gurobi matrix-friendly APIs) issues far fewer Python-level operations than a per-scalar `for` loop assembling each row term by term (verify against the currency brief for your version).
