## Scientific computing (SciPy / Numba / Cython)

> Scope: SciPy's sparse, linear-algebra, optimization, signal/FFT, ODE, statistics, and spatial subpackages, plus the two ahead-of-time/JIT escape hatches (Numba, Cython) for the loops that vectorization cannot reach. The recurring themes are: **the right data structure for the operation owns most of the cost** (a sparse format chosen for construction is pathological for arithmetic; a transfer-function filter is unstable where a cascaded one is fine); **factor/build/plan once, reuse many times** (an LU factorization, a KDTree, a filter design, an FFT length, a compiled kernel — the expensive step amortizes only if it is hoisted out of the loop); and **the per-call work you didn't supply gets recomputed numerically** (a missing analytic Jacobian turns one objective call per step into many finite-difference probes). Cross-reference the **Algorithmic complexity** lane and the BLAS-linkage bullet in `../python.md`, the **Memory & allocation** lane for densification/copy traps, and the Runtime & interpreter notes on the GIL and the Python↔C boundary — Numba and Cython exist precisely to cross that boundary in bulk.

- **A sparse format chosen for construction used for arithmetic (or vice versa)**: building a matrix by repeated assignment into a `csr_array`/`csc_array` is pathological — the CSR docs list "changes to the sparsity structure are expensive (consider LIL or DOK)" as an explicit disadvantage, because each new nonzero shifts the packed `indices`/`data` arrays. Build incrementally in `lil_array`/`dok_array`/`coo_array`, then convert once to CSR/CSC for computation ("To perform manipulations such as multiplication or inversion, first convert to CSC or CSR"). Conversions among CSR/CSC/COO are linear-time, so a single conversion is cheap; what is expensive is converting back and forth inside a loop. In power-flow / state-estimation code, assemble the admittance or Jacobian matrix in COO (triplet) form and convert to CSC once before solving (verify against the currency brief for your version).

- **Sparse system re-solved from scratch when the matrix is reused**: calling `scipy.sparse.linalg.spsolve(A, b)` in a loop refactorizes `A` every iteration. `splu(A)` returns a `SuperLU` object whose `.solve()` method reuses one factorization across many right-hand sides — the dominant win when the same matrix is solved against changing loads (Newton power-flow iterations reusing a Jacobian, repeated state-estimation solves, contingency screening). `splu` is "most efficient when provided in CSC format; other formats will be converted to CSC before factorization," so feed it CSC to avoid a hidden conversion. For very large symmetric-positive-definite systems, an iterative solver (`cg`/`gmres`) with a preconditioner can beat a direct factorization that fills in heavily — but only with a preconditioner; unpreconditioned iterative solvers on ill-conditioned grids stall (verify against the currency brief for your version).

- **Sparse-times-dense silently densifying**: mixing a sparse matrix with a dense NumPy array, or calling `.todense()`/`np.asarray()` "just to check," materializes the full dense matrix — for a grid admittance matrix that is mostly zeros this can be the single largest allocation in the program and defeats the entire point of sparsity. Keep operands sparse, use the sparse matvec (`A @ x` stays sparse-aware), and treat any `.toarray()`/`.todense()` on a hot path as a red flag. Cross-reference the **Memory & allocation** lane in `../python.md`. Separately, the package is migrating from the `*_matrix` (matrix-semantics) interface to the `*_array` (NumPy-semantics) interface; the docs recommend `*_array` for new code, and `@`/`*` mean different things across the two — mixing them is a correctness-and-performance hazard (verify against the currency brief for your version).

- **`inv()`-then-multiply instead of a solve, and `numpy.linalg` where `scipy.linalg` has more knobs**: computing `inv(A) @ b` forms the full inverse (more flops, worse conditioning) where `scipy.linalg.solve(A, b)` does not; for repeated solves against one `A`, `lu_factor` + `lu_solve` factors once. `scipy.linalg.solve` also exposes `assume_a=` to dispatch a structure-aware solver ('pos' → Cholesky, 'sym'/'her', triangular, banded, tridiagonal) — a symmetric-positive-definite solve via Cholesky is roughly half the flops of a general LU. On trusted hot paths, `overwrite_a=True`/`overwrite_b=True` ("may enhance performance" by skipping the input copy) and `check_finite=False` ("disabling may give a performance gain, but may result in crashes/non-termination if inputs contain NaN/inf") trade a safety check for speed — only after the inputs are known clean (verify against the currency brief for your version).

- **Finite-difference Jacobians dominating an optimization or fit**: `scipy.optimize.minimize`/`least_squares`/`curve_fit` estimate the gradient by "2-point finite difference estimation" when `jac` is not supplied — each gradient costs O(n) extra objective evaluations per iteration, which is usually the bottleneck. Supplying an analytic (or autodiff) `jac` collapses that to one call per step; for Newton-type methods (`Newton-CG`, `trust-ncg`, `trust-exact`, `trust-krylov`) supplying `hess`/`hessp` matters too. For a large nonlinear least-squares fit (e.g. fitting many parameters to PMU/measurement residuals), pass `jac_sparsity` to `least_squares`: "if the Jacobian has only few non-zero elements in each row, providing the sparsity structure will greatly speed up the computations" by skipping known-zero finite-difference probes (it forces the 'trf'/'lsmr' path) (verify against the currency brief for your version).

- **Derivative-free or wrong-class method on a problem that has structure**: choosing `Nelder-Mead`/`Powell` (no derivatives) for a smooth differentiable objective wastes the gradient information that would converge in far fewer iterations; conversely, gradient methods on a noisy/discontinuous objective thrash. Match the method to the problem class, and for LP/MILP use `linprog`/`milp` rather than a general nonlinear solver. Warm-starting from the previous solution (carrying `x0` forward across a sequence of similar solves — successive OPF or unit-commitment runs as load shifts) cuts iteration count substantially. A per-element Python objective is its own tax — vectorize the objective and residual evaluation over the parameter/data arrays so each solver step is one C-level call, not a Python loop (cross-reference the **Algorithmic complexity** lane in `../python.md`) (verify against the currency brief for your version).

- **Direct convolution where FFT convolution is asymptotically cheaper**: `scipy.signal.convolve` direct mode is O(n·m); `fftconvolve` is "generally much faster than convolve for large arrays (n > ~500), but can be slower when only a few output values are needed." When the two inputs differ greatly in length (a long PMU/waveform stream against a short kernel), `oaconvolve` (overlap-add) "is generally faster when the input arrays are large and significantly different in size" because it avoids transforming the whole long signal at once. Note both cast integer input to float. The crossover is size-dependent — for short kernels on short signals, direct convolution still wins (verify against the currency brief for your version).

- **High-order IIR filtering through transfer-function `(b, a)` coefficients**: `lfilter`/`filtfilt` with `b,a` of a high-order filter is numerically fragile — the SOS docs note that for a 13th-order design "the numerical error pushes some poles outside of the unit circle," producing an unstable/wrong result, and `sosfilt`/`sosfiltfilt` (second-order sections, "designed to minimize numerical precision errors for high-order filters") is both more stable and well-conditioned. Design with `output='sos'`. Independently, the filter design call belongs *outside* the loop — designing the same filter once per sample or per window is pure waste; design once, filter many (verify against the currency brief for your version).

- **FFT on the slow module, the wrong transform, or an awkward length**: `scipy.fftpack` is documented legacy ("considered legacy... new code should use scipy.fft"); `scipy.fft` is the maintained pocketfft-backed module and exposes a `workers=` parameter ("maximum number of workers to use for parallel computation; if negative, wraps around from os.cpu_count()") to parallelize independent transforms along non-transformed axes. For real-valued input use `rfft`/`rfftn`, which compute roughly half the spectrum and cost — using full complex `fft` on real PMU/waveform data wastes half the work. And pad to a smooth length via `next_fast_len`: SciPy's FFT "gain[s] speed by a recursive divide-and-conquer strategy [relying] on efficient functions for small prime factors," so a prime-length transform is far slower (the doc shows a 7.3x speedup from padding 93,059 → 93,312) (verify against the currency brief for your version).

- **Non-stiff ODE method on a stiff system, or a missing ODE Jacobian**: `solve_ivp` defaults to `RK45` (explicit); the docs state "explicit Runge-Kutta methods should be used for non-stiff problems and implicit methods for stiff problems" — running `RK45` on a stiff system (fast electromechanical transients alongside slow dynamics) drives the step size to near-zero and stalls. Switch to `BDF`/`Radau`/`LSODA` (the last auto-detects stiffness). For those implicit methods, supplying `jac` avoids a finite-differenced Jacobian each step, and `vectorized=True` "results in faster finite difference approximation of the Jacobian by methods 'Radau' and 'BDF'" (but can slow explicit methods — set it only for implicit ones). `dense_output=True` stores interpolants for every step; skip it when you only need the endpoint (verify against the currency brief for your version).

- **Per-element distribution calls and re-fitting where vectorize/freeze would do**: `scipy.stats` `pdf`/`cdf`/`logpdf` are vectorized — calling them in a Python loop over samples reincurs the Python↔C crossing per element (cross-reference the Runtime & interpreter notes in `../python.md`); pass the whole array once. `.fit()` does MLE by "minimizing the negative log-likelihood function" via numerical optimization, which is expensive on large reliability samples (Weibull/lognormal failure fitting) — fix known parameters with `floc`/`fscale`/`f0…` and supply good starting guesses to cut iterations, since each removed free parameter shrinks the search. When the same parametrized distribution is reused, *freeze* it (`dist(*params)`) once so loc/scale validation isn't redone on every call (verify against the currency brief for your version).

- **Brute-force pairwise distances where a spatial index amortizes**: repeated nearest-neighbor or radius queries computed by full pairwise distance are O(n²) per query; build a `KDTree` (`cKDTree` is now "functionally identical to KDTree", a backward-compat alias since SciPy 1.6) once and query it many times. The build cost only pays off when queries are repeated or the point set is reused — for a single one-shot query on a small set, the tree build can exceed a direct computation. Same shape for `scipy.interpolate`: construct the interpolator object once outside the loop and call it on batched query points, rather than re-fitting per evaluation (verify against the currency brief for your version).

- **Numba `@jit` silently running in object mode, or jitting the wrong thing**: when nopython compilation fails, `@jit` can fall back to object mode (`forceobj`), which runs through the interpreter with added Numba overhead — i.e. *slower* than plain Python, with no error. Use `@njit` (equivalent to `@jit(nopython=True)`) so an unsupported construct *raises* instead of silently degrading. Numba "works best on code that uses NumPy arrays and functions, and loops"; it does not understand pandas (`"Pandas is not understood by Numba"`) or arbitrary Python objects, so wrapping a pandas-heavy function gains nothing. Compilation is a one-time cost paid on first call — `cache=True` writes the compiled result to disk to avoid recompiling across process restarts (caveat: it treats globals as compile-time constants and does not detect changes in imported modules) — and trivial functions are not worth jitting at all (verify against the currency brief for your version).

- **Numba `parallel=True`/`prange` oversubscribing cores against BLAS/other pools**: `parallel=True` with `prange` runs loops on native threads with no GIL, but Numba spawns its own thread pool sized to the core count — stacked on top of BLAS threads (NumPy/SciPy linear algebra) or multiple worker processes, total threads exceed cores and thrash (the threading-layer docs show 4 processes × 8 threads = 32 on 8 cores as oversubscription). Bound it with `NUMBA_NUM_THREADS`/`set_num_threads()` and pin BLAS thread counts the same way (cross-reference the BLAS-oversubscription guidance in the **Data stack** sibling and Runtime notes in `../python.md`). Note `parallel=True` only helps loops that are actually parallelizable (`prange` reductions, element-wise); on a sequential-dependency loop it adds overhead. `fastmath=True` relaxes IEEE rigor (reassociation, no-NaN assumptions) for speed — safe only when the data cannot produce NaN/inf and bitwise reproducibility is not required (verify against the currency brief for your version).

- **Cython numeric loops leaving the safety directives on, or crossing into Python objects inside the kernel**: untyped Cython is just Python with overhead — the speed comes from `cdef`-typed scalars and **typed memoryviews** (`cdef double[:, :] a`) that access array buffers at C speed. The directives `boundscheck` (default True), `wraparound` (default True), and `cdivision` (default False, "up to a 35% speed penalty" because it adds Python-semantics modulo/sign handling and a ZeroDivisionError check) each guard correctness at a per-access cost; disable them on a verified-correct inner loop (out-of-bounds then crashes or corrupts data, so disable only once indices are provably in range). The largest hidden cost is touching a Python object (a list, a dict, an untyped variable, `numpy` attribute access) inside the hot loop — every such access re-acquires the GIL and crosses the boundary, so a `with nogil:` block must be pure C. Reach for Cython over Numba when you need fine-grained C-struct/pointer control, to call into C/C++ libraries, or ahead-of-time compilation without a JIT warmup; reach for Numba when a NumPy-centric kernel just needs to be fast without a build step (verify against the currency brief for your version).
