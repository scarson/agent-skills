# R performance module: data.table
> Load when `data.table`/`fread`/`:=`/`setDT` is detected — see the module map in `../r.md`. Core lanes + Runtime & copy-on-modify notes live in `../r.md`; this file is the data.table lens only.

## data.table

> Scope: data.table is, at bottom, an engine for *escaping R's copy-on-modify semantics* on large tables — meter/AMI/SCADA frames of tens of millions of rows where base-R idioms silently deep-copy the whole column (or whole table) on every edit. The recurring themes: **modify in place** (`:=`/`set()` mutate by reference; `<-` and base subassignment copy — see the **Runtime & copy-on-modify notes** in `../r.md`, of which this module is the direct mitigation), **index, don't scan** (`setkey`/`setindex` turn O(n) vector scans into O(log n) binary search for subsets and joins), **let the C engine do the group-work** (GForce replaces per-group R dispatch with internal `g*` implementations — but only for a closed set of functions, and trivially defeated), and **read/reshape less, in C** (`fread`/`fwrite`/`melt`/`dcast` are multi-threaded C and pruning-aware). The danger is that data.table *looks* like base R, so a copying base-R idiom hides in plain sight inside a fast-looking pipeline. Cross-reference the `algorithmic`, `memory`, `data-access`, and `concurrency` lanes in `../r.md`.

- **Base-R column assignment inside a data.table pipeline (the copy that hides in plain sight)**: `DT$col <- x`, `DT[["col"]] <- x`, or `DT$col[idx] <- v` fall back to R's copy-on-modify and *deep-copy the entire column* (pre-R-3.1 the whole table) on every assignment — exactly the cost data.table exists to avoid (see **Runtime & copy-on-modify notes** in `../r.md`). The fix is `:=` (`DT[, col := x]`) or `set()`, which update by reference with **no copy at all**. This is the single highest-value finding in R/data.table code: the syntax is one character different, the cost is O(n) memory traffic per edit vs zero. On a 40M-row meter table built one derived column at a time, base assignment is the difference between minutes-with-GC-thrash and milliseconds. Audit every `$<-`, `[[<-`, and `[<-` against a `data.table` object (verify against the currency brief for your version).

- **`:=` in a tight `for` loop pays `[.data.table` dispatch per iteration**: `:=` lives inside `[.data.table`, so calling it once per row in a loop re-incurs the (non-trivial) `[` method-dispatch and argument-parsing overhead every iteration — the data is mutated cheaply, but the *call* is not free. For genuinely row-wise updates that can't be vectorized, `set(DT, i, j, value)` bypasses `[` dispatch entirely; the docs benchmark this as thousands of times faster than the equivalent `[`-based loop. First ask whether the loop is needed at all (a vectorized `:=` over the whole column, or a grouped `:=` with `by=`, usually replaces it); `set()` is the escape hatch only when per-row logic is irreducible (verify against the currency brief for your version).

- **`fread`/`fwrite` over `read.csv`/`write.csv` for large flat files**: `fread` is multi-threaded C with memory-mapped lazy reads and sampled type inference — order(s) of magnitude faster than `read.csv` (the docs show ~21x on a 500 MiB file, larger gaps at scale). Beyond raw speed, push the *pruning* args so you parse less: `select=`/`drop=` to read only the columns used downstream, `colClasses=` to skip type inference (and avoid a wrong guess silently widening to character), `nrows=` for sampling, `nThread=` to bound or exploit cores. A reader that ingests all 60 SCADA columns then drops 50 is doing the work `select=` would have skipped. Cross-reference the `data-access` lane in `../r.md` (verify against the currency brief for your version).

- **Repeated `==`/`%in%` subsetting on an unkeyed, unindexed column (O(n) scan every time)**: `DT[col == val]` without a key or index does a full vector scan and materializes a logical mask of length n — O(n) per call, repeated for every distinct lookup. `setkey(DT, col)` physically reorders the table by reference so subsets and joins use binary search (O(log n)); `setindex(DT, col)` builds a secondary index *without* reordering (stores only an order vector), allowing multiple indices and preserving row order. Auto-indexing builds an index on first `==`/`%in%` use automatically — but only for those two operators, and the first call still pays the build. For a substation lookup hit thousands of times across a session, the keyed/indexed form is the seconds-vs-instant difference; for a one-shot filter the scan is fine. Cross-reference the `data-access` lane in `../r.md` (verify against the currency brief for your version).

- **GForce defeated by a wrapped or arithmetic-combined aggregate in `j`**: under `by`/`keyby`, data.table swaps a closed set of reducers — `min, max, mean, median, var, sd, sum, prod, first, last, head, tail` and `.N` — for internal `g*` implementations (GForce) that aggregate per group in C with no per-group R call. The optimization is **fragile**: wrapping the function (`function(x) mean(x)`, `as.numeric(median(x))`), namespacing (`stats::median`), or combining within one expression (`max(x) - min(x)`) drops back to invoking R once per group — catastrophic at tens of thousands of groups (one meter-ID per group). Audit `j` expressions under `by` for anything that isn't a bare GForce call; split `max(x)-min(x)` into two GForce columns and subtract after. Verify the current optimized-function set and GForce level against the currency brief for your version.

- **`.SD` without `.SDcols` materializes every non-group column per group**: when `j` references `.SD` (e.g. `lapply(.SD, fun)`), data.table builds a sub-table of *all* non-grouping columns for each group unless `.SDcols` restricts it. On a wide AMI frame grouped by meter, that means copying dozens of unused columns once per group — pure waste. Always scope `.SDcols=` to the columns the aggregation actually touches. Relatedly, `.N` gives the group size as a fast special symbol — never compute group counts via `nrow(.SD)` or `length(...)`, which forces `.SD` materialization for a number data.table already knows (verify against the currency brief for your version).

- **Prefer `keyby` over `by` when the result is consumed sorted; `by` when first-appearance order must be preserved**: `keyby` sets a key on the result (sorted, binary-searchable downstream) and the docs note it is typically *faster* than `by` because it avoids a separate re-sort; `by` preserves first-appearance group order at the cost of that ordering benefit. Picking `by` and then immediately `setkey`-ing or `order`-ing the result is doing the sort twice — use `keyby`. Picking `keyby` when downstream code depends on original group order is a correctness bug masquerading as a speedup. Match the choice to how the aggregate is consumed (verify against the currency brief for your version).

- **`merge(DT1, DT2)` where an `X[Y, on=]` join would avoid the copy**: data.table's native join is `X[Y, on=.(key)]`, which can join, filter (`nomatch=NULL`), pick (`mult=`), and — critically — *update in place* in one pass: `X[Y, col := i.val, on=]` writes matched values into X by reference with no intermediate result. `merge()` always allocates a new joined table. For enriching a 40M-row reading table with a small asset-attribute lookup, update-on-join touches only the assigned column; `merge` rebuilds the whole frame. Also exploit join forms base R can't express cheaply: `roll=TRUE`/`roll="nearest"` rolling joins to align readings to the nearest prior tariff/interval boundary (LOCF without a self-join loop), and non-equi joins (`on=.(t >= start, t <= end)`) for interval/range matching instead of a cartesian filter (verify against the currency brief for your version).

- **`reshape2`/`tidyr` (or base `reshape`) on a large frame instead of `melt`/`dcast`**: data.table's `melt` (wide→long) and `dcast` (long→wide) are C implementations built on radix order and binary search, far faster and more memory-efficient than reshape2/tidyr at scale (docs cite the 10GB+ regime). They also reshape *multiple measure groups at once* via `measure.vars=list(...)` / `patterns()` — pivoting a meter frame with 96 interval columns into long form in one C pass rather than melt-then-split-then-rebind. Flag large-frame reshapes routed through tidyverse reshapers when the data is already a data.table (verify against the currency brief for your version).

- **`as.data.table()` / un-needed `copy()` where `setDT` would convert by reference**: `as.data.table(x)` makes a *full copy* before converting; `setDT(x)` converts a list/data.frame to data.table **by reference**, no copy (and `setDF` reverses it). Converting a freshly-`fread` frame or an intermediate list with `as.data.table` doubles peak memory for nothing. Conversely, an over-defensive `copy()` before a `:=` (added "to be safe") reintroduces exactly the full-table deep copy this engine avoids — keep `copy()` only where a genuine independent snapshot is required, since `:=`'s by-reference mutation otherwise propagates to every binding of that object (the flip side: an *unintended* shared-reference mutation is a correctness bug — `copy()` is the deliberate fix, not noise to strip blindly) (verify against the currency brief for your version).

- **Over-allocation exhaustion forcing a silent table reallocation**: a data.table over-allocates spare column-pointer slots (`truelength` beyond `length`) so `:=` can add columns by reference without reallocating the column list; the `datatable.alloccol` option sets the default headroom and `setalloccol()` adjusts it. When a script adds *many* columns one `:=` at a time and exhausts the spare slots, data.table must reallocate the pointer vector — and a table arriving via `setDT` on a base structure may start with no headroom. Symptom: a loop of `:=` calls that degrades as columns accumulate. Pre-size with `setalloccol()`, or add all the columns in **one** `:=` call (the functional `DT[, ':='(a=..., b=..., c=...)]` form) so allocation happens once instead of once per column (verify against the currency brief for your version).

- **R-level loops where a vectorized C helper exists**: row-by-row lag/lead, NA-fill, rolling means, and coalesce written as R loops or `apply` run at R speed; data.table ships multi-threaded C equivalents — `shift()` (lag/lead/cyclic, group-aware via `by`), `nafill()`/`setnafill()` (const/LOCF/NOCB), `fcoalesce()` (first non-NA across columns), and `frollmean`/`frollsum`/`frollapply` (sliding-window, with `algo="fast"` online single-pass vs the O(n·k) naive loop). On interval meter data — gap-filling, prior-reading deltas, rolling demand averages per meter — these turn an explicit per-row loop into one parallelized call. Flag hand-rolled rolling/lag/fill loops over a data.table column. Cross-reference the `concurrency` lane in `../r.md` for the multi-threading these helpers (and `fread`) draw on (verify against the currency brief for your version).
