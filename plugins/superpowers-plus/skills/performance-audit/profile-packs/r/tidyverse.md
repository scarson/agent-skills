## tidyverse (dplyr / tidyr / purrr / readr)

> Scope: dplyr verb pipelines, tidyr reshaping, purrr functional iteration, and readr/vroom ingestion — the grammar most R data work is written in. The recurring theme is: **R-level per-row dispatch is the dominant avoidable cost** (`rowwise()`, `map()` over rows, `apply(MARGIN=1)`, `do()` each call R once per row and defeat the vectorized column engine underneath); **grouped operations scale with the number of groups, not just the data**, so at AMI scale (per-meter, per-feeder) the group-dispatch overhead — not arithmetic — is the bottleneck; **pipelines materialize intermediates** under copy-on-modify, so growing a frame in a loop is O(n²); and **the work should happen in the fastest engine that can do it** — push filters/aggregations to the DB/Arrow/data.table *before* the data lands in a tibble. Core `r.md` carries the **Runtime & copy-on-modify notes** (R's copy-on-modify, ALTREP, the `[`/`[[`/growing-vectors story) and the cross-cutting lanes; the sibling [`data-table.md`](data-table.md) is the data.table engine itself — reach for it (or `dtplyr` as the bridge) wherever this lens flags "the tidyverse pipeline is the bottleneck." This file is the tidyverse lens only.

- **`rowwise()` / `purrr::map` over rows defeating vectorization** (core `algorithmic` lane): `rowwise()` makes each row its own group — dplyr splits the frame into one-row pieces, evaluates the expression, and rejoins, paying per-row R dispatch the column engine was built to avoid. The same trap wears other faces: `map(seq_len(nrow(df)), ~ f(df[.x, ]))`, `apply(df, 1, ...)` (which also coerces the whole frame to a matrix, collapsing mixed types to character). The fix is almost always to reframe the per-row logic as **vectorized column expressions** — `case_when()`/`if_else()` for conditionals, arithmetic across columns, `rowSums(pick(...))`/`rowMeans()` for row aggregates (these "operate on the data frame as a whole" instead of splitting). Only when the per-row function is genuinely non-vectorizable (a scalar-in/scalar-out C-less computation, a model fit per row) is row dispatch justified — and even then `group_by()` over a key with vectorized work per group beats one-group-per-row.

- **Grouped `summarise()`/`mutate()` cost scales with group count, not row count**: dplyr's grouped engine pays a fixed dispatch + result-recombination cost *per group*, so a summarise over millions of singleton-ish groups (one per meter × interval) can spend most of its time in grouping machinery rather than the `mean()`/`sum()` inside. When the per-group computation is a standard statistic, this is exactly where a faster backend is a step-change, not a constant factor — `data.table` (see [`data.table.md`](data-table.md)) and `collapse` are built for high-cardinality grouped aggregation. Watch for grouped `mutate()` especially: it computes a group value and broadcasts it back to every row (`x - mean(x)` within group), so it pays the grouping cost *and* keeps the full row count.

- **`.by=` / `across()` vs persistent `group_by()`**: `group_by()` creates a `grouped_df` whose grouping *persists* across every subsequent verb until an explicit `ungroup()` — a frequent footgun is a pipeline that stays grouped far past where it needed to, so later `mutate()`/`filter()` silently pay per-group dispatch nobody intended. Per-operation `.by=` (e.g. `summarise(..., .by = c(meter_id, day))`) scopes grouping to the single verb and returns ungrouped, removing the stale-grouping cost and the forgotten-`ungroup()` bug class. `across()` collapses N copy-pasted column expressions into one tidy-selected call (`across(where(is.numeric), mean)`); it is a maintainability/correctness win, roughly cost-neutral per column — not itself a speedup (verify against the currency brief for your version).

- **Growing a frame with `bind_rows()`/`bind_cols()` inside a loop is O(n²)** (core `algorithmic` lane + Runtime & copy-on-modify notes): `acc <- bind_rows(acc, chunk)` inside a `for`/`map` reallocates and copies the entire accumulator every iteration under R's copy-on-modify semantics — total work quadratic in the number of chunks. Collect into a **list** and bind once at the end: `purrr::map(x, f) |> list_rbind()` (or `list_cbind()`), a single recombination pass. Note `map_dfr()`/`map_dfc()` were **superseded in purrr 1.0.0** in favor of `map() |> list_rbind()` (clearer semantics, no implicit dplyr dependency) — both do the one-shot bind correctly; the loop-append antipattern is the thing to flag (verify against the currency brief for your version).

- **`do()` / `group_modify()` / nested-tibble model dispatch**: `do()` (now superseded) and `group_modify()` run an R function once per group returning a data frame — the per-group R-call cost is the same family as `rowwise()`, acceptable for genuinely per-group complex work (fitting a model per feeder) but pure overhead when the body is a vectorizable summary that `summarise(across(...))` or a backend could express. `nest()` + `map()` over a list-column is the same trade: powerful for per-group modeling, wasteful when it's hiding an aggregation that the grouped engine or `collapse`/`data.table` would do in one vectorized pass. Justify the list-column by an irreducibly per-group object (a model, a plot), not by a statistic.

- **`collect()` placement with dbplyr / arrow** (core `data-access` lane): dbplyr and arrow build a **lazy** query — `filter()`/`select()`/`summarise()`/`group_by()` translate to SQL or an `arrow_dplyr_query` and push down to the database/Arrow engine; nothing moves into R until `collect()`. The dominant antipattern is `collect()` too early — pulling a whole meter-reads table into a tibble and *then* filtering/aggregating in R, which discards predicate/projection pushdown and out-of-core execution. Place every filter, column selection, and aggregation **before** `collect()` so the engine returns only the reduced result; use `show_query()` (dbplyr) to confirm the work actually pushed down. `compute()` (arrow) materializes to an Arrow Table for further Arrow ops without crossing into R (verify against the currency brief for your version).

- **`dtplyr` — lazy data.table backend for dplyr code**: when a *profiled* dplyr pipeline is the bottleneck (high-cardinality grouped aggregation, large joins, big grouped `mutate`s), `lazy_dt(df)` lets you keep dplyr verb syntax while dtplyr translates the chain to one optimized data.table expression, collected by `as_tibble()`/`as.data.table()`. Translation overhead is "proportional to the complexity of the input code, not the input data" (sub-millisecond per verb), so the win is the data.table engine's grouping speed and reduced copying. Caveat: to preserve dplyr semantics, `mutate()` copies by default; pass `lazy_dt(df, immutable = FALSE)` to allow in-place modification when you own the input. For native data.table idioms and the deeper engine model see [`data.table.md`](data-table.md) (verify against the currency brief for your version).

- **`arrow::open_dataset()` over reading whole CSVs**: for partitioned/columnar data at scale, `open_dataset()` on a Parquet (or partitioned CSV) directory gives a lazy dplyr-able Dataset with predicate pushdown (skip row-groups/files that fail the filter), projection pushdown (read only selected columns), and out-of-core execution — so a per-feeder query reads a fraction of disk instead of slurping every CSV into RAM. Reading the whole AMI history as CSV and filtering in a tibble loses all three. Repartitioning raw CSV deliveries to partitioned Parquet (by date/feeder) is often the single highest-leverage IO change; flag pipelines that re-read large CSVs every run (verify against the currency brief for your version).

- **Lazy/indexed ingestion: `vroom` and readr `lazy=`** (core `data-access` lane): `vroom()` indexes the file and reads column values **lazily via ALTREP** — import is near-instant and a column's data is materialized only when first touched, so reading a wide meter file and using a handful of columns avoids parsing the rest. The load-bearing caveat: the benefit *evaporates the moment an operation touches all the data* (a full-table summary, writing every column back out) — vroom shines for select-few-columns / preview / wide-file workflows, not for "read everything then crunch everything." Pair with `col_select=` (vroom) / `col_select`/`usecols`-style selection so unused columns are never indexed. Specifying `col_types=` explicitly also skips readr's type-guessing scan (default guesses from the first ~1000 rows) and prevents a mis-guessed `character` column silently disabling downstream vectorized math (verify against the currency brief for your version).

- **`collapse` for fast grouped statistics inside a tidy pipeline**: when the bottleneck is grouped aggregation but you want to stay close to dplyr syntax, `collapse` offers C/C++ fast statistical functions (`fmean`, `fsum`, `fmedian`, `fsd`, `fmode`, `fnth`, weighted variants) with "extremely low evaluation overheads," composable as `df |> fgroup_by(meter_id) |> fsummarise(load = fmean(kw))`. The aggregation primitives are dramatically faster than base R's per-group apply and competitive with data.table for many grouped reductions; note collapse's general *manipulation verbs* are not always faster than dplyr — the win is concentrated in the fast statistical/grouping functions, so reach for it on the aggregation hot spot specifically, not as a blanket pipeline rewrite (verify against the currency brief for your version).

- **Long `%>%`/`|>` chains and tibble copying on large frames** (core `memory` lane + Runtime & copy-on-modify notes): each verb in a pipe conceptually produces a new frame, and under copy-on-modify a `mutate()` that changes one column may copy the column (or, for some operations, force a fuller copy) — on a multi-GB meter table a chain of many `mutate()`s can multiply peak memory through intermediates that exist only between pipe steps. tibbles and data.frames share this copy model (a tibble is not faster to mutate — its wins are subsetting/printing semantics, not mutation cost); the genuine escape from per-step copying is a backend that mutates in place (`dtplyr` with `immutable = FALSE`, or native data.table — see [`data.table.md`](data-table.md)), or fusing several `mutate()`s into one verb so the engine allocates once (verify against the currency brief for your version).
