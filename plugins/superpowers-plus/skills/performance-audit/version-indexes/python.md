# Python performance version index

> `covered_through`.

## Interpreter / Runtime (CPython)

- **Faster CPython (Specializing Adaptive Interpreter, PEP 659)** — landed in **3.11** — adaptive bytecode specialises hot call sites for common types (binary ops, subscript, attribute load, method calls, globals); 10–25% speedup per operation class, geometric mean ~25% faster than 3.10 on pyperformance — automatic; no API change required; benefit is greatest in pure-Python CPU-bound loops.
- **Zero-cost `try/except` (when no exception is raised)** — landed in **3.11** — `try` blocks now have zero runtime overhead on the happy path; removes prior per-`try` cost — no API change; upgrade to 3.11+ to receive automatically; eliminates historic hesitation to wrap hot code in `try`.
- **Cheap, lazy frame objects** — landed in **3.11** — Python call frames reuse C-stack space and defer allocation of full frame objects until debugger/introspection requests them; 3–7% overall speedup, up to 1.7× for deeply recursive functions — automatic on 3.11+.
- **Frozen stdlib module imports** — landed in **3.11** — core startup modules are statically allocated as frozen bytecode; interpreter startup 10–15% faster — automatic; no API change; most impactful for short-lived scripts and CLI tools.
- **Comprehension inlining (PEP 709)** — landed in **3.12** — list/dict/set comprehensions are inlined into the enclosing frame rather than creating a disposable function object; up to 2× faster comprehension execution — automatic on 3.12+; no API change.
- **Per-interpreter GIL (PEP 684)** — landed in **3.12** (C-API only), Python-level `interpreters` module anticipated 3.13+ — each sub-interpreter can hold its own GIL enabling true CPU parallelism across interpreters without sharing the GIL — use when building multi-core parallel workloads via C extension or the `interpreters` stdlib module; not yet a drop-in `threading` replacement.
- **`sys.monitoring` low-overhead instrumentation (PEP 669)** — landed in **3.12** — pay-as-you-go event hooks for profilers/debuggers/coverage tools; replaces `sys.settrace`/`sys.setprofile` with near-zero overhead when no events are subscribed — supersedes `sys.settrace` for custom profilers; use when writing profiling/coverage tooling, not as an end-user perf feature.
- **Free-threaded / no-GIL build (PEP 703 / PEP 779)** — **experimental in 3.13** (`python3.13t`), **officially *supported* in 3.14** (`python3.14t`, PEP 779) — GIL-free CPython build enables true threading parallelism for CPU-bound multi-threaded code; the single-threaded penalty dropped from **~40% (3.13) to ~5–10% (3.14)** — still a separate opt-in build, and all C extensions must declare `Py_mod_gil` support, but on 3.14 it is a supported configuration to evaluate (not merely experimental); real workloads report large throughput gains when thread-bound. Frame as "evaluate on 3.14+", and verify C-extension readiness.
- **Experimental JIT compiler (PEP 744)** — experimental in **3.13** (`--enable-experimental-jit`), shipped in the **official macOS/Windows binaries in 3.14** (still experimental, opt-in via `PYTHON_JIT=1`) — copy-and-patch JIT translating Tier-2 IR to machine code; gains modest so far ("a few percent"); enable to measure on 3.14+, don't assume large wins yet.
- **Tail-calling interpreter** — **3.14** — the eval loop is rebuilt from one giant C switch into tail-called per-opcode functions; on supporting compilers/platforms ~3–5% overall speedup on the benchmark suite — automatic on 3.14+ builds compiled with a capable toolchain; no API change.
- **Linux `perf` profiler support** — landed in **3.12** (improved in **3.13** with `PYTHON_PERF_JIT_SUPPORT`/`-X perf_jit`) — annotates the process so Linux `perf` can resolve Python frames by name; 3.13 removes the frame-pointer requirement — use `PYTHONPERFSUPPORT=1` / `-X perf` (3.12+) or `-X perf_jit` (3.13+) for CPU profiling without Py-Spy; requires a Linux host.

## asyncio

- **`asyncio.TaskGroup`** — landed in **3.11** — structured-concurrency context manager that creates and awaits a group of tasks, cancels all siblings on first failure; recommended over bare `asyncio.gather()` for new code — supersedes `asyncio.gather()` for fan-out patterns; safer cancellation semantics and `ExceptionGroup` error reporting; use when spawning independent concurrent coroutines.
- **`asyncio.eager_task_factory` / `asyncio.create_eager_task_factory`** — landed in **3.12** — tasks that complete synchronously (e.g. cache-hit coroutines) skip the event loop scheduling round-trip; 2–5× faster for workloads with many synchronous-completing coroutines — supersedes default `loop.set_task_factory(None)` — use by passing `asyncio.eager_task_factory` to `loop.set_task_factory()` or via `asyncio.run(…, loop_factory=…)`; net negative if most tasks are genuinely async.
- **`asyncio.current_task()` 4–6× speedup** — landed in **3.12** — internal implementation rework; hot-path cost dropped substantially — automatic on 3.12+; no API change.
- **`asyncio.timeout()` context manager** — landed in **3.11** — structured timeout via `async with asyncio.timeout(n):` block; recommended over `asyncio.wait_for()` for new code — supersedes `asyncio.wait_for()` for deadline management; more composable, no per-task wrapping overhead.

## Library fast paths (version-independent)
> Durable client-library idioms (not tied to a Python release), carried so the idiom-currency lane is
> grounded on the common backend libraries rather than falling back to model knowledge. Verify the exact
> API against the library version in the lockfile.
>
> **Coverage note:** this index curates the high-traffic libraries below. A finding on a library *not*
> listed here (a niche client or driver) legitimately carries **LOW/Heuristic** confidence pending a
> live currency brief — that is *expected*, not a research failure; the lane should say so rather than
> fabricate a version-specific claim.

- **httpx — reuse one `AsyncClient` with explicit `httpx.Limits`** — version-independent — a client built per request opens a fresh connection pool and gets no keep-alive reuse; construct one long-lived `AsyncClient(limits=httpx.Limits(max_connections=…, max_keepalive_connections=…))` and share it. The default `max_connections=100` is both a fan-out ceiling and, under an unbounded `gather`, a rate-limit hazard — size the pool to your concurrency cap — supersedes per-call `httpx.AsyncClient()` and unbounded fan-out — for many small requests to one host, `http2=True` multiplexes them over one connection.
- **httpx — `AsyncHTTPTransport(retries=…)` over a hand-rolled reconnect loop** — version-independent — transport-level connection retries avoid reimplementing backoff in app code (covers connection errors; status-code retry policy still belongs to the caller) — supersedes a manual `for attempt in range(...)` reconnect loop around the request.
- **redis-py (`redis.asyncio`) — reuse the client/pool; coalesce round-trips; `aclose()`** — version-independent — create one `redis.asyncio.Redis` (it owns a connection pool) and reuse it; a client per call defeats pooling. Replace N sequential `GET`s with one `MGET`/pipeline to collapse round-trips. Close async clients with `await client.aclose()` — the bare `.close()` is the sync/deprecated path on the async client — supersedes per-call clients and serial single-key gets.
- **APScheduler — `AsyncIOScheduler` + misfire/overlap guards** — version-independent — run async jobs on `AsyncIOScheduler` (the blocking/background schedulers execute the job off the loop or in a thread, serializing async work); set `max_instances`, `coalesce=True`, and a `misfire_grace_time` so a job that runs longer than its interval can't pile up overlapping executions (the "the run exceeds its own interval" failure mode) — supersedes default scheduler settings for any job whose duration can approach its interval.
- **kubernetes-client (official Python client) — narrow list/watch, don't re-list, run off the loop** — version-independent — the client is **synchronous/blocking** (urllib3); in an async app call it in a threadpool, never directly on the event loop. Use `field_selector`/`label_selector` and `limit`+continue paging to shrink payloads instead of listing whole namespaces; **`watch.Watch().stream(...)` with `resource_version`** to follow changes rather than re-`list_*` in a poll loop (re-listing every tick is the dominant cost); reuse one `ApiClient` (it owns the connection pool). Supersedes per-tick full re-list and per-call client construction. *(The dominant framework in operator/controller services and a common index blind spot — treat as Heuristic until a brief grounds the exact client version.)*
- **pydantic v2 — validate once; `TypeAdapter` for repeated shapes; avoid double validation** — version-independent — v2's core is Rust (fast), but **validating the same payload twice is the common waste** — e.g. a framework `response_model` re-validating data the handler already built/validated. Use `TypeAdapter(T).validate_python(...)` (built once, reused) for hot repeated non-`BaseModel` shapes, `model_validate`/`model_dump` over the deprecated `.parse_obj()`/`.dict()`, and `model_construct()` to skip validation for data you already trust — supersedes re-validating trusted data on a hot path.
- **SQLModel — it *is* SQLAlchemy: eager-load relationships, don't lazy-load in a loop** — version-independent — SQLModel wraps SQLAlchemy 2.0, so the ORM-database lane's rules apply: a relationship accessed per row triggers a lazy SELECT (N+1) — use `selectinload`/`joinedload`; `session.exec(select(...))` returns ORM objects (change-tracked) — for read-only list endpoints select only the needed columns; mind that a `SQLModel` with `table=True` is a mapped class, a plain one is a Pydantic model — supersedes per-row relationship access on list/serialize paths.

## stdlib

- **`itertools.batched(iterable, n)`** — landed in **3.12** — yields fixed-size tuples from an iterable without materialising a list; the canonical chunked-iteration primitive — supersedes `more-itertools.batched`, manual `zip(iter, iter, …)`, or `[seq[i:i+n] for i range(…)]` patterns — use for bulk processing, pagination, and chunk-based writes.
- **`ExceptionGroup` / `except*` (PEP 654)** — landed in **3.11** — raise/catch multiple independent exceptions in one `except*` block; enables `TaskGroup` error semantics — not a direct perf feature; included because it unlocks `TaskGroup` (above) which IS perf-relevant; use when handling multi-task failures.
- **`re` engine (computed gotos / threaded code)** — landed in **3.11** — regex matching engine refactored to use computed gotos on supported platforms; up to 10% faster than 3.10 on regex benchmarks — automatic on 3.11+; no API change.
- **`sum()` integer fast path** — landed in **3.11** — ~30% faster for integers smaller than a machine word — automatic.
- **List comprehension resize streamlining** — landed in **3.11** — up to 20–30% faster list comprehensions from smarter growth strategy — automatic on 3.11+.
- **`struct.pack`/`unpack` and `re` substitution** — landed in **3.12** — regex substitution with group references 2–3× faster; `struct` operations significantly faster (part of broader 3.12 stdlib improvements) — automatic on 3.12+.
- **`math.fma(x, y, z)`** — landed in **3.13** — fused multiply-add with single rounding; avoids double-rounding of `x*y + z` — use in numerically sensitive inner loops where rounding accuracy matters.

## Django ORM

- **`QuerySet.bulk_create(update_conflicts=True)`** — landed in **Django 4.1** — single-statement upsert (INSERT … ON CONFLICT DO UPDATE) on MariaDB, MySQL, PostgreSQL, SQLite 3.24+ — supersedes `get_or_create()` / load-then-save loop for bulk upsert scenarios; eliminates N round-trips — use when inserting or updating many rows where uniqueness conflicts are expected.
- **Async ORM interface (`aget`, `afilter`, `asave`, `abulk_create`, etc.)** — landed in **Django 4.1** — native `async def` view handlers and async queryset methods (prefixed `a…`) allow true async ORM calls under ASGI without `sync_to_async()` wrappers — supersedes `sync_to_async()` wrapping of ORM calls in async views — use when deploying on ASGI (Daphne/Uvicorn) with I/O-bound views.
- **`QuerySet.iterator(chunk_size=…)` with `prefetch_related`** — landed in **Django 4.1** — streaming iteration over large querysets now supports prefetch, previously all prefetch was skipped with `iterator()` — use for large result sets where related objects are needed but full materialisation into a list is undesirable.
- **`QuerySet.bulk_create()` / `abulk_create()` returning PKs** — landed in **Django 5.0** — methods now populate `pk` on each model instance after insert; PostgreSQL 15+ can also use `DEFAULT` keyword in bulk INSERT — supersedes a follow-up SELECT to retrieve generated IDs after bulk insert.
- **Persistent database connection health checks (`CONN_HEALTH_CHECKS`)** — landed in **Django 4.1** — reuses `CONN_MAX_AGE` connections but adds a health-check ping to avoid errors on stale TCP connections — use with `CONN_MAX_AGE > 0` in production to reduce per-request connection overhead without silent connection failures.
- **Built-in Background Tasks framework** — **Django 6.0** (Dec 2025) — a first-party task abstraction (`django.tasks`) to run work outside the request/response cycle with pluggable backends — relevant when auditing simple "fire work off the request path" needs: it can replace a hand-rolled thread or a heavyweight Celery dependency for basic cases (Celery/Dramatiq still win for complex routing/retries/scheduling — see the `task-queues` module). Django **5.2 is the current LTS** (supports Python 3.10–3.14); 6.0 supports 3.12–3.14. *(verify backend capabilities for your version.)*

## SQLAlchemy

- **`session.execute(select(Model))` unified API** — landed in **SQLAlchemy 2.0** — single `Session.execute()` entry point replaces `Session.query()` legacy API; internally caches compiled SQL per statement shape — supersedes `session.query(Model).filter(…).all()` — use `select(Model).where(…)` fed to `session.execute()` or `session.scalars()` for all new code; 1.x-style query API still works but is legacy.
- **SQL compilation caching (transparent, `query_cache_size`)** — landed in **SQLAlchemy 1.4**, universally effective in **2.0** — compiled SQL strings are cached by statement structure (not bind values); amortises Python-side compilation across repeated executions; check logs for `[cached since N s]` — tune `create_engine(query_cache_size=N)` (default 500) for workloads with many distinct statement shapes; dynamic query builders that produce unbounded distinct shapes disable caching.
- **`insertmanyvalues` bulk INSERT with RETURNING** — landed in **SQLAlchemy 2.0** — transparently replaces `executemany()` with batched single-statement `INSERT … VALUES (…),(…) RETURNING …` for PostgreSQL, SQLite 3.35+, MariaDB, SQL Server; resolves the ORM server-generated PK retrieval bottleneck — automatic; no API change; controlled by `insertmanyvalues_page_size` (default 1000) engine parameter — replaces `session.bulk_insert_mappings()` for multi-row ORM inserts.
- **`session.bulk_insert_mappings()` / `session.bulk_update_mappings()`** — available since **1.x** (superseded path) — bypasses ORM unit-of-work overhead for raw dict inserts; still faster than adding tracked objects one-by-one — superseded by `session.execute(insert(Model), [dicts])` in 2.0 which benefits from `insertmanyvalues` — use legacy bulk APIs only when targeting 1.x compatibility.
- **`relationship(lazy="write_only")`** — landed in **SQLAlchemy 2.0** — write-only relationship strategy never issues a SELECT on access; raises `InvalidRequestError` on read attempt — supersedes `lazy="dynamic"` (deprecated in 2.0) for append-heavy one-to-many collections — use when a collection is large enough that loading it would be prohibitive; append/remove only.
- **Async Core + ORM (`AsyncSession`, `AsyncEngine`)** — landed in **SQLAlchemy 1.4**, stabilised in **2.0** — full async support via `sqlalchemy.ext.asyncio`; `AsyncSession.execute()` and `AsyncSession.scalars()` mirror the sync API — use with `asyncpg` or `aiosqlite` drivers under async frameworks (FastAPI, Starlette); avoid mixing sync and async sessions.
- **`Result.yield_per(n)` / server-side cursors** — available since **1.4** — streams result rows in batches of `n` without buffering the full result set — supersedes `.all()` for large result sets where incremental processing is possible — use `conn.execution_options(stream_results=True)` + `yield_per()` for large exports/migrations.

## pandas / NumPy

- **Copy-on-Write (CoW, `pd.options.mode.copy_on_write = True`)** — opt-in in **pandas 2.0**, default in **pandas 3.0** — chained operations return views and only copy lazily when a mutation occurs; eliminates defensive copies in read-heavy pipelines — supersedes manual `.copy()` guards and avoids silent chained-assignment mutations — use when upgrading from 1.x; enable opt-in CoW early to catch code that relied on chained-assignment side-effects.
- **PyArrow-backed dtypes (`dtype_backend="pyarrow"`)** — landed in **pandas 2.0** — I/O functions (`read_csv`, `read_parquet`, etc.) accept `dtype_backend="pyarrow"` to store columns as Arrow arrays; zero-copy interchange with Arrow ecosystem, faster string/bool/nullable-int operations — supersedes object-dtype string columns and numpy float64 for nullable numerics — use `pd.read_csv(…, dtype_backend="pyarrow")` or `df.convert_dtypes(dtype_backend="pyarrow")` for string-heavy or nullable workloads.
- **`ArrowDtype` for individual columns** — landed in **pandas 2.0** — explicitly assign `pd.ArrowDtype(pa.large_string())` etc. per column for mixed backends — use when only specific columns benefit from Arrow storage while rest remain numpy-backed.
- **NumPy 2.0 sort/partition SIMD acceleration** — landed in **NumPy 2.0** — `np.sort`, `np.argsort`, `np.partition`, `np.argpartition` accelerated via Intel x86-simd-sort and Google Highway; hardware-specific speedups can be large — automatic on NumPy 2.0+; no API change.
- **NumPy 2.0 `StringDType` and `numpy.strings` ufuncs** — landed in **NumPy 2.0** — variable-length UTF-8 string dtype with dedicated SIMD-backed ufuncs in `numpy.strings`; faster string operations than object-dtype arrays — supersedes `dtype=object` string arrays for vectorised string work — use `np.array(data, dtype=np.dtypes.StringDType())`.
- **NumPy 2.0 macOS Accelerate linear algebra** — landed in **NumPy 2.0** — macOS ≥14 wheels link against Apple's Accelerate framework; linear algebra operations (`np.linalg.*`) significantly faster and wheel size ~3× smaller — automatic when installing from PyPI on macOS 14+; no API change.
- **`np.fft` / `np.linalg` via OpenBLAS / MKL** — durable baseline (all versions) — NumPy's BLAS/LAPACK operations are only fast when linked against optimised BLAS (OpenBLAS, MKL, or Accelerate); `numpy.show_config()` confirms linkage — if benchmarking shows slow `np.dot`/`np.matmul`, the NumPy wheel may lack optimised BLAS; reinstall via `conda` or use `numpy[openblas]` extras.

## SciPy / Numba (scientific computing)
> Version-pinned facts for the `scientific-computing` module. The durable idioms (factor-once `splu`, analytic Jacobians, SOS filters, smooth FFT lengths) live in `../profile-packs/python/scientific-computing.md`.

- **`scipy.fft` (pocketfft) replaces `scipy.fftpack`** — `scipy.fft` landed in **SciPy 1.4** and `scipy.fftpack` is now documented *legacy* ("new code should use scipy.fft") — `scipy.fft` adds a `workers=` parameter to parallelise transforms over batch/non-transformed axes, supports `next_fast_len` for smooth (small-prime-factor) lengths, and is faster than the old fftpack — supersedes `scipy.fftpack` and bare `numpy.fft` where multi-threaded or real-input (`rfft`) transforms help.
- **`cKDTree` is a backward-compat alias for `KDTree`** — since **SciPy 1.6.0** the two are functionally identical ("prior to v1.6.0 cKDTree had better performance… now the two names exist only for backward-compatibility reasons") — use either; no perf reason to prefer `cKDTree` on modern SciPy.
- **Sparse *array* API (`csr_array`/`csc_array`/`coo_array`) over the *matrix* API** — the NumPy-semantics sparse classes are the recommended interface for new code; `@`/`*` differ between the array and matrix interfaces, so mixing them is a hazard — prefer the `*_array` classes; build in COO/LIL/DOK and convert once to CSR/CSC for arithmetic (verify the exact migration status against your SciPy version).
- **Numba `cache=True`** — durable — writes the compiled artifact to disk so the first-call compilation cost isn't re-paid across process restarts (caveat: globals are baked in as compile-time constants and changes in imported modules aren't detected) — use for short-lived processes (CLIs, workers) that re-import the same jitted kernels.

## scikit-learn / gradient boosting
> Version-pinned facts for the `machine-learning` module.

- **`HistGradientBoostingClassifier`/`Regressor`** — histogram boosting landed in **scikit-learn 0.21**; **native NaN handling** and **native categorical support** (`categorical_features=`, with `"from_dtype"` accepted in recent versions) since **0.24** — orders of magnitude faster than the exact `GradientBoosting*` once `n_samples` exceeds ~10k, OpenMP-parallel, and removes the need for imputation/one-hot preprocessing — supersedes `GradientBoostingClassifier`/`Regressor` for large tabular data.
- **Successive-halving search (`HalvingGridSearchCV` / `HalvingRandomSearchCV`)** — since **scikit-learn 0.24**, still **experimental** (requires `from sklearn.experimental import enable_halving_search_cv`) — evaluates all candidates on a small resource first and promotes only survivors, "much faster at finding a good combination" than exhaustive `GridSearchCV` — use when the grid × folds × per-fit cost is large.
- **joblib `loky` backend auto-caps inner threads** — durable behaviour — scikit-learn parallelism (`n_jobs`) defaults to the `loky` process backend, which caps each worker's inner BLAS/OpenMP threads to ~`n_cpus // n_jobs` via `threadpoolctl` to avoid over-subscription — **but manually setting `OMP_NUM_THREADS`/`OPENBLAS_NUM_THREADS`/`MKL_NUM_THREADS` overrides and disables that mitigation**, re-introducing the thread storm — verify active thread counts with `threadpoolctl` before adding workers.
- **XGBoost 2.0 — `tree_method='hist'` is the default; unified `device=` replaces `gpu_hist`** — landed in **XGBoost 2.0** (current stable 3.2, Feb 2026) — previously the default was chosen between `approx`/`exact` from the data/environment; now `hist` is default and `device='cuda'` selects GPU (the old `tree_method='gpu_hist'` is **deprecated** → use `device='cuda'` + `tree_method='hist'`); `QuantileDMatrix` quantises directly from source, avoiding the full-precision `DMatrix` copy — supersedes `gpu_hist` and manual tree-method selection on 2.0+.
- **LightGBM `Dataset(free_raw_data=True)` default** — durable — the binned `Dataset` discards the raw data after construction by default, so a later operation needing raw data forces a rebuild; validation sets must be built with `reference=train_set` to share bin edges — build the `Dataset` once and reuse it across boosting rounds/folds.

## Deep learning (PyTorch)
> Version-pinned facts for the `deep-learning` module. Durable async/sync, DataLoader, and AMP idioms live in `../profile-packs/python/deep-learning.md`.

- **`torch.compile`** — landed in **PyTorch 2.0** — graph capture + kernel fusion via TorchDynamo/Inductor; pair with `torch.set_float32_matmul_precision('high')` to enable TF32 matmuls on Ampere+; can re-compile on input-shape churn (pin/bucket shapes) — supersedes TorchScript (`torch.jit`) for most new optimisation work; opt-in via `model = torch.compile(model)`.
- **`torch.inference_mode()`** — landed in **PyTorch 1.9** — "the extreme version of `no_grad`": additionally skips version-counter bumps and view tracking for a larger speedup, with the restriction that tensors created inside it can't later join an autograd-recorded computation — supersedes `torch.no_grad()` for pure inference paths (a deployed forecaster); fall back to `no_grad` when an output must feed a grad-tracked path.
- **`torch.set_float32_matmul_precision('high'|'medium')`** — modern TF32 control (PyTorch 1.12+) — runs fp32 matmuls at reduced internal precision on supported GPUs for large speedups on matmul-bound (LSTM/GRU/transformer) training — supersedes the older `torch.backends.cuda.matmul.allow_tf32` flag for setting matmul precision.
- **`DistributedDataParallel` over `DataParallel`** — durable — `DataParallel` is single-process/multi-thread and bottlenecks on the GIL; `DistributedDataParallel` is one process per GPU and is the documented recommendation **even on a single node** — supersedes `DataParallel` for all real multi-GPU training.

## Distributed (Spark / Dask)
> Version-pinned facts for the `distributed-data` module.

- **Spark Adaptive Query Execution (`spark.sql.adaptive.enabled`) default `true` since Spark 3.2.0** — AQE coalesces shuffle partitions at runtime (`spark.sql.adaptive.coalescePartitions.enabled`) and handles skewed joins (`spark.sql.adaptive.skewJoin.enabled`) — so on **3.2+** the static `spark.sql.shuffle.partitions` (default **200**) matters far less, while on a pre-3.2 cluster (AQE off) hand-tuning it is high-leverage — confirm AQE is actually on before assuming 200 is harmless.
- **`spark.sql.execution.arrow.pyspark.enabled` default `false`** — durable — Arrow-based transfer (vectorised pandas UDFs, `mapInPandas`/`applyInPandas`, faster `toPandas()`) is **off by default** and must be enabled; even enabled, `toPandas()`/`toArrow()` still collect *all* rows to the driver — Arrow makes the transfer cheaper, not bounded.
- **`spark.sql.autoBroadcastJoinThreshold` default 10 MB** — durable — a small dimension table under this size auto-broadcasts (no shuffle join); above it, use the explicit `broadcast(df)` hint to force a broadcast join and avoid shuffling the large side — set to `-1` to disable broadcasting.
- **Dask DataFrame ~100 MB partition target** — durable guidance — "aim for partitions that have around 100MB of data each"; thousands of tiny files/partitions make scheduler overhead (~200µs–1ms/task) dominate — `repartition(partition_size="128MB")` after a `read_parquet` that inherited a tiny-file layout.

## Geospatial (GeoPandas / Shapely)
> Version-pinned facts for the `geospatial` module.

- **Shapely 2.0 vectorized GEOS ufuncs** — landed in **Shapely 2.0** (Dec 2022) — geometry operations are exposed as NumPy ufuncs over *arrays* of geometries (`shapely.intersects`/`buffer`/`distance`, …), so GeoSeries methods run the whole column in one C call — supersedes per-geometry Python loops / `.apply(shapely_fn)`; GeoPandas requires Shapely 2.0 since GeoPandas 1.0.
- **pyogrio is the default `read_file`/`to_file` engine since GeoPandas 1.0** — landed in **GeoPandas 1.0** (Jun 2024) — replaced the row-by-row `fiona` engine with vectorized GDAL/OGR I/O; the migration guide cites **>5–20×** bulk read/write speedups, and `use_arrow=True` adds a further documented ~2–4× — on **<1.0** pass `engine="pyogrio"` explicitly; supersedes the fiona default. Read-pushdown args (`bbox=`/`mask=`/`columns=`/`rows=`/`where=`) apply on either engine.
- **GeoParquet** — durable — `GeoDataFrame.to_parquet`/`read_parquet` (and Feather) preserve CRS + geometry encoding, support column pruning, and round-trip far faster than Shapefile/GeoJSON for re-read intermediates — supersedes text/Shapefile working formats (keep those for interchange only).

## Scientific / optimization modeling (Pyomo / CVXPY / Gurobi / OR-Tools)
> Durable client-library idioms for the `optimization-modeling` module (mostly version-independent — verify the exact API against the lockfile). The dominant theme is **build once, modify-and-resolve** across a scenario/rolling-horizon sweep.

- **CVXPY DPP `Parameter` canonicalization caching** — version-independent (DPP) — the first solve of a DPP-compliant problem caches the parameter→problem-data mapping, so subsequent solves that only change `Parameter.value` skip re-canonicalization ("substantially faster") — supersedes rebuilding `Problem` or using plain constants in a resolve loop; a DPP-violating product (e.g. `Parameter`×`Parameter`) silently forces full re-canonicalization. Warm start is on by default, but factorization reuse is lost if the matrix itself is a `Parameter`.
- **Pyomo persistent / APPSI solvers** — version-independent — `SolverFactory('gurobi_persistent'/'cplex_persistent')` and the APPSI solvers (`appsi_gurobi`/`appsi_highs`) keep the solver model in memory and apply incremental changes, avoiding the file-based interface's LP/NL re-write + solver-process restart on every solve; the manual persistent interface requires notifying it of changes (`add_constraint`/`update_var`/…), APPSI auto-detects them — supersedes a fresh `SolverFactory('gurobi')` file solve per scenario.
- **Gurobi lazy update + model reuse** — version-independent — modifications are queued and applied only on `Model.update()`, `Model.optimize()`, or `Model.write()`; for a sweep that changes a few coefficients/bounds, mutate the live model (`Model.chgCoeff`, `Var/Constr.setAttr`) and supply warm starts (`Var.Start` for MIP; `VBasis`+`CBasis` or `PStart`+`DStart` for LP — each pair required together) rather than rebuilding — and **reuse one `Env`** across models ("much more efficient than creating and destroying multiple environments"; not thread-safe, so one `Env` per thread). Updating after *every* edit defeats the lazy batching.
- **OR-Tools CP-SAT hints + workers** — version-independent — `CpModel.add_hint(var, value)` seeds a start that need not satisfy all constraints and "can decrease solving times substantially… solving a similar model repeatedly"; `solver.parameters.num_search_workers` sizes the parallel strategy portfolio (set + measure) — build the `CpModel` once and re-solve.
